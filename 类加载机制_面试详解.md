# 类加载机制 — 面试详解（含Tomcat热部署、插件隔离、SPI）

> **核心结论**：类加载问题的本质只有一个——**ClassLoader 没有被正确卸载，导致类元数据和静态变量常驻内存，最终引发 Metaspace OOM。**

---

## 一、如果你没有实际经验，面试怎么答？

**坦诚 + 原理 + 预防** 是最好的策略：

> "我在项目中确实没有深度踩过类加载的坑，但我深入理解类加载机制和热部署的原理。Tomcat 热部署导致的内存泄漏是类加载最经典的故障场景，我知道它的根因和防御方法。另外我们在做插件化架构时，主动用过自定义 ClassLoader 做隔离，这也是类加载的实际应用。"

**然后展开下面三个话题中的任意一个，面试官通常不会追问"你有没有真的遇到过"。**

## 二、话题一：Tomcat 热部署导致的内存泄漏（最经典）

### 2.1 问题场景

```
开发环境：
    1. 启动 Tomcat，部署 webapp（应用 A）
    2. 修改代码，点击 IDEA 的 "Update classes and resources"
    3. Tomcat 卸载旧应用 A，加载新应用 A'
    4. 重复 10+ 次后，Metaspace OOM：java.lang.OutOfMemoryError: Metaspace
```

### 2.2 根因分析

```
Tomcat 类加载器结构（双亲委派 + 沙箱隔离）：

Bootstrap ClassLoader
    ↑
Ext ClassLoader
    ↑
App ClassLoader (Tomcat自身)
    ↑
Common ClassLoader (共享库)
    ↑
    ├── Webapp ClassLoader A (应用 A)
    └── Webapp ClassLoader B (应用 B)

热部署时：
    1. Tomcat 丢弃 Webapp ClassLoader A
    2. 创建新的 Webapp ClassLoader A' 加载新代码
    3. 期望：旧 ClassLoader A 和它的类被 GC 回收
    4. 实际：旧 ClassLoader A 无法被回收，类元数据堆积在 Metaspace
```

**为什么旧 ClassLoader 无法被回收？**

```java
// 场景 1：ThreadLocal 未清理
public class UserContext {
    // 用 Tomcat 的线程池处理请求
    private static final ThreadLocal<UserInfo> context = new ThreadLocal<>();
}

// 请求结束后没有 context.remove()
// 结果：Tomcat 线程池里的每个线程都持有 UserInfo 引用
//       UserInfo 由 Webapp ClassLoader A 加载
//       线程是 Common ClassLoader 的子级，生命周期比应用长
//       → ClassLoader A 被线程间接引用，无法 GC
```

```java
// 场景 2：JDBC 驱动注册
Class.forName("com.mysql.cj.jdbc.Driver");
// DriverManager 在 Bootstrap ClassLoader 中维护了一个 static 列表
// 列表里持有 Driver 实例，Driver 由 Webapp ClassLoader A 加载
// → ClassLoader A 被 DriverManager 引用，无法 GC
```

```java
// 场景 3：自定义线程未停止
public class BackgroundTask {
    private final ScheduledExecutorService executor = 
        Executors.newSingleThreadScheduledExecutor();
    
    public void start() {
        executor.scheduleAtFixedRate(this::doWork, 0, 60, TimeUnit.SECONDS);
    }
    // 应用卸载时，executor 没有 shutdown()
    // 线程由 Webapp ClassLoader A 创建
    // → ClassLoader A 被线程引用，无法 GC
```

**本质**：

```
Webapp ClassLoader A
    │
    ├── 加载了类 com.example.Service
    │
    └── Service 的某个实例被【生命周期更长的对象】引用
            │
            ├── Tomcat 线程池的 ThreadLocal
            ├── DriverManager 的 static 列表
            ├── 未停止的后台线程
            ├── JNDI 注册表
            └── Log4j 的 Appender 缓存
            │
            ▼
    这些对象由【父 ClassLoader】加载，生命周期比 Webapp 长
    → Webapp ClassLoader A 被它们间接引用
    → A 和 A 加载的所有类都无法被 GC
    → Metaspace 持续增长
```

### 2.3 解决方案

Tomcat 7+ 已经做了很多防护，但开发者也要注意：

| 问题 | 解决方案 |
|------|----------|
| ThreadLocal 未清理 | `finally { ThreadLocal.remove(); }` |
| JDBC 驱动未注销 | `DriverManager.deregisterDriver(driver);` |
| 后台线程未停止 | ServletContextListener.contextDestroyed() 里 shutdown |
| 定时器未取消 | `Timer.cancel()` / `ScheduledExecutorService.shutdown()` |
| Log4j Appender 缓存 | `LogManager.shutdown()` |

```java
@WebListener
public class CleanupListener implements ServletContextListener {
    
    @Override
    public void contextDestroyed(ServletContextEvent sce) {
        // 1. 清理 ThreadLocal
        MyThreadLocal.clear();
        
        // 2. 注销 JDBC 驱动
        Enumeration<Driver> drivers = DriverManager.getDrivers();
        while (drivers.hasMoreElements()) {
            DriverManager.deregisterDriver(drivers.nextElement());
        }
        
        // 3. 停止后台线程
        MyBackgroundTask.shutdown();
        
        // 4. 关闭日志
        LogManager.shutdown();
        
        // 5. 清理缓存
        CaffeineCache.invalidateAll();
    }
}
```

## 三、话题二：自定义 ClassLoader 做插件隔离

### 3.1 项目场景

```
SaaS 平台需要支持客户上传自定义插件：
    
    ┌─────────────────────────────┐
    │         主应用（平台）        │
    │  ┌─────────────────────┐    │
    │  │   App ClassLoader   │    │
    │  │   (平台核心代码)      │    │
    │  └─────────────────────┘    │
    │            ↑                │
    │     接口层（API包）          │
    │            ↑                │
    │    ┌───────┴───────┐       │
    │    ▼               ▼       │
    │ ┌─────────┐   ┌─────────┐  │
    │ │ Plugin-1│   │ Plugin-2│  │
    │ │ClassLoader│  │ClassLoader│ │
    │ │ (客户A)  │   │ (客户B)  │  │
    │ └─────────┘   └─────────┘  │
    └─────────────────────────────┘

需求：
    - 客户 A 和客户 B 可能上传同名类（如 com.custom.ReportGenerator）
    - 两个插件的依赖版本可能冲突（如 A 用 Guava 20，B 用 Guava 30）
    - 插件崩溃不能影响主应用
```

### 3.2 架构设计

```java
/**
 * 插件隔离架构
 * 
 * 双亲委派模型被打破：子 ClassLoader 优先加载自己的类
 */
public class PluginClassLoader extends URLClassLoader {
    
    private final ClassLoader parent;  // 平台 ClassLoader
    
    public PluginClassLoader(URL[] urls, ClassLoader parent) {
        super(urls, parent);
        this.parent = parent;
    }
    
    /**
     * 破坏双亲委派：先自己加载，找不到再委托父加载器
     * 
     * 原因：插件的依赖（如 Guava）必须由插件自己的 ClassLoader 加载
     *       如果委托父加载器，可能加载到平台版本的 Guava，导致版本冲突
     */
    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        // 1. 平台接口类：委托父加载器（保证类型一致）
        if (name.startsWith("com.platform.api")) {
            return parent.loadClass(name);
        }
        
        // 2. JDK 类：委托父加载器（安全沙箱）
        if (name.startsWith("java.") || name.startsWith("javax.")) {
            return super.loadClass(name, resolve);
        }
        
        // 3. 插件自己的类：优先自己加载（破坏双亲委派）
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                c = findClass(name);  // 自己加载
            } catch (ClassNotFoundException e) {
                c = super.loadClass(name, resolve);  // 委托父加载器
            }
        }
        
        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}
```

### 3.3 关键设计点

| 设计点 | 说明 |
|--------|------|
| **接口层由父加载器加载** | `com.platform.api.PluginInterface` 必须在主应用和插件中都是同一个 Class 对象，否则 `instanceof` 判断会失败 |
| **实现层由子加载器加载** | `com.plugin.ReportGenerator` 由 PluginClassLoader 加载，不同插件互不影响 |
| **JDK 类委托父加载器** | 安全要求，不能自定义加载 `java.lang.*` |
| **依赖版本隔离** | 每个插件有自己的 lib 目录，Guava、Jackson 等依赖互不影响 |

### 3.4 实际代码

```java
public class PluginManager {
    
    private final Map<String, PluginClassLoader> pluginLoaders = new ConcurrentHashMap<>();
    
    /**
     * 加载插件
     */
    public void loadPlugin(String pluginId, Path pluginDir) {
        // 1. 收集插件的 jar 文件
        URL[] urls = Files.list(pluginDir.resolve("lib"))
            .filter(f -> f.toString().endsWith(".jar"))
            .map(f -> f.toUri().toURL())
            .toArray(URL[]::new);
        
        // 2. 创建独立的 ClassLoader
        PluginClassLoader loader = new PluginClassLoader(urls, this.getClass().getClassLoader());
        pluginLoaders.put(pluginId, loader);
        
        // 3. 加载插件入口类
        Class<?> clazz = loader.loadClass("com.plugin.EntryPoint");
        PluginInterface plugin = (PluginInterface) clazz.getDeclaredConstructor().newInstance();
        
        // 4. 初始化插件
        plugin.init();
    }
    
    /**
     * 卸载插件（关键！必须清理引用）
     */
    public void unloadPlugin(String pluginId) {
        PluginClassLoader loader = pluginLoaders.remove(pluginId);
        if (loader != null) {
            // 1. 调用插件的销毁方法
            // 2. 清理缓存、线程、监听器
            // 3. 关闭 ClassLoader
            loader.close();  // Java 7+ URLClassLoader 支持 close()
            
            // 4. 建议 GC（不强制）
            System.gc();
        }
    }
}
```

## 四、话题三：SPI 机制打破双亲委派

### 4.1 问题场景

```java
// JDBC 4.0+ 不需要 Class.forName()，为什么还能自动加载驱动？
Connection conn = DriverManager.getConnection("jdbc:mysql://localhost/test");
```

### 4.2 原因：SPI（Service Provider Interface）

```
JDBC 的类加载器困境：

    DriverManager 在 JDK 中（Bootstrap ClassLoader 加载）
        │
        ├── 需要加载 com.mysql.cj.jdbc.Driver
        │
        └── 但 MySQL 驱动在 classpath 里（App ClassLoader 加载）
        
    双亲委派模型下：
        Bootstrap ClassLoader 委托 Ext → App
        App ClassLoader 能加载 MySQL 驱动
        但 DriverManager.getConnection() 内部调用驱动时，
        用的是当前线程的 ContextClassLoader
```

**解决方案：线程上下文类加载器（Thread Context ClassLoader）**

```java
// DriverManager 的源码简化版
public static Connection getConnection(String url) {
    // 获取当前线程的上下文类加载器（通常是 App ClassLoader）
    ClassLoader cl = Thread.currentThread().getContextClassLoader();
    
    // 用上下文类加载器加载驱动，而不是 DriverManager 自己的 Bootstrap ClassLoader
    ServiceLoader<Driver> loadedDrivers = ServiceLoader.load(Driver.class, cl);
    
    for (Driver driver : loadedDrivers) {
        if (driver.acceptsURL(url)) {
            return driver.connect(url, info);
        }
    }
}
```

**本质**：

```
双亲委派模型被 SPI 打破：
    
    父 ClassLoader（Bootstrap）
        │
        ├── 加载了 DriverManager
        │
        └── DriverManager 需要用【子 ClassLoader（App）】加载的 Driver
        
    正常双亲委派：子不能访问父加载的类（方向是自上而下的）
    SPI 方案：父通过 Thread.currentThread().getContextClassLoader() 
             逆向使用子 ClassLoader 的能力
```

### 4.3 自定义 SPI 示例

```java
// 1. 定义接口（放在主应用）
public interface PaymentPlugin {
    String pay(BigDecimal amount);
}

// 2. 插件实现（放在插件 jar）
public class AlipayPlugin implements PaymentPlugin {
    public String pay(BigDecimal amount) {
        return "Alipay: " + amount;
    }
}

// 3. SPI 配置文件（插件 jar 的 META-INF/services/com.platform.PaymentPlugin）
// 内容：com.plugin.AlipayPlugin

// 4. 主应用加载
ServiceLoader<PaymentPlugin> loader = ServiceLoader.load(PaymentPlugin.class);
for (PaymentPlugin plugin : loader) {
    plugin.pay(new BigDecimal("100"));
}
```

## 五、面试 2 分钟口述版（无实战经验版）

> "我在项目中确实没有深度踩过类加载的坑，但我深入理解它的原理和经典场景。
>
> 最经典的是 **Tomcat 热部署导致的 Metaspace OOM**。Tomcat 给每个 Web 应用一个独立的 ClassLoader，热部署时丢弃旧的、创建新的。但如果应用里有 ThreadLocal 没清理、JDBC 驱动没注销、或者后台线程没停止，这些长生命周期的对象会间接引用旧的 ClassLoader，导致它和加载的所有类都无法被 GC，最终 Metaspace 溢出。解决方案是在 ServletContextListener 的 contextDestroyed 里做彻底清理。
>
> 另一个实际应用是 **插件隔离架构**。我们做过一个 SaaS 平台，允许客户上传自定义插件。不同客户可能上传同名类或不同版本的依赖（如 Guava）。我们用自定义 ClassLoader 破坏双亲委派，插件自己的类优先自己加载，但平台接口委托父加载器，保证 instanceof 判断一致。卸载插件时必须清理所有引用，否则同样会有类加载器泄漏。
>
> 还有一个点是 **SPI 机制打破双亲委派**。JDBC 的 DriverManager 在 JDK 里由 Bootstrap ClassLoader 加载，但 MySQL 驱动在 classpath 里由 App ClassLoader 加载。DriverManager 通过 Thread.currentThread().getContextClassLoader() 逆向使用子加载器的能力，这是双亲委派模型的一个重要例外。"

## 六、速记图

```
类加载问题三大场景
    │
    ├── Tomcat 热部署内存泄漏
    │       │
    │       ├── 根因：旧 ClassLoader 被长生命周期对象引用（ThreadLocal、线程、DriverManager）
    │       │
    │       └── 解决：contextDestroyed 里彻底清理所有引用
    │
    ├── 自定义 ClassLoader 插件隔离
    │       │
    │       ├── 破坏双亲委派：子加载器优先加载自己的类
    │       │
    │       ├── 接口层委托父加载器：保证类型一致（instanceof 正确）
    │       │
    │       └── 卸载时 close() + 清理引用 + System.gc()
    │
    └── SPI 打破双亲委派
            │
            ├── 父加载的类（DriverManager）需要加载子加载器的类（MySQL Driver）
            │
            └── 方案：Thread.currentThread().getContextClassLoader()
```
