# Agent 记忆管理 — 面试深度 QA

> 面试题：Agent 的记忆管理怎么设计？短期记忆、长期记忆、外部记忆各自的存储方案和检索策略是什么？
> 候选人背景：8年 Java 后端，智能运维 Agent + RAG 知识库 + 自研流程引擎

---

## 一、标准答案框架

Agent 的记忆管理本质上解决一个问题：**在有限的上下文窗口里，给模型最相关的信息。** 三类记忆各司其职。

### 1.1 短期记忆（Short-term Memory）

**是什么**：当前对话的上下文，一次会话内有效，会话结束即丢弃。

**存储方案**：

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| 全量保留 | 把所有对话历史塞进 context | 对话轮次少（< 10 轮） |
| 滑动窗口 | 只保留最近 N 轮 | 对话长但早期内容不重要 |
| 摘要压缩 | 每隔 N 轮用 LLM 总结前文，用摘要替代原始对话 | 长对话且早期内容有持续影响 |
| 分层压缩 | 近期全量 + 远期摘要 + 关键信息提取 | 生产系统首选 |

**检索策略**：短期记忆不需要检索——直接作为 LLM 的 context 输入。核心挑战是**压缩**。

**工程要点**：

- Token 预算分配：context window 128K 时，系统 Prompt 约 2K，工具定义约 5K，短期记忆预算约 30K，剩余留给工具返回结果和生成空间
- 角色消息（system/user/assistant）的结构化存储，方便做选择性截断

### 1.2 长期记忆（Long-term Memory）

**是什么**：跨会话持久化的信息——用户偏好、历史交互摘要、Agent 学到的知识。

**存储方案**：

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| 向量数据库 | Embedding 后存入 Qdrant/Milvus，语义检索 | 记忆条目多、需要模糊匹配 |
| 关系数据库 | 结构化字段存储，精确查询 | 记忆有明确 schema |
| 混合方案 | 向量库管语义检索，关系库管结构化查询 | 生产系统首选 |

**检索策略**：

- 语义检索：当前对话 Embedding → Top-K 相似记忆 → 注入 context
- 时间衰减：近期记忆权重更高，`score × decay(days)`
- 相关性 + 重要性双维度：有些记忆和当前话题不相关但很重要

**工程要点**：

- 写入时机：每轮对话结束后异步提取关键信息写入，不阻塞响应
- 去重：同一信息多次提到时 merge 而不是重复存储
- 淘汰：LRU 或基于访问频率，防止无限膨胀

### 1.3 外部记忆（External Memory）

**是什么**：Agent 的知识库——文档、数据库、API 返回的结构化数据。不属于 Agent 自身，但可检索引用。

**存储方案**：就是 RAG 系统——文档分块、Embedding 后存入向量库，保留原始文档的结构化元数据。

**检索策略**：

- Dense + BM25 + RRF 融合
- 查询重写（HyDE）提高召回率
- Re-rank：初筛 Top-20 → Cross-Encoder 精排 → 取 Top-5

### 1.4 三类记忆的协作流程

```
用户输入
  │
  ├─① 短期记忆：加载当前会话的对话历史
  ├─② 长期记忆：语义检索历史偏好/摘要
  ├─③ 外部记忆：RAG 检索相关知识 chunks
  │
  ▼
组装 context = system prompt + 长期记忆 + 外部记忆 + 短期记忆 + 用户输入
  │
  ▼
LLM 推理 → 工具调用 → 结果回传 → 更新短期记忆
  │
  ▼
异步：提取本轮关键信息 → 写入长期记忆
```

## 二、结合项目的真实回答（关键：诚实分析自己的系统）

### 2.1 你的三个模块的记忆需求分析

| 模块 | 交互模式 | 记忆需求 | 实际存储 |
|------|---------|---------|---------|
| 运维 Agent | 告警事件 → 单轮推理 → 执行 | 几乎无 | 告警上下文（结构化 JSON） |
| RAG 知识库 | 用户提问 → 检索 → 生成 | 几乎无 | 查询结果（即查即用） |
| 流程引擎 | 有状态工作流 | 有状态，但不是"记忆" | 流程实例的持久化状态（数据库行） |

**结论**：三个模块都是**无对话状态**或**确定性状态管理**，不需要"记住用户说过什么偏好"这种 Agent 长期记忆。

### 2.2 为什么不需要？TOB SaaS 的"记忆"长什么样

你的系统服务的是**企业用户操作固定流程**，不是**个人用户的开放式对话**：

- **对话式 Agent**（ChatGPT、个人助手）：用户说"我上次问的那个问题"，系统得知道"上次"是什么 → 需要长期记忆
- **流程式 Agent**（你的运维 Agent）：告警来了就处理，处理完就结束，不需要知道上次告警时用户说了什么 → 不需要长期记忆

**TOB SaaS 的"长期记忆"其实是配置和数据，不是 Agent Memory：**

| 需求 | 实现方式 | 为什么不用 Agent 记忆 |
|------|---------|---------------------|
| 租户 A 偏好什么告警阈值 | 租户配置表（MySQL） | 确定性查询比语义检索更可靠 |
| 租户 B 的历史告警趋势 | 监控数据库（Prometheus） | 结构化时序数据，SQL 聚合更高效 |
| 租户 C 的自定义处理模板 | 配置中心（Nacos） | 运行时热加载，不需要 Agent 记住 |
| 租户的领域术语映射 | 租户级同义词库 | 写在配置里，不是 Agent 学到的 |

这些都是**确定性数据管理**，比 Agent 记忆可靠得多。

### 2.3 但系统有类似"长期记忆"的设计（面试话术）

"我的运维 Agent 是**单轮事件驱动**的——告警进来，处理完就结束，不存在多轮对话。所以不需要传统的对话记忆。

但系统有类似'长期记忆'的设计，只是实现方式不同：

**第一，租户级告警模式统计。** 每个租户的历史告警频率、高频故障类型，存在 MySQL 里，用于根因分析时的上下文补充。比如某个租户一周内第 3 次 MQ 堆积，Agent 会优先排查消费端配置变更。这本质上就是长期记忆，但用结构化查询实现，比向量检索更精确。

**第二，租户级同义词库。** RAG 知识库为每个租户维护领域术语映射，比如'灭火器'→'消防器材'。这个库随使用自动积累，本质上也是一种长期记忆。

**第三，Bad Case 归集。** 每次人工介入的案例都会归档分析，优化 Prompt 模板和工具调用策略。这相当于 Agent 的'经验积累'，只是不是实时的，而是离线迭代的。"

### 2.4 如果要做长期记忆的方案（加分项）

"当前系统没有用到 Agent 记忆，但如果要加，我会这样做——

**场景**：运维 Agent 的告警模式分析。如果某个租户频繁出现同一类告警（比如每周一上午 MQ 堆积），Agent 应该能'记住'这个模式，在新告警来临时主动关联历史上下文。

**实现**：

```
告警事件 → 规则匹配 → 查询该租户的历史告警模式（长期记忆）
                      → 如果近 7 天同类告警 > 3 次
                      → 注入 context: '该租户近 7 天已出现 3 次 MQ 堆积，
                        上次根因是消费端线程池配置变更'
                      → LLM 推理时参考历史模式，输出更精准的诊断
```

存储：MySQL 的告警记录表 + 应用层缓存（热点租户的近 7 天统计）。不需要向量库——告警类型是结构化枚举，SQL 聚合比语义检索更高效。"

## 三、加分项（体现架构师思维深度）

1. **主动提 Token 预算分配。** "128K 的 context window 不是全给对话历史的——要给系统 Prompt、工具定义、工具返回结果、生成空间都留预算。"——说明有工程化的资源管理意识。

2. **提到记忆的写入和检索是异步的。** "记忆写入不能阻塞响应——用户问完问题，Agent 先返回结果，异步协程再提取关键信息写入长期记忆。"——说明理解生产系统的性能要求。

3. **结合自己项目讲"为什么不需要"。** "我的场景是单轮事件驱动，不需要对话记忆。但 TOB SaaS 有自己的'记忆'方式——配置表、历史统计、同义词库——这些比 Agent 记忆更可靠。"——说明有工程判断力，不会为了用新技术而用新技术。

4. **提到记忆安全和隔离。** "多租户场景下，记忆必须严格隔离——租户 A 的记忆绝不能出现在租户 B 的 context 里。向量检索时必须加 tenant_id 过滤，应用层还要做二次校验。"——这是你的 RAG 项目的真实经验。

5. **讲一个"如果做"的方案。** "如果要加长期记忆，我会用告警模式统计——MySQL 存历史告警 + 应用层缓存热点租户数据。不需要向量库，结构化查询更高效。"——展示设计能力，同时保持务实。

## 四、追问应对

### Q1：对话窗口太长时，你会用滑动窗口还是摘要压缩？

"选摘要压缩，但不是全用，而是**分层策略**：

```
最近 5 轮：全量保留（精确上下文）
5-20 轮前：每 5 轮压缩成一段摘要（保留关键决策和结论）
20 轮以前：只保留结构化摘要（用户画像、关键决策点、未完成的任务）
```

| 方案 | 延迟 | 成本 | 信息保真度 | 实现复杂度 |
|------|------|------|-----------|-----------|
| 滑动窗口 | 0 | 0 | 低（丢信息） | 低 |
| 全量摘要 | 高（每轮调 LLM） | 高 | 中（摘要有损） | 中 |
| 分层压缩 | 中 | 中 | 高（关键信息保留） | 高 |

**关键细节**：摘要压缩本身也要调 LLM，所以要**异步执行**——用户发消息时先用当前上下文响应，摘要压缩在后台完成，下次请求时再用压缩后的结果。"

> **结合项目**："我的运维 Agent 不涉及长对话，但流程引擎有类似场景——一个流程实例可能运行几天甚至几周，中间经历多个节点。状态持久化用的是主键更新 + 乐观锁，每个节点完成后把状态写入数据库。流程恢复时从数据库加载最新状态，不需要在 context 里保留完整历史。这其实是一种确定性的'压缩'——只保留当前状态，不保留历史过程。"

### Q2：长期记忆用向量检索的话，怎么处理记忆的更新和淘汰？

"三个问题：更新、去重、淘汰。

**更新**——同一个信息可能变化。比如用户说过'我在 A 公司'，后来又说'我跳槽到 B 公司'。用记忆的 key（用户 ID + 主题）做 upsert，新记忆覆盖旧记忆，但保留历史版本方便回溯。

**去重**——同一个信息在多次对话中被提到，Embedding 后可能生成多条相似记忆。写入前先做相似度检索，如果已有 > 0.95 相似度的记忆，就 merge 而不是新增。

**淘汰**——三个策略组合：
- LRU：超过 N 天未被检索到的记忆降级
- 访问频率：被检索到但从未被注入 context 的记忆，说明相关性低
- 硬上限：每个用户的记忆条目设上限（比如 1000 条），超限淘汰最旧的"

> **结合项目**："我的 RAG 知识库有类似的淘汰逻辑——租户的同义词库需要定期清理不再使用的术语。但我们用的是更简单的策略：按最后使用时间排序，超过 90 天未命中的同义词条目标记为候选删除项，由租户管理员确认后删除。比 Agent 记忆的自动淘汰更保守，因为 TOB 场景误删的代价更高。"

### Q3：多轮对话的上下文截断策略有哪些？

"从粗到细，四种策略：

**策略一：按轮次截断**——最简单，只保留最近 N 轮。问题：第 N+1 轮可能有重要信息。

**策略二：按 Token 截断**——从最新消息往回数，直到 Token 预算用完。比按轮次更精确，但仍然是时间序截断，不区分重要性。

**策略三：按重要性截断**——每条消息标注重要性分数（包含决策的消息分数高，闲聊分数低），优先保留高分消息。用规则（包含关键词、是 user 消息、包含数字/结论）或小模型打分。

**策略四：选择性注入**——把所有历史消息 Embedding 存入向量库，当前 query 做语义检索，只注入 Top-K 相关的历史片段。效果最好但成本最高。

**生产系统通常是策略二 + 策略三的组合**：先按 Token 预算截断，在预算内按重要性排序。"

> **结合项目**："我的运维 Agent 不涉及多轮对话，但告警收敛是一个类似的场景——同一个故障可能触发 100 条告警，我们需要把它们收敛成一个'故障事件'。这本质上也是一种'上下文截断'——把 100 条告警压缩成 1 条结构化的故障摘要，只保留：故障类型、影响范围、首次发生时间、最新状态。和对话压缩的思路是一样的。"

│  → 结果: 近 7 天第 3 次 MQ 堆积                                      │
│  → 注入 context: "该租户近期频繁出现 MQ 堆积，建议排查消费端配置变更"   │
└──────────────────────────────┬───────────────────────────────────────┘
                               │
                               ▼
┌──────────────────────────────────────────────────────────────────────┐
│              ③ LLM 推理（根因分析）                                   │

│              → 诊断结论 → 处置建议                                    │

│              ④ 动作执行 + 异步写入                                    │
│  执行: 服务重启 / 流量切换 / 资源扩容                                 │
│  异步: 本次告警记录写入 alert_history（供下次查询）                    │
└──────────────────────────────────────────────────────────────────────┘

  关键设计: "记忆"用 MySQL 结构化查询，不用向量检索
  原因: 告警类型是有限枚举，SQL 聚合比语义检索更精确、更快
  效果: 人工介入率 100% → 15%，MTTR 45min → 8min

## 六、面试策略总结

| 场景 | 怎么说 |
|------|--------|
| 面试官问通用方案 | 讲三类记忆 + 存储方案 + 检索策略，展示知识广度 |
| 面试官问你的项目 | 诚实说"单轮事件驱动不需要传统 Agent 记忆"，但讲三个类比设计 |
| 面试官追问为什么不做 | 讲 TOB SaaS 的"记忆"是配置和数据，比 Agent 记忆更可靠 |
| 面试官追问如果要做 | 讲告警模式统计方案（MySQL + 应用层缓存，不需向量库） |
| 面试官问记忆隔离 | 用 RAG 多租户隔离的真实经验回答（tenant_id 过滤 + 二次校验） |

**核心原则：不要为了显得"懂 Agent"而硬凑记忆概念。诚实分析自己的系统，讲清楚"为什么不需要"本身就是架构师的判断力。把面试官引向你真正擅长的领域——多租户隔离、缓存降本、状态持久化。**
