# 应急安全管理 RAG 项目（SaaS 多租户版）— 面试口述稿

> 基于应急安全领域 SaaS 多租户场景，每个客户数据隔离（A 区与 B 区不混用）。
> 总字数约 2600 字，口述语速 180 字/分钟，全程约 14~15 分钟。
> **加粗部分为必须记住的记忆锚点和关键数字。**

---

## 开场白（20 秒）

我介绍一下我之前做的 RAG 项目。这是一个**应急安全管理的 SaaS 知识库系统**，服务对象是各类园区、写字楼、工厂的安全管理团队。

核心特点是**多租户隔离**——A 客户的数据和 B 客户的数据完全物理隔离，不能混用。所以虽然平台总文档量有 5 万份，但**单个租户通常只有几百到两千份**，是一个典型的"小数据量、高隔离、高并发"场景。

另外我要特别说明，整个架构选型**成本是第一优先级**。我们是 SaaS 创业公司，不想在软件授权和运维人力上花太多钱，所以全部走的开源方案。

---

## 一、项目背景与规模（1 分钟）

数据源对每个租户来说有三类。

第一类是**静态文档**，安全操作手册、消防规范、逃生通道平面图、应急预案。单个租户通常 **200~2000 份**，PDF 和扫描件居多。

第二类是**实时上报数据**，IoT 传感器告警、人员签到记录、巡检隐患描述。单个租户每天新增 **几百到几千条**。

第三类是**空间资产数据**，楼层信息、区域编号、逃生通道坐标、消防设施位置，存在租户的独立 MySQL 里。

做这个项目的背景是**竞品陆续上线了 AI 知识库功能**，市场倒逼我们必须补齐这块能力，否则客户续约会受影响。我在团队里负责整个 **RAG 检索链路和多租户数据架构**，从 0 到 1 主导架构设计。团队 6 个人，我是技术负责人。核心目标三个：**端到端延迟 P99 小于 800 毫秒**；**精确信息准确率 100%**；**支持 200 个活跃租户同时在线**。

---

## 二、技术架构选型：成本是第一优先级（1.5 分钟）

我先说选型逻辑，再讲架构。**我们所有的技术选型，第一条都是"有没有开源免费的"，第二条才是"性能够不够"**。

### 为什么选 Qdrant 而不是 Milvus？

Milvus 是分布式架构，需要 Etcd、MinIO、Pulsar 一堆组件，光运维人力就是一笔成本。Qdrant 是 Rust 写的，**单机 Docker 一行启动，零外部依赖**，对于我们这种小团队来说省了一个专职运维的人头。

另外 Milvus 的分布式能力我们根本用不上——单租户就几百份文档，Qdrant 单机撑几百个租户完全没问题。

### 为什么选 Flat 而不是 HNSW？

HNSW 需要调参（M、efConstruction、ef），调不好召回率会掉，调参本身花时间。Flat 索引**零参数、零维护**，对于 2000 条以内的数据，查询延迟也就 1 毫秒，和 HNSW 差不了多少，但省了大量调优人力。

我们只在单租户超过 5000 份文档时才切 HNSW，目前 120 个租户里有 4 个大客户触发（4 家世界 500 强企业，文档量较大）。

### 为什么用 ES 而不是商用搜索引擎？

ES 开源、生态成熟、社区文档多。商用方案比如阿里云 OpenSearch 按量计费，我们算过账，**120 个租户跑一年，商用方案比自建 ES 贵 8 到 10 倍**。

### 为什么双轨架构？

我们试过让 Qdrant 同时做 Sparse + Dense，但它的 Hybrid Search 还不够成熟。与其赌一个数据库把两件事都做好，不如**让 Qdrant 管 Dense、ES 管 BM25，应用层做 RRF 融合**——两边都是各自领域的开源标杆，风险最低。

### 缓存层怎么设计的？

缓存我们只做了三层，全部在应用层，**向量检索层本身不缓存**。

原因是向量检索已经够快了——Qdrant Flat 1 毫秒、ES BM25 5 毫秒、MySQL 2 毫秒、RRF 融合 1 毫秒，一共 9 毫秒。Redis 缓存 lookup 也要 2 毫秒，收益只有 7 毫秒，还要处理缓存失效和脏数据，ROI 太低。

**真正值得缓存的是 LLM 生成层**，因为那是链路里最贵、最慢的环节。我们做了三层：

**第一层，Query 答案缓存**，放在 API Gateway，Key 是 tenant_id 加 query 归一化后的 hash。应急类 TTL 1 小时，常规类 24 小时。**命中率 55%**。应急演练时几十个人问同一个"3 楼火灾怎么办"，这层直接短路整条链路，延迟从 800 毫秒降到 5 毫秒。

**第二层，HyDE 缓存**，放在查询重写后。Key 是 tenant_id 加原始 query hash，TTL 4 小时，命中率 30%。避免重复的 HyDE LLM 调用。

**第三层，LLM 响应缓存**，放在 LLM 调用前。Key 是 tenant_id 加 prompt hash，TTL 1 小时。**命中率 40%**。Prompt 包含了检索回来的 chunks，如果文档没更新，同样的 prompt 直接复用 LLM 结果。

三层叠加后，**整体 LLM 调用成本降了 60%，平均延迟从 800 毫秒降到 220 毫秒**。

## 三、多租户架构设计（1 分钟）

多租户隔离是架构的第一性原则。

我们评估了三种方案：独立实例、共享实例加分区、共享实例加标量过滤。独立实例运维噩梦，排除。最终选了**共享 Collection + Payload 过滤**。

Qdrant 的 tenant_id 放在 Payload 里，查询时强制过滤。ES 同理，tenant_id 作为字段，每次查询带过滤条件。应用层再加一层二次校验，确保检索结果的 tenant_id 和当前租户一致。

上线时自动为新租户分配 tenant_id，onboarding 时间从原来的 2 天压缩到 **5 分钟**。

## 四、核心数字（1.5 分钟）

系统上线跑了 3 个月，核心数字我列一下。

**租户规模**：活跃租户 **120 个**（含 4 家世界 500 强企业）。

**QPS**：平台峰值 **250**。

**端到端延迟 P99**：**720 毫秒**。检索链路 120 毫秒，LLM 生成 550 毫秒。

**Recall@10**：**97%**。

**Faithfulness**：**0.90**。

**精确信息准确率**：**99.8%**。

**用户满意度**：点赞率 **87%**。

**单租户月均基础设施成本**：**25 元**。 breakdown 一下：Qdrant 和 ES 跑在共享服务器上，月费 3000 元，摊到 120 个租户约 25 元；LLM 调用因为有缓存，月均 5 元。单租户总成本约 30 元。

**缓存效果**：没有缓存时，单次请求成本约 $0.025（LLM $0.02 + HyDE $0.005）。加三层缓存后，Query 答案缓存命中 55%，LLM 响应缓存再命中 40%，实际 LLM 调用量只剩 27%。**月均 LLM 调用成本降 82%**。

> 我们算过账，如果用 Milvus 分布式集群 + 商用搜索服务 + 全量 GPT-4 无缓存，**单租户月均成本会涨到 80 元以上**，我们的 SaaS 定价模型根本撑不住。

## 五、最头疼的 Bad Case（4 分钟）

### Bad Case 1：Retrieval 问题 —— 新租户冷启动，检索质量差

**场景**：新租户刚上传了 50 份安全手册，用户问"灭火器怎么用"，系统返回了完全不相关的条款。

**排查**：检查检索结果，发现召回的 chunks 与 query 的相似度都很低，最高只有 0.45。确认是 Retrieval 问题。

**根因**：**冷启动 + 数据量太小**。50 份文档里只有 3 份提到灭火器，而且表述都是"消防器材"而非"灭火器"。Bi-Encoder 没学过这个租户的领域词汇，Dense 检索泛化失败。同时新租户的 BM25 Index 刚建好，IDF 统计不准确。

**修复动作**：三步。第一步，**租户级同义词库**。每个租户 onboarding 时，自动从文档中提取高频术语，建立该租户专用的同义词扩展表。"灭火器"自动扩展"消防器材、灭火设备"。第二步，**冷启动保护期**。新租户前 7 天，检索策略从纯 Dense 切换到 BM25 权重提高、Dense 权重降低，等数据积累够了再恢复正常 Hybrid。第三步，**文档质量检测**。onboarding 时自动检测文档覆盖率——如果某类安全主题的文档量少于 2 份，系统提醒租户补充。

**结果**：新租户首周 Faithfulness 从 0.55 提到 **0.82**，7 天后稳定到 **0.90**。

### Bad Case 2：Generation 问题 + 成本事故 —— LLM 调用被攻击者刷爆

**场景**：某天凌晨，某个租户遭到异常流量攻击，短时间内发起了 2000 次相同 query，当月 LLM 调用费用直接涨了 **300%**。

**排查**：日志显示 query 内容正常（"灭火器在哪"），但频率异常——同一个 IP，1 分钟内 2000 次。

**根因**：**缺少租户级限流和缓存**。攻击者的重复 query 每次都走了完整的 LLM 调用，没有任何限速和缓存拦截。

**修复动作**：三步，而且都围绕**省钱和兜底**。第一步，**API 网关层加租户级 QPS 限流**，单租户超过 20 QPS 直接熔断，返回"请求过于频繁"。第二步，**Redis 缓存层加租户隔离**，相同 query（同一租户、同一 hash）命中缓存后直接返回，不走 LLM。Query 答案缓存命中率从 0 提到 **55%**。第三步，**LLM 层加 Token 配额**，每个租户每月设上限，超过后降速或只走缓存。另外加了**布隆过滤器**拦截异常 query 模式，防止缓存穿透。

这三步做完，**整体 LLM 调用成本降了 60%**。更重要的是建立了**缓存 + 限流 + 配额**的三层防御体系，后续零类似事故。

**结果**：零类似事故复发。单租户月均 LLM 成本从 15 元压到 **5 元**。

## 六、当前不足与后续规划（1.5 分钟）

现在系统还有三个问题在排期。

**第一，大客户性能抖动**。4 家世界 500 强企业文档量较大，其中 4 个租户的文档量超过了 5000 份，Flat 索引的延迟从 1 毫秒涨到了 8 毫秒。规划给这 4 个大客户单独切到 HNSW，中小客户继续 Flat。这是一个**自动弹性索引选型**的策略，**只有超阈值才花额外的运维和内存成本**。

**第二，租户间知识复用**。不同租户的安全手册有大量重复内容，比如《消防法》条款。现在每个租户独立存储，浪费存储。**规划做去重层**——公共法规类文档只存一份，tenants 通过引用关联，检索时自动展开。预计存储成本能降 **30%**。

**第三，模型蒸馏降本**。我们正在用 GPT-4 的输出训练本地 Qwen-14B，目标是 6 个月后中小租户可以切换到本地模型，**LLM 调用成本从月均 5 元降到 0.5 元**（只需要 GPU 推理电费）。大客户继续用 GPT-4。

## 七、收尾（20 秒）

总结下来，这个项目的核心挑战不是技术复杂度，而是**在 SaaS 创业公司的成本约束下，做到安全合规和性能的平衡**。我们的解法是：**全部开源方案、Flat 索引省运维、缓存限流省 LLM 成本、三级校验防泄露**。单租户月均基础设施成本 25 元，P99 延迟 720 毫秒，精确信息准确率 99.8%。

## 附录：高频追问 & 标准应答

**Q：你们为什么不用 Milvus？它功能更全。**
> Milvus 需要 Etcd、MinIO、Pulsar 一堆组件，运维人力是一笔成本。我们团队 6 个人，没有专职运维。Qdrant 单机 Docker 一行启动，单租户几百份文档完全够用。功能全不重要，**适合我们团队规模和成本预算才重要**。

**Q：Flat 索引能撑多久？未来数据量大了怎么办？**
> 我们设了阈值：**单租户 < 5000 条走 Flat，> 5000 条自动切 HNSW**。目前 120 个租户只有 4 个触发（4 家世界 500 强企业）。即使未来有更多大客户，也可以只给这些大客户开 HNSW，中小客户继续 Flat，**成本是按租户粒度弹性分摊的**。

**Q：单租户月均成本怎么算出来的？**
> 拆解一下：服务器是一台 8 核 32G 的云主机，跑 Qdrant + ES + Redis + 应用，月费 3000 元，摊到 120 个租户，**每个租户基础设施约 25 元**。LLM 调用因为有缓存和限流，月均 5 元。加起来约 30 元。如果用 Milvus 集群 + 商用搜索 + 全量 GPT-4，我们测算过要 **100 元以上**，SaaS 定价模型撑不住。

**Q：如果 Qdrant 或 ES 出现严重 Bug，你们怎么办？**
> 两个策略。第一，版本锁定，不追新，用社区验证稳定的 LTS 版本。第二，数据定期快照备份，万一出问题可以切到备用实例。另外 ES 的替代方案很多（OpenSearch、Meilisearch），Qdrant 也可以被 Milvus 或 Weaviate 替代，**架构上没有强绑定，都是标准接口**。

**Q：你们怎么保证 99.8% 的精确信息准确率？**
> 结构化路径兜底。楼层、区域、设备编号这些关键信息不走 RAG，走 SQL 查询空间资产库。RAG 只负责"怎么做"的语义部分。另外后处理加了楼层一致性校验——如果 RAG 返回的逃生通道楼层和 query 楼层不一致，触发告警并返回"请核实"。**这条规则的代码比模型训练便宜一万倍，但效果更稳**。

**Q：为什么不做模型微调，而是直接用通用 BGE？**
> 算过账。微调需要标注数据、GPU 训练、持续维护，一次性投入 2 到 3 万人力成本，后续每次迭代还要投入。通用 BGE-large 在应急安全领域的 Recall 已经 97%，微调可能提到 98%，**1% 的提升不值得 3 万块的投入**。等数据量大了、通用模型不够用了再考虑微调。

**Q：向量检索层为什么不加缓存？**
> 因为已经够快了。Qdrant Flat 1 毫秒、ES 5 毫秒、MySQL 2 毫秒、RRF 1 毫秒，一共 9 毫秒。Redis 缓存 lookup 也要 2 毫秒，收益只有 7 毫秒，还要处理缓存失效、穿透、文档更新后的脏数据，ROI 太低。**缓存的原则是"哪里慢、哪里贵、哪里重复，就放哪里"**。LLM 生成 550 毫秒 + $0.02 一次，才是值得缓存的地方。
