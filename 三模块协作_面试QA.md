# 应急安全智能平台 — 三模块协作与多租户隔离 — 面试深度 QA

> 面试题：你在应急安全智能平台中设计了三个核心模块（流程引擎、RAG 知识库、智能运维 Agent），它们之间是怎么协作的？共享的多租户隔离体系具体怎么实现？
> 候选人背景：8年 Java 后端，120 租户（含 4 家世界 500 强），三模块共享多租户体系
> **这道题考的是系统全局视角——不是单个模块的技术深度，而是三个模块怎么串成一个整体。**

---

## 一、最佳答案

### 1.1 先讲三模块各自解决什么问题

"三个模块解决的是应急安全管理中的三类不同问题：

| 模块 | 解决的问题 | 本质 | 举例 |
|------|-----------|------|------|
| **流程引擎** | 确定性业务流程 | 状态机驱动，每个节点做什么事先定义好 | 访客审批、应急演练流程、巡检任务派发 |
| **RAG 知识库** | 非结构化知识检索 | 语义检索 + LLM 生成，回答'怎么做'的问题 | '灭火器怎么用'、'3 楼火灾应急预案是什么' |
| **智能运维 Agent** | 线上故障自动诊断 | LLM 推理 + 工具调用，处理事先没预见的问题 | MQ 堆积自动诊断、DB 慢查询根因分析 |

三者不是并列关系，是**互补关系**——确定性流程用引擎，知识查询用 RAG，非确定性推理用 Agent。"

### 1.2 三模块怎么协作

"协作方式是**事件驱动 + 数据共享**，不是直接调用。每个模块独立运行，通过事件和共享数据协作：

**场景一：应急演练（流程引擎 + RAG 知识库）**

```
流程引擎驱动演练流程:
  ① 创建演练任务 → ② 通知相关人员 → ③ 执行疏散 → ④ 人员签到 → ⑤ 总结评估

执行到第③步时，参演人员可能问:
  "3 楼东侧逃生通道怎么走？"
  → 这个问题由 RAG 知识库回答（检索逃生通道平面图 + 应急预案）
  → 流程引擎不关心回答内容，只关心流程是否继续
```

**场景二：线上故障（智能运维 Agent + 流程引擎）**

```
Agent 检测到 MQ 堆积告警:
  ① Agent 自动诊断 → 根因: 消费端线程池打满
  ② Agent 触发自愈动作: 服务重启
  ③ 自动创建流程引擎工单: "MQ 消费异常 - 需要排查根因"
  ④ 流程引擎驱动后续流程: 通知值班人员 → 排查 → 修复 → 复盘

Agent 负责诊断和初步处理，流程引擎负责后续的流程闭环
```

**场景三：安全知识查询（RAG 知识库 + 流程引擎）**

```
用户在流程引擎的审批节点中:
  审批消防设备采购单 → 需要查看《消防设备配置标准》
  → 审批页面内嵌 RAG 知识库查询入口
  → 用户查询后返回审批页面继续操作
```

**协作的核心设计原则**：
- **松耦合**：三个模块通过事件总线（RabbitMQ）和共享数据（MySQL）协作，不直接调用对方的 API
- **独立部署**：每个模块可以独立扩缩容——RAG 的 QPS 高就单独加 RAG 实例，流程引擎的 TPS 高就加流程引擎实例
- **统一入口**：用户通过统一的 SaaS 平台入口访问，前端根据租户购买的模块动态展示功能菜单"

### 1.3 共享的多租户隔离体系

"三个模块共享同一套多租户基础设施，核心是**一个 tenant_id 贯穿所有模块**：

```
                    ┌──────────────────────┐
                    │   租户管理中心        │
                    │   tenant_id 分配      │
                    │   套餐/权限/配额      │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │  流程引擎     │  │  RAG 知识库   │  │  运维 Agent   │
  │              │  │              │  │              │
  │  instance_id │  │  chunk payload│  │  alert_event │
  │  带 tenant_id│  │  带 tenant_id │  │  带 tenant_id│
  └──────────────┘  └──────────────┘  └──────────────┘
```

**统一隔离层**：

| 隔离维度 | 实现方式 | 三个模块共享 |
|---------|---------|-------------|
| **入口限流** | API Gateway + Redis 令牌桶，按 tenant_id 限流 | ✅ 统一网关 |
| **数据隔离** | 每条数据都带 tenant_id，查询时强制过滤 | ✅ 统一过滤逻辑 |
| **权限控制** | RBAC 权限表，按 tenant_id + role 控制功能访问 | ✅ 统一权限中心 |
| **配额管理** | Redis 按 tenant_id 统计调用次数/Token 消耗 | ✅ 统一配额中心 |
| **计费统计** | 异步计费流水，按 tenant_id 聚合 | ✅ 统一计费服务 |
| **熔断降级** | Resilience4j 按 tenant_id 熔断，大客户独立阈值 | ✅ 统一熔断策略 |

**关键设计**：tenant_id 不是每个模块自己管理的，而是由**统一的租户管理中心**分配和管理。新租户注册时，租户管理中心分配 tenant_id，然后自动下发到三个模块的配置里。"

## 三、加分项

1. **讲清楚"为什么是三个模块而不是一个大系统"。** "三个模块的交互模式完全不同——流程引擎是确定性的状态机，RAG 是无状态的检索生成，Agent 是事件驱动的推理执行。如果做成一个系统，要么过度复杂，要么互相拖累。拆开后各自独立扩缩容、独立迭代、独立容错。"——说明你有架构拆分的判断力。

2. **提到"事件驱动"而不是"直接调用"。** "三个模块通过 RabbitMQ 事件总线协作，不直接调用对方的 API。Agent 诊断完成后发一个事件，流程引擎订阅这个事件自动创建工单。这样模块间是松耦合的——Agent 挂了不影响流程引擎，流程引擎挂了不影响 RAG。"——说明你理解事件驱动架构的价值。

3. **讲到"统一 tenant_id 贯穿所有模块"。** "tenant_id 不是每个模块自己生成的，而是由统一的租户管理中心分配。新租户注册时分配一个 tenant_id，自动下发到三个模块的配置里。这样保证了 tenant_id 的全局唯一性和一致性。"——说明你理解多租户的基础设施设计。

4. **提到模块间的"能力互补"而不是"功能重复"。** "流程引擎管'怎么做'（确定性流程），RAG 管'是什么'（知识查询），Agent 管'为什么'（根因分析）。三个模块组合起来才是完整的应急安全管理能力。"——说明你有产品思维。

5. **结合 500 强客户讲。** "4 家世界 500 强企业客户，三个模块都买了。他们的典型场景是：流程引擎驱动日常巡检和审批，RAG 知识库提供安全规范查询，运维 Agent 保障平台 7×24 可用。三个模块共享同一套多租户隔离，数据互不干扰。"

6. **提到独立扩缩容。** "三个模块的流量模式不同——RAG 在白天查询多，Agent 在凌晨告警多，流程引擎全天均匀。独立部署后可以按需扩缩容，不用整体扩容浪费资源。"

## 四、追问应对

### Q1：三个模块的数据存储是怎么隔离的？共享 Collection 和独立 Collection 的选型依据是什么？

"三个模块的数据存储完全不同，没有'共享 Collection vs 独立 Collection'的选择题——它们本来就不是同一类数据：

| 模块 | 存储 | 隔离方式 |
|------|------|---------|
| 流程引擎 | MySQL（流程实例、状态、历史） | tenant_id 字段，查询时过滤 |
| RAG 知识库 | Qdrant（向量）+ Elasticsearch（关键词） | 共享 Collection + tenant_id Payload 过滤 |
| 运维 Agent | 无持久化（告警上下文是临时的） + MySQL（告警历史） | tenant_id 字段 |

**RAG 为什么用共享 Collection 而不是独立 Collection**：

```
独立 Collection:
  120 个租户 = 120 个 Collection
  每个 Collection 要单独建索引、单独监控、单独备份
  升级时要逐个迁移 → 运维噩梦

共享 Collection:
  1 个 Collection，所有租户的 chunks 混在一起
  tenant_id 作为 Payload 字段，查询时过滤
  运维成本和租户数量无关
```

选型依据：**单租户数据量小（200-2000 份文档），共享 Collection 的过滤开销可以忽略（1ms）。如果单租户数据量到了百万级，才会考虑独立 Collection。"

### Q2：如果一个租户只买了 RAG 模块没买流程引擎，权限怎么控制？

"两层控制：

**第一层：前端菜单控制。** 租户的套餐信息存在 `tenant_module` 表里：

```sql
CREATE TABLE tenant_module (
  tenant_id   VARCHAR(32),
  module_id   VARCHAR(16),   -- flow_engine / rag / agent
  status      VARCHAR(8),    -- active / expired / disabled
  PRIMARY KEY (tenant_id, module_id)
);
```

前端根据这个表动态渲染菜单——没买的模块不展示入口。

**第二层：后端接口鉴权。** 即使用户直接调 API（绕过前端），后端也会校验：

```
请求进入 → 提取 tenant_id + 目标模块
  → 查询 tenant_module
  → 该租户没有这个模块的权限 → 返回 403 Forbidden
```

**关键点**：权限校验在 API Gateway 层统一做，不在每个模块内部做。这样新增模块时不需要改已有模块的代码。

**实际场景**：我们的基础版套餐只包含 RAG 知识库，流程引擎和运维 Agent 是专业版才有的功能。120 个租户中约 60% 只买了基础版。"

### Q3：120 个活跃租户的 onboarding 流程是怎样的？新租户从注册到可用要多久？

"onboarding 流程分自动和手动两部分：

```
自动流程 (系统完成，约 5 分钟):
  ① 注册 → 租户管理中心分配 tenant_id
  ② 自动创建租户配置:
     - 默认模型配置 (Qwen + GPT-4)
     - 默认配额 (基础版 1 万次/月)
     - 默认权限 (RBAC 角色)
  ③ 自动初始化数据空间:
     - Qdrant: 无需创建 Collection (共享)，只需分配 tenant_id
     - MySQL: 创建租户级配置记录
     - Redis: 初始化配额 Key
  ④ 发送欢迎邮件 + 管理后台登录链接

手动流程 (租户自行完成):
  ⑤ 上传安全文档 → Embedding → 入库 (RAG)
  ⑥ 配置组织架构 (楼层、区域、人员)
  ⑦ 配置告警规则 (如需要 Agent 模块)
```

**从注册到可用：5 分钟**（自动流程完成即可使用基础功能）。文档上传和配置由租户自行完成，不影响系统可用性。

**为什么这么快**：共享 Collection + Payload 过滤方案不需要为新租户创建任何新的数据结构——只需要分配一个 tenant_id，所有模块的查询都自动按 tenant_id 过滤。如果用独立 Collection，每个模块都要创建新的数据结构，onboarding 至少要 2 天。

**4 家世界 500 强客户的 onboarding**：他们的文档量大（5000+ 份），onboarding 时会批量上传，Embedding 处理需要几个小时。但这不影响系统可用性——上传过程中已上传的部分已经可以检索了。"

## 五、面试策略总结

| 场景 | 怎么说 |
|------|--------|
| 面试官问三模块关系 | 讲"互补关系"：确定性流程用引擎，知识查询用 RAG，非确定性推理用 Agent |
| 面试官问协作方式 | 讲三个协作场景：演练（流程+RAG）、故障（Agent+流程）、审批（流程+RAG） |
| 面试官问多租户隔离 | 讲"统一 tenant_id 贯穿所有模块" + 六个共享维度 |
| 面试官追问数据隔离 | 讲各模块存储不同，RAG 用共享 Collection（120 租户数据量小） |
| 面试官追问权限控制 | 讲两层：前端菜单 + 后端接口鉴权，Gateway 统一校验 |
| 面试官追问 onboarding | 讲 5 分钟自动流程 + 手动配置，共享方案让 onboarding 极快 |

**核心叙事：三个模块不是三个独立项目，是一个平台的三个有机组成部分。它们共享同一套多租户基础设施，通过事件驱动协作，组合起来才是完整的应急安全管理能力。**
