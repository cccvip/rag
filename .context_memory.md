## 三、技术架构（成本驱动选型）

### 核心原则
- **成本 > 性能 > 功能**：先问"有没有开源免费的"，再问"性能够不够"
- **小数据量不追大方案**：单租户 2000 条以内，拒绝过度工程

### 存储层（双轨架构）

| 组件 | 用途 | 选型理由 | 索引 |
|------|------|---------|------|
| **Qdrant** | Dense 向量检索 | Rust 单机、零依赖、运维简单 | **Flat**（<5000 条），>5000 切 HNSW |
| **Elasticsearch** | BM25 稀疏检索 | 开源、生态成熟、商用方案贵 8~10 倍 | 倒排索引 |
| **MySQL** | 结构化数据（楼层/区域/坐标） | 已有生态、标量过滤 | B+Tree |
| **Redis** | 三层缓存 | 高频 query 短路、LLM 响应复用 | — |

### 为什么不用 Milvus？
- Milvus 需要 Etcd + MinIO + Pulsar，运维人力成本高
- 团队 6 人无专职运维
- 单租户几百份文档，Qdrant 单机完全够用

### 为什么用 Flat 不用 HNSW？
- Flat 零参数、零维护、查询 1ms
- HNSW 需要调参（M、efConstruction、ef），调参花时间
- 单租户 >5000 条才切 HNSW，目前仅 3 个大客户触发

### 为什么双轨（Qdrant + ES）而不是单一数据库？
- Qdrant 的 Hybrid Search 不够成熟
- 让 Qdrant 管 Dense、ES 管 BM25，应用层 RRF 融合
- 两边都是各自领域开源标杆，风险最低

### 多租户隔离方案
- **共享 Collection + Payload 过滤**（tenant_id）
- Qdrant：Payload 里放 tenant_id，查询时强制过滤
- ES：tenant_id 字段，每次查询带过滤条件
- 应用层：**二次校验**检索结果的 tenant_id
- 缓存：Redis key 带 tenant_id 前缀
- onboarding：自动分配 tenant_id，**5 分钟**完成

## 四、RAG 链路（10 个环节 + 缓存层）

```
用户 Query
    ↓
🟦 缓存层 1：Query 答案缓存（命中率 55%，TTL 应急 1h/常规 24h）
    ├─→ HIT → 直接返回（5ms，$0）
    └─→ MISS → Query Router（实体提取：楼层/区域/事件）
            ↓
        ├─→ MySQL 标量过滤（精确约束）
        └─→ 查询重写（同义词扩展）
                ↓
            🟩 缓存层 2：HyDE 缓存（命中率 30%，TTL 4h）
                ↓
            Qdrant（Dense，1ms）┐
            ES（BM25，5ms）     ├→ RRF 融合（1ms）→ CE 精排（70ms）
            MySQL（2ms）        ┘
                ↓
            Prompt 组装
                ↓
            🟨 缓存层 3：LLM 响应缓存（命中率 40%，TTL 1h）
                ├─→ HIT → 直接返回（5ms，$0）
                └─→ MISS → GPT-4 生成（550ms，$0.02）
                            ↓
                    后处理校验（引用真实性 / 楼层一致性 / 安全关键词）
                            ↓
                    返回用户 + 写入缓存层 1 & 3
```

### 缓存设计原则
- **向量检索层不缓存**（Qdrant+ES+MySQL+RRF = 9ms，缓存 lookup 也要 2ms，ROI 太低）
- **缓存放在"慢、贵、重复"的地方**：LLM 生成（550ms+$0.02）、HyDE（200ms+$0.01）

## 五、关键 Bad Case

### Case 1：Retrieval —— 新租户冷启动检索质量差
- **根因**：数据量小（50 份）+ 领域词汇未对齐 + BM25 IDF 统计不准
- **修复**：租户级同义词库 + 冷启动保护期（前 7 天 BM25 权重提高）+ 文档覆盖率检测
- **结果**：Faithfulness 从 0.55 → 0.82（首周），稳定后 0.90

### Case 2：Generation + 成本事故 —— LLM 被攻击刷爆
- **根因**：缺少租户级限流和缓存，重复 query 每次都走完整 LLM 调用
- **修复**：API 网关 QPS 限流（20 QPS 熔断）+ Redis 缓存（命中率 55%）+ Token 配额 + 布隆过滤器防穿透
- **结果**：LLM 成本降 60%，单租户月均 LLM 成本从 15 元 → 5 元
