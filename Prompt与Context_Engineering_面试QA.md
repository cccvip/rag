# Prompt Engineering vs Context Engineering — 面试深度 QA

> 面试题：Prompt Engineering 和 Context Engineering 的区别是什么？为什么说 Context Engineering 比 Prompt Engineering 更重要？
> 候选人背景：8年 Java 后端，运维 Agent 结构化 Prompt 模板 + RAG 知识库 + MCP 数据源接入

---

## 一、最佳答案

### 1.1 一句话区分

**Prompt Engineering 管的是"怎么问"，Context Engineering 管的是"给模型看什么"。**

Prompt 是你写给模型的指令——角色设定、输出格式、约束条件。Context 是模型在推理时能看到的全部信息——对话历史、检索结果、工具返回、系统配置。

### 1.2 详细对比

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|-------------------|
| **关注点** | 指令的措辞和结构 | 信息的选择和组织 |
| **核心问题** | "怎么问才能让模型回答好" | "给模型什么信息才能让它做出正确判断" |
| **作用范围** | 单次请求的指令文本 | 整个 context window 的内容编排 |
| **典型手段** | 角色设定、Few-shot、CoT、输出格式约束 | RAG 检索、对话历史压缩、工具结果注入、记忆管理 |
| **类比** | 你问同事一个问题的方式 | 你给同事看的全部背景材料 |

### 1.3 为什么 Context Engineering 更重要

**核心论点：模型的能力上限由 context 决定，不由 prompt 决定。**

一个类比：你是一个非常聪明的分析师（LLM），我让你写一份市场报告。

- **Prompt Engineering**：我精心措辞——"请以专业分析师的口吻，分三个维度，用数据支撑……"——问法很好，但如果你手头没有市场数据，你只能编。
- **Context Engineering**：我把竞品报告、用户调研、销售数据整理好放在你桌上——即使我只说"帮我写个报告"，你也能写出高质量的内容。

**Prompt 再好，context 不对，模型也只能幻觉。Context 对了，prompt 粗糙一点，结果也不会差。**

具体来说：

**① Prompt 的边际收益在递减**

早期 Prompt Engineering 的红利很大——加一句"让我们一步步思考"就能显著提升推理质量。但现在模型越来越聪明，Prompt 技巧的提升空间越来越小。同样的精力投入到"给模型更好的信息"，收益更大。

**② Context 决定了模型的"知识边界"**

模型的知识 = 预训练知识 + context 中的信息。预训练知识是固定的、可能过时的。context 是动态的、实时的、和业务相关的。RAG、工具返回、对话历史——这些都是 context 的来源，它们决定了模型能"看到什么"。

**③ Context Engineering 是系统工程，Prompt Engineering 是文本工程**

Prompt 优化到一定程度就很难再提升了。但 Context Engineering 涉及检索策略、压缩算法、优先级排序、缓存策略——这是一个完整的工程系统，优化空间大得多。

### 1.4 两者的关系

不是替代关系，是包含关系：

```
Context Engineering（大）
  ├── Prompt（指令文本）
  ├── Retrieval（RAG 检索结果）
  ├── Memory（对话历史、长期记忆）
  ├── Tool Results（工具返回）
  └── System Config（系统配置、租户信息）
```

Prompt 是 Context 的一部分。Context Engineering 是更上层的系统设计。

## 二、结合项目的实际回答

"这两个概念我在运维 Agent 和 RAG 知识库中都有深刻的体会。

**Prompt Engineering 层面**：我的运维 Agent 针对不同告警场景设计了结构化 Prompt 模板——MQ 堆积用一套模板，DB 慢查询用另一套。每个模板都约束了输出格式：必须包含根因分析、影响范围、处置建议。这是典型的 Prompt Engineering——通过精心设计的指令引导模型输出可执行的诊断结论。

**但真正决定诊断质量的不是 Prompt，而是 Context。** 同一个 Prompt 模板，给模型看不同的信息，诊断结论完全不同：

- 只给告警信息 → 模型猜
- 给告警信息 + 最近 1 小时的日志摘要 → 模型能定位到具体异常
- 给告警信息 + 日志 + Prometheus 指标 + 该租户的历史告警模式 → 模型能给出精准的根因分析

**我做的三层缓存本质上就是 Context Engineering**——决定给模型看什么信息、信息从哪来、怎么组织：

| Context 来源 | 工程化手段 | 对诊断质量的影响 |
|-------------|-----------|----------------|
| 告警事件 | 规则匹配 → 结构化注入 | 基础输入，决定走哪个模板 |
| ELK 日志 | Function Calling → 结果压缩 | 从'猜'变成'有证据' |
| Prometheus 指标 | Function Calling → 异常区间提取 | 从'单维度'变成'多维度' |
| 历史告警模式 | MySQL 查询 → 应用层缓存 | 从'单次分析'变成'模式关联' |
| RAG 知识库 | Hybrid Search → Re-rank | 从'经验判断'变成'知识增强' |

**Prompt 是固定的模板，Context 是动态组装的。** 模板一个月可能才改一次，但每次请求的 Context 都不同——不同租户、不同告警类型、不同时间窗口。Context Engineering 的工程量远大于 Prompt Engineering。"

## 三、加分项

1. **用"信息不对称"框架解释。** "LLM 的本质问题是信息不对称——模型不知道你的业务数据、不知道当前的上下文、不知道工具返回了什么。Prompt Engineering 只解决了'怎么问'，Context Engineering 解决了'给模型看什么'。后者才是消除信息不对称的关键。"

2. **提到 Context Window 是稀缺资源。** "128K 的 context window 看似很大，但真正可用的远不到 128K——系统 Prompt、工具定义、对话历史、检索结果、工具返回、生成空间都要抢这个预算。Context Engineering 的核心挑战之一就是在有限的窗口里做信息优先级排序。"——说明你有资源管理意识。

3. **结合三层缓存讲 Context 的成本控制。** "我的 RAG 系统做了三层缓存——Query 答案缓存 55%、HyDE 缓存 30%、LLM 响应缓存 40%，LLM 调用成本降了 82%。这本质上也是 Context Engineering——缓存命中时，模型不需要重新处理整个 context，直接返回结果。Context 的组装是有成本的，缓存是降低这个成本的手段。"

4. **提到 Context 的质量评估。** "Context Engineering 还有一个被忽视的维度——质量评估。RAG 检索回来的内容如果不相关，反而会干扰模型判断（'lost in the middle' 问题）。所以不仅要给模型看信息，还要评估信息的质量和相关性。"

5. **类比软件工程。** "Prompt Engineering 像是写一个函数的接口——参数名、返回类型、文档注释。Context Engineering 像是设计整个系统的数据流——数据从哪来、怎么处理、怎么组装、怎么传递。前者是局部优化，后者是系统设计。"

## 四、追问应对

### Q1：Few-shot 和 CoT 提示分别在什么场景下使用？

"两者解决不同的问题：

**Few-shot（少样本提示）**——给模型几个输入-输出的示例，让它'照葫芦画瓢'。适合**输出格式固定、难以用文字描述**的场景。

```
示例场景：从告警文本中提取结构化信息
输入: "order-svc CPU 85% 持续 15 分钟"
输出: { service: "order-svc", metric: "cpu", value: 85, duration: "15min" }

输入: "payment-svc RT 从 50ms 涨到 500ms"
输出: { service: "payment-svc", metric: "rt", value: 500, baseline: 50 }
```

给 2-3 个示例，模型就能学会这个格式，比用文字描述'请输出 JSON 格式，包含 service、metric、value 字段'更可靠。

**CoT（思维链提示）**——引导模型'一步步思考'。适合**需要推理、分析、判断**的场景。

```
示例场景：故障根因分析
Prompt: "请按以下步骤分析：
1. 先看告警指标，判断是性能问题还是可用性问题
2. 再看日志，定位异常类型
3. 结合历史模式，判断是否是已知问题
4. 给出处置建议"
```

**选型决策**：

| 场景 | 用什么 | 原因 |
|------|--------|------|
| 信息提取、格式转换 | Few-shot | 输出格式是核心，示例最直观 |
| 根因分析、方案设计 | CoT | 推理过程是核心，步骤引导更重要 |
| 两者结合 | Few-shot + CoT | 先给示例展示格式，再用 CoT 引导推理 |

**我的运维 Agent 实际用的是 CoT**——每个 Prompt 模板都设计了结构化的分析步骤：先看告警类型 → 再查日志 → 再看指标 → 最后给结论。不是让模型自由发挥，而是引导它按工程师的思维路径走。"

### Q2：上下文窗口有限时，信息的优先级怎么排序？

"这是一个资源分配问题。我的排序策略：

**优先级从高到低**：

```
1. 系统 Prompt + 角色设定     (~2K)   ← 必须保留，定义模型行为
2. 当前用户输入               (~1K)   ← 必须保留，当前任务
3. 工具定义 (Schema)          (~5K)   ← 必须保留，模型才知道能调什么工具
4. 高相关性检索结果            (~10K)  ← 优先保留，直接影响回答质量
5. 工具返回结果               (~10K)  ← 按相关性截断，压缩后注入
6. 近期对话历史               (~5K)   ← 保留最近 3-5 轮
7. 长期记忆 / 低相关检索       (~3K)   ← 有空间才加
8. 生成空间 (headroom)        (~20K)  ← 必须预留，否则模型输出被截断
```

**总预算分配**：128K context window，实际可用约 60K（剩余留给生成空间和不可预见的工具返回）。

**压缩策略**：

- 检索结果：按相关性分数排序，超出预算的直接截断
- 工具返回：后端压缩后再注入（ELK 10000 条 → Top 20 摘要）
- 对话历史：分层压缩（近期全量 + 远期摘要）
- 工具定义：只注入当前场景可能用到的工具，不是所有工具都塞进去

**关键原则：宁可少给，不要给错。** 低质量的 context 比没有 context 更差——模型可能会被不相关的信息误导（'lost in the middle' 问题）。"

### Q3：你实际项目中的 Prompt 模板是怎么管理的？有版本控制吗？

"有。Prompt 模板在我们的系统里是**一等公民**，不是硬编码在代码里的字符串。

**管理方式**：

```
1. 存储：Prompt 模板存在 Nacos 配置中心，按租户 + 告警类型分组
   - 通用模板: /prompts/common/mq_backlog.txt
   - 租户定制: /prompts/tenant/T001/mq_backlog.txt

2. 版本控制：Nacos 自带版本历史，每次修改自动记录
   - v1: 基础模板
   - v2: 增加'历史告警模式'分析步骤
   - v3: 增加'关联指标'分析维度

3. 热加载：模板修改后实时生效，不需要重启服务

4. A/B 测试：新模板先在 10% 的租户灰度，效果指标达标后再全量
```

**为什么这样管理**：

- **租户定制**：不同租户的业务场景不同，通用模板覆盖不了所有情况。比如化工厂的告警模板和写字楼的完全不同。
- **快速迭代**：Bad Case 归集后发现模板有问题，改 Nacos 配置就能修复，不需要发版。
- **可回滚**：新模板效果不好，一键回滚到上一个版本。

**Prompt 模板的结构**：

```
模板 = 固定部分 + 动态部分

固定部分（人工编写）:
  - 角色设定: "你是一个资深运维工程师"
  - 分析步骤: "1. 看告警类型 → 2. 查日志 → 3. 看指标 → 4. 给结论"
  - 输出约束: "必须包含根因、影响范围、处置建议"

动态部分（系统注入）:
  - {{ALERT_INFO}}      ← 结构化告警数据
  - {{LOG_SUMMARY}}     ← ELK 日志摘要
  - {{METRIC_SUMMARY}}  ← Prometheus 指标摘要
  - {{HISTORY_PATTERN}} ← 该租户的历史告警模式
```

固定部分是 Prompt Engineering，动态部分是 Context Engineering。两者结合才是完整的 prompt 管理体系。"

## 六、面试策略总结

| 场景 | 怎么说 |
|------|--------|
| 面试官问区别 | 一句话："Prompt 管怎么问，Context 管给模型看什么" |
| 面试官问为什么 CE 更重要 | 讲三个论点：边际收益递减、知识边界、系统工程 vs 文本工程 |
| 面试官问你的项目 | 讲 Prompt 模板（固定）+ Context 组装（动态），两者结合 |
| 面试官追问 Few-shot vs CoT | Few-shot 适合格式固定，CoT 适合需要推理，运维 Agent 用 CoT |
| 面试官追问优先级排序 | 讲 P0-P3 四级优先级 + Token 预算分配表 |
| 面试官追问模板管理 | 讲 Nacos 配置中心 + 版本控制 + 热加载 + A/B 测试 |

**核心观点：Prompt Engineering 是 1.0 时代的手艺活，Context Engineering 是 2.0 时代的系统工程。前者优化的是指令文本，后者优化的是整个信息供给链路。作为工程师，应该把更多精力放在 Context Engineering 上——这是系统设计问题，不是文案问题。**
