## 开场白（20 秒）

检索优化我分三块来讲：第一，BM25、Dense Retrieval 和 Hybrid Search 的原理和选型；第二，Cross-Encoder 为什么比 Bi-Encoder 更准但更慢；第三，ColBERT 的晚期交互机制是怎么折中速度和精度的。

---

## 一、BM25 vs Dense vs Hybrid

### BM25：精确的词项匹配

BM25 是在 TF-IDF 基础上改进的，做了两件事：第一，**词频饱和**——一个词在文档里出现 10 次和 100 次，相关性分数不会差 10 倍；第二，**文档长度归一化**——长文档天然词频高，需要惩罚，防止长篇大论占便宜。

它的本质就是查字典：query 里的词在文档里出现了几次、出现在什么位置。所以它对专业术语、ID、型号、代码这类**精确匹配**效果极好，而且零训练成本，可解释性强，能告诉用户"因为文档里出现了这些词"。

但它的死穴是**无法理解同义词**。比如用户问"充电桩安装费用"，文档里写的是"家用充电设备报装价格"，BM25 眼里这两个完全不相关，因为没有一个词对得上。对口语化表达和长尾查询效果也很差。

### Dense Retrieval：语义联想

Dense Retrieval 是用 Transformer 把 query 和 document 分别编码成 768 维或 1024 维的稠密向量，然后在向量空间里找最近邻。

它的优势是**语义泛化**。刚才那个例子，"充电桩安装费用"和"家用充电设备报装价格"在向量空间里距离很近，因为 Encoder 学过它们的语义是相关的。对同义词、口语化表达、甚至跨语言检索都很自然。

但它的弱点也很明显：第一，对精确术语不如 BM25，比如 "Model-X-2024" 和 "Model-Y-2024" 在向量空间里可能非常接近，这是致命的；第二，领域外效果可能崩，需要做对比学习微调；第三，有额外的向量存储和计算成本。

### Hybrid Search：左右脑并用

生产环境我**最推荐 Hybrid Search**，因为两者是互补的，不是互斥的。

融合算法我们用 **RRF**，也就是倒数排名融合。公式很简单，每个文档在各路召回里的排名取倒数，然后求和。k 取 60 是一个经验值。它的好处是不需要给 BM25 和 Dense 分配权重，也不需要做分数归一化，鲁棒性极强。

举个例子，BM25 召回的 Top 3 是 A、C、B，Dense 召回的 Top 3 是 B、A、E。RRF 一算，A 的分数最高，因为它在两路里都排得靠前。最终排序就是 A > B > C > E。这个算法工程上非常省心。

> **一句话总结：BM25 是查字典，Dense 是联想记忆，Hybrid 是左右脑并用。**

## 二、Cross-Encoder 为什么比 Bi-Encoder 更准但更慢

### 为什么更准？

核心原因是**交互时机不同**。

Bi-Encoder 是双塔架构，Query 和 Document 各自过 Encoder，编码的时候彼此完全看不到对方。Query 里"充电桩"的向量，只由 Query 自己的上下文决定；Document 里"充电桩"的向量，只由 Document 自己的上下文决定。然后算点积，信息损失很大。

Cross-Encoder 是把 Query 和 Document 拼接在一起，中间用 SEP 隔开，一起过完整的 Transformer。每一层的 Self-Attention 都让 Query 的每个 token 和 Document 的每个 token 双向交互。所以"充电桩"在 concat 后的表示，能同时感知到"安装"、"费用"、"标准"这些上下文，能捕获否定、转折这些细粒度语义关系。

### 为什么更慢？

因为 Bi-Encoder 的文档向量可以**离线预计算、一次算好、反复使用**。查询时只需要编码一次 Query，然后和预存的向量做点积，100 万文档也就 10 毫秒。

Cross-Encoder 的每次打分都要把 Query 和 Document 重新拼接后过一遍完整的 Transformer。100 万条文档就要过 100 万次前向传播。在 A100 上，单次大概 5 毫秒，100 万次也要 1.4 小时，完全无法在线服务。

**根本原因**：文档中每个 token 的表示在 Cross-Encoder 里是 Query-Dependent 的，搭配不同的 Query，同一个词的向量完全不同，所以无法预计算。

### 工程解法

生产环境必须用**两阶段架构**：

第一阶段 Bi-Encoder 做粗筛，从 100 万召回 Top 100，耗时约 10 毫秒；
第二阶段 Cross-Encoder 做精排，从 100 条里选出 Top 5，耗时约 500 毫秒。

Bi-Encoder 解决"找得到"，Cross-Encoder 解决"找得准"。

## 三、ColBERT 的晚期交互机制

### 问题

Bi-Encoder 太快但太粗，Cross-Encoder 太准但太慢。有没有折中？

ColBERT 的思路是：**预计算文档的 token 级向量，查询时只做轻量级的 token 交互。**

### 原理

编码阶段，Document 的每个 token 都单独编码成一个 128 维的小向量。一篇 100 个 token 的文档，存的是 100 × 128 的矩阵，而不是单个 768 维向量。Query  likewise，每个 token 编码成 128 维。

检索阶段，对 Query 的每个 token，在 Document 的所有 token 向量里找最相似的那个，取最大相似度。然后把所有 Query token 的最大相似度加起来，就是最终的相关性分数。

这个过程叫 **MaxSim**，也就是最大相似度聚合。

### 为什么叫"晚期交互"？

对比三种架构：

- Bi-Encoder：**没有交互**，编码完直接点积
- Cross-Encoder：**早期交互**，从第一层 Attention 就开始交互
- ColBERT：**晚期交互**，编码阶段各自独立，只在最后检索时做 token 级匹配

ColBERT 把交互推迟到了检索阶段，这样既保留了预计算的能力，又在最后一步做了细粒度匹配。

### 性能

精度上，ColBERT 接近 Cross-Encoder；速度上，比 CE 快 10 到 100 倍；代价是存储量大幅增加，每个文档存的是 token 向量矩阵。

代表模型是 **BGE-M3**，它支持 Dense、Sparse 和 ColBERT 三种模式。

## 四、补充：SPLADE

SPLADE 是 BM25 和 Dense 的中间态，用神经网络生成稀疏向量。输入文档过 BERT 后，对每个词在词汇表上的 logits 做扩展，输出一个稀疏向量。

比如原文是"家用充电桩安装费用"，SPLADE 可能扩展出"设备"、"报装"、"价格"这些原文没有的同义词或相关词。非零维度仍然很少，索引结构仍是倒排索引，所以检索很快。

它兼具 BM25 的精确匹配能力和 Dense 的语义扩展能力，是一个很有前景的方向。

## 收尾（15 秒）

总结下来，检索优化的技术选型我记一张表：精确匹配选 BM25，语义泛化选 Dense，生产默认选 Hybrid。召回阶段用 Bi-Encoder，精排阶段用 Cross-Encoder。如果存储可接受，ColBERT 是速度和精度的最优折中。

## 面试官可能的追问

**Q：为什么 RRF 不需要调权重？**
> RRF 用排名而非原始分数，天然消除了不同召回路径的分数分布差异。加权融合需要调 α，而且不同查询的最优 α 可能不同，调参成本很高。

**Q：ColBERT 的存储成本具体增加多少？**
> 假设文档平均 100 个 token，每个 token 128 维浮点，那每个文档需要 100 × 128 × 4 字节 = 50KB。Bi-Encoder 只需要 768 × 4 = 3KB。存储涨约 15 倍。工程上需要做向量压缩或者 ANN 索引优化。

**Q：SPLADE 和 ColBERT 怎么选？**
> SPLADE 输出的是稀疏向量，走倒排索引，存储小、检索快，适合大规模文档库。ColBERT 输出的是多向量矩阵，精度更高，但存储大，适合对精度要求极高且文档量可控的场景。
