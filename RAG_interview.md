# RAG 全流程 + 检索优化 — 15 分钟面试口述稿

> 总字数约 2800 字，口述语速 180 字/分钟，全程约 15 分钟。
> 加粗部分为必须记住的**记忆锚点**和**关键数字**。

---

## 开场白（30 秒）

我把 RAG 的完整链路拆成两块来讲。

第一块是**全流程**：从文档摄入、解析、分块、向量化、索引构建、查询重写、向量检索、重排序，到 LLM 生成和引用溯源，一共十个环节。

第二块是**检索优化**：BM25、Dense Retrieval、Hybrid Search 怎么选，Cross-Encoder 和 Bi-Encoder 的取舍，以及 ColBERT 的折中方案。

核心心法一句话：**RAG 不是向量库加 LLM 的简单拼接，而是检索精度、生成质量、系统延迟和成本四者之间的持续权衡。**

---

## 第一块：RAG 全流程（约 10 分钟）

### 1. 文档摄入

数据源通常分四类。

**批量文件**（PDF、Word、Excel）适合冷启动一次性灌库。**结构化业务数据**走 CDC 或定时同步，实时性好但要维护链路。**动态内容**比如政策公告，用爬虫或 Webhook，注意反爬和去重。**扫描件和图片**要先走 OCR，不建议直接丢给大模型，成本不可控。

关键决策是**增量更新**。我们的做法是给文档算 MD5 指纹，配合版本号去重，避免重复解析和嵌入。

### 2. 文档解析

解析最大的坑是**复杂排版**。纯文本好处理，但 PDF 有多栏、图文混排、表格嵌套。

分层处理：结构化文档用 Unstructured 加 Layout 模型提取标题层级。复杂 PDF 用 Marker 或 MinerU 做深度学习布局识别。扫描件走 PaddleOCR。

特别提一下**表格**，这是重灾区。直接展平成文本，行列关系全丢。我们的做法是把表格转成 HTML 或 Markdown 格式，保留结构信息再喂给下游。

### 3. 分块

分块质量直接决定检索天花板。常见策略四种：固定长度、递归字符、语义分块、**父子块**。

生产环境最推荐**父子块**：检索时用小颗粒度匹配，返回时拉父节点的大上下文，兼顾召回率和完整性。

经验参数记三个数字：**精确事实查询用 128~256 tokens，上下文推理用 512~1024 tokens，overlap 必须加 10%~20%**，防止边界信息被切断。

### 4. 向量化

选型先用通用模型跑通链路，比如 **BGE-large** 或 **E5**，再用领域内数据评测。术语密集型领域（医疗、法律）建议做对比学习微调，ROI 很高。

一个容易被忽视的权衡是**维度**。**768 维和 1024 维相比，精度提升通常不到 2%，但存储和计算成本涨 30% 以上**。不是越大越好，要实测。

多语言推荐 **BGE-M3**，长文本推荐 **GTE** 或支持 8192 token 的 BGE。

### 5. 索引构建

核心是选索引类型。**HNSW** 适合百万到十亿级，召回率高但构建慢、内存大。**IVF** 适合更大规模，速度快但参数调优复杂。**Flat** 十万级以下精度 100%，但不可扩展。

数据库选型：企业级用 **Milvus / Zilliz**，已有 PG 生态用 **pgvector**，轻量高性能选 **Qdrant**（Rust 实现）。

### 6. 查询重写

**这一步不是可选优化，是解决语义鸿沟的关键。**

用户问题往往是口语化、有歧义的。比如问"装那个要多少钱"，直接 Embedding 检索，"那个"的向量根本对不齐"充电桩安装费用"。

常见技术：对话历史压缩解决指代消解；**HyDE** 解决口语和书面语词汇不匹配；**Multi-Query** 多角度覆盖提升召回；子问题分解适合复杂多跳查询。

HyDE 会增加一次 LLM 调用，但召回率提升显著。简单场景用规则层兜底，比如同义词扩展和缓存。

### 7. 向量检索

检索策略三种：Dense 做语义匹配，Sparse（BM25）做精确词项匹配，生产环境最推荐 **Hybrid Search**。

融合算法用 **RRF**，倒数排名融合。公式是 score 等于各路径排名倒数的总和，**k 取 60**。好处是不需要分配权重，不需要分数归一化，鲁棒性极强。

BM25 像字典精确查找，Dense 像语义联想，两者天然互补。

### 8. 重排序

这里是面试官最爱追问的点：**为什么不直接用 Cross-Encoder？**

因为 CE 精度极高但计算复杂度是 O(N)。文档库一百万条，逐对过 CE 要 **1.4 小时**（A100），无法在线服务。

所以必须用**两阶段架构**：第一阶段 **Bi-Encoder 粗筛**，从一百万召回 Top 100，耗时约 **10 毫秒**（文档向量预计算好了，点积即可）。第二阶段 **Cross-Encoder 精排**，从 100 条选 Top 5，耗时约 **500 毫秒**。

Bi-Encoder 解决"找得到"，Cross-Encoder 解决"找得准"。

CE 更准的核心原因是**交互时机**：BE 编码时 Q 和 D 彼此不可见，信息损失大；CE 把 Q 和 D 拼接后过 Transformer，每层 Attention 都双向交互，能捕获否定、转折等细粒度语义。

CE 更慢的核心原因是**无法预计算**：文档中每个 token 的表示是 Query-Dependent 的，搭配不同 Query 向量完全不同。

模型选型：高并发用 **bge-reranker-base**（278M），精度优先用 **bge-reranker-large**（560M），多语言用 **jina-reranker-v2**。

### 9. LLM 生成

控制三个维度：模型选型、温度、流式输出。

简单问答用 GPT-3.5 或 Claude Haiku，复杂推理上 GPT-4 或 Claude Sonnet。Temperature **事实类 0~0.3，创意类 0.7+**。**流式输出必开**，降低首字延迟 TTFT。

幻觉控制：Prompt 明确约束"只基于参考资料，不确定就说未提及"。上下文留 **20%~30% buffer** 给生成输出。

### 10. 引用溯源

分三层：第一层 Prompt 里注入来源 ID，要求 LLM 按格式引用。第二层返回原始元数据（标题、链接、页码）。第三层最关键，做**引用校验**——后处理验证 LLM 输出的引用是否真实对应，过滤伪造引用。

## 第二块：检索优化深度（约 4 分钟）

### BM25 的本质

BM25 在 TF-IDF 基础上加了**词频饱和**和**文档长度归一化**。它的本质就是查字典：query 里的词在文档里出现了几次。对专业术语、ID、型号精确匹配极强，零训练成本，可解释性强。

死穴是无法理解同义词。"充电桩安装费用"和"家用充电设备报装价格"词完全不重叠，BM25 眼里就是零相关。

### Dense Retrieval 的本质

用 Transformer 把文本编码成 768 维稠密向量，在向量空间找最近邻。优势是语义泛化，能解决词汇鸿沟。弱点是对精确术语不如 BM25，领域外效果可能崩。

### Hybrid Search 的本质

**生产环境默认选 Hybrid**。BM25 和 Dense 是互补的，不是互斥的。RRF 融合鲁棒性强，不需要调权重。

### ColBERT 的折中

Bi-Encoder 太快但太粗，Cross-Encoder 太准但太慢。ColBERT 的思路是**预计算文档的 token 级向量，查询时做 Late Interaction**。

编码阶段各自独立，Document 每个 token 编码成 128 维小向量。检索阶段对 Query 的每个 token，在 Doc 的 token 向量里找最相似的，取 **MaxSim** 聚合。这样既保留预计算能力，又做了细粒度匹配。

精度接近 CE，速度比 CE 快 **10~100 倍**，代价是存储涨约 **15 倍**（每文档存 token 矩阵而非单向量）。代表模型 **BGE-M3**。

### SPLADE 补充

SPLADE 是 BM25 和 Dense 的中间态，用神经网络生成稀疏向量。输入文档过 BERT 后扩展出同义词和相关词，输出稀疏向量走倒排索引。兼具 BM25 的精确匹配和 Dense 的语义扩展。

## 收尾（30 秒）

总结下来，RAG 十个环节环环相扣。我的工程经验是：**分块决定检索天花板，检索决定生成下限，查询重写和重排序是性价比最高的两个优化点。**

检索技术选型记一张表：**精确匹配选 BM25，语义泛化选 Dense，生产默认选 Hybrid。召回用 Bi-Encoder，精排用 Cross-Encoder，存储可接受时 ColBERT 是最优折中。**

实际落地建议先做端到端 baseline，再基于 bad case 分析决定优化哪一步。

## 附录：高频追问 & 标准应答

**Q：如果召回率不够，先优化哪一步？**
> 先看分块策略是不是太粗或太细，然后加 Hybrid Search，最后考虑 Multi-Query 或 HyDE。

**Q：Bi-Encoder 和 Cross-Encoder 必须同系列吗？**
> 不必须，但同系列语义空间更一致。Bi-Encoder 用 BGE-large，CE 配 bge-reranker-large 是最稳妥组合。

**Q：为什么 RRF 不需要调权重？**
> RRF 用排名而非原始分数，天然消除不同召回路径的分数分布差异。加权融合需要调 α，不同查询最优 α 可能不同。

**Q：查询重写的成本怎么控制？**
> 分层：规则层做同义词和缓存兜底，命中就不走 LLM；只有复杂查询才启用 HyDE 或 Multi-Query。

**Q：ColBERT 存储成本涨多少？**
> 假设文档平均 100 token，每个 token 128 维浮点，每文档约 50KB。Bi-Encoder 只需 3KB。存储涨约 15 倍。

**Q：SPLADE 和 ColBERT 怎么选？**
> SPLADE 走倒排索引，存储小、检索快，适合大规模文档库。ColBERT 精度更高但存储大，适合对精度要求极高且文档量可控的场景。

**Q：表格数据怎么向量化？**
> 财务报表这类结构化数据不适合直接 Embedding，应该走 Text-to-SQL 查数据库，文本类分析文档才走向量检索。

**Q：768 维和 1024 维怎么选？**
> 先用 768 维跑 baseline，实测召回率和延迟。1024 维精度提升通常不到 2%，但成本涨 30%+，多数场景不值得。
