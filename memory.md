## 三、核心技术讨论（面试储备）

### 3.1 节点类型注册机制
- **绑定方式**：`ActivityBehavior` 接口 + `@FlowNodeType` 注解
- **注册时序**：Spring启动时SPI扫描 → NodeTypeRegistry建立type→Bean映射
- **编译期校验**：发布流程时校验type是否已注册，杜绝运行时非法流程
- **类比XXL-JOB**：都是"注解标记 + 注册表路由 + 业务扩展"，但流程引擎多了DAG拓扑和状态机

### 3.2 线程模型（类Reactor）
```
调度层（FlowDriver）：单线程无锁，处理状态流转和DAG路由
执行层（ActivityExecutor）：多线程池，执行业务Behavior
```
- 与Netty Reactor对比：EventLoop≈FlowDriver，Worker≈Executor
- **分层线程池**：按资源特性聚合（轻量IO / 重IO / 阻塞等待 / 计算型）
- 阻塞型节点用虚拟线程（JDK 21）或缓存线程池

### 3.3 人工审批 + 超时自动流转
- MANUAL模式：进入RUNNING后挂起，写待办任务，释放线程
- 超时检测：定时任务扫描 + Redis ZSET分布式延迟队列
- 超时后策略：催办/升级/自动转交/按失败处理
- **竞态处理**：乐观锁保证"人工完成"与"超时扫描"不会同时驱动状态机

### 3.4 服务重启恢复
- 活动状态同步持久化（MySQL + 乐观锁）
- 当前执行节点位置持久化
- 启动时扫描恢复：AUTO节点看心跳超时，MANUAL节点看待办是否存在

### 3.5 "契约即意图"设计
Behavior接口可声明的意图：
- `getResourceProfile()` → 引擎推导线程池路由
- `getExecutionMode()` → 引擎推导执行策略
- `isIdempotent()` → 引擎推导重试策略
- `getExpectedTimeout()` → 引擎推导告警阈值
- `getFailureStrategy()` → 引擎推导降级分支

## 四、最终简历关键表述（建议背诵）

### 业务流引擎
> 基于DIDI开源LogicFlow的JSON DSL，**自研应急安全领域专用流程引擎**。核心架构采用"编译时严格校验 + 运行时轻量执行 + 故障时快速恢复"：编译层完成DAG环检测与全量校验；运行时采用**类Reactor单线程无锁调度**（参考Netty EventLoop），业务逻辑异步线程池执行，单机5000+ TPS；持久化层状态变更同步写MySQL（乐观锁），保障服务重启不丢状态。设计多级超时升级链与设备指令可靠投递机制。遵循**"契约即意图"**设计，业务方通过注解声明节点资源特性，引擎自动推导线程池路由与降级策略。

### IM信号发射器
> 采用**无锁环形队列（Disruptor模式）+ 批量聚合推送**，单机QPS 3k→2.5w，消息到达率99.2%→99.97%。

### 运维Agent
> 整合ELK/Prometheus/LLM根因分析，夜间告警人工介入率100%→15%，MTTR 45min→8min。

## 五、高频追问 & 建议应答方向

| 问题 | 应答要点 |
|------|---------|
| 为什么自研不用Flowable？ | 应急是"指令调度流"非"审批流"，BPMN过重，超时升级/设备投递无法映射到BPMN规范 |
| 单线程调度会不会是瓶颈？ | 调度器只做内存DAG遍历，瓶颈在业务侧DB/HTTP；水平扩展按instance_id哈希分片 |
| 怎么保证服务重启不丢状态？ | 活动快照+当前节点位置持久化+启动扫描恢复，AUTO看心跳，MANUAL看待办 |
| 线程池怎么配？ | 按资源特性聚合为3~4个池，基于任务特征给初始值，上线后监控渐进调优 |
| 契约即意图是什么？ | 业务方通过注解声明"我是什么任务"，引擎自动推导"怎么执行"，而非配YAML |
