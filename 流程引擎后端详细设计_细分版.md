## 一、整体架构

### 部署形态

本引擎以**独立服务**形态部署，业务系统通过 HTTP 与引擎交互（引擎回调业务服务走 Netty 轻量 HTTP，业务服务回调引擎走标准 REST API）。引擎本身不执行业务逻辑，只负责**状态机调度 + 持久化 + 超时检测**，业务 Handler 在业务服务进程中执行，执行完成后回调引擎推进状态。

> 设计参考 XXL-JOB 的"调度中心 + 执行器"模型：引擎是调度中心，业务服务是执行器。

**分层原则：**

| 层级 | 职责 | 设计原则 |
|------|------|---------|
| 编译层 | 将外部 JSON 转换为内部可执行模型 | **一次编译，多次运行**；编译时做尽所有校验 |
| 运行时核心 | 驱动状态流转，不碰业务逻辑 | **无锁、单线程调度**；状态转移原子化 |
| 调度执行层 | 决定"什么时候执行什么"，向业务服务发起回调 | **调度与执行分离**；引擎不执行业务代码 |
| 基础设施 | 存储、消息、缓存 | 运行时核心**不直接依赖**基础设施，通过接口抽象 |

### 交互协议（引擎 ↔ 业务服务）

引擎与业务服务之间的交互采用**双向异步回调**模型，参考 XXL-JOB 的"调度中心 + 执行器"通信设计：

**双向通道：**

| 方向 | 协议 | 路由方式 | 说明 |
|------|------|---------|------|
| 业务服务 → 引擎 | HTTP REST API | 经 API 网关 | 业务服务主动调用引擎的 start / complete / query 接口 |
| 引擎 → 业务服务 | Netty 轻量 HTTP Server | 直连（内网） | 引擎向业务服务的 SDK 内嵌 Netty HTTP Server 发送执行指令 |

**通信流程：**

1. **启动流程**：业务服务调用引擎 `POST /api/flow/start`（经网关）
2. **节点触发**：引擎状态机将节点置为 RUNNING，引擎通过**内网直连**向业务服务的 Netty HTTP Server 发送**执行指令**（含 instance_id, node_id, input, callback_token）
3. **业务执行**：业务服务在本地进程中执行 Handler
4. **结果回调**：业务服务通过 HTTP 回调引擎 `POST /api/flow/callback`（经网关，COMPLETE / FAIL）
5. **状态推进**：引擎接收回调，驱动状态机继续流转

**为什么用 Netty 内嵌 Server 而不是 Spring Controller？**

- **零侵入**：业务服务不需要为引擎额外暴露 Spring MVC 端点，SDK 自动启动 Netty HTTP Server（端口可配，默认 9999）
- **轻量**：Netty HTTP Server 仅处理引擎的执行指令请求，资源占用极低（几十 MB 内存）
- **绕过网关**：引擎与业务服务在同一内网，直连 IP:Port，避免网关转发延迟
- **可穿透**：业务服务若在 K8s Pod 内，通过服务发现获取 Pod IP 直连；若需出网，通过 NodePort / LoadBalancer 暴露
- **协议简单**：本质仍是 HTTP，调试时可用 curl 直接调业务服务的 Netty Server

> 业务服务需引入轻量级 **Client SDK**，SDK 负责：① 内嵌 Netty HTTP Server 暴露回调接口；② 根据 `handlerName` 路由到本地 Handler；③ 内置幂等、超时、熔断等治理能力。

## 二、流程定义层：DSL 输入与编译

### 2.1 为什么选 LogicFlow 的 JSON 格式作为 DSL？

我们对比过三种输入格式：

| 格式 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **BPMN 2.0 XML** | 工业标准，开源引擎兼容 | 过于复杂，元素过多，解析慢，应急场景用不到 90% 的规范 | ❌ 过重 |
| **自定义 XML** | 结构严谨 | 前后端解析成本高，扩展需改 Schema | ❌ 不友好 |
| **LogicFlow JSON** | 轻量、前后端同构、人类可读、易扩展 | 非标准，需自行定义执行语义 | ✅ 合适 |

LogicFlow JSON 的核心结构就是 `nodes[] + edges[]`，这与图论中的 DAG 定义天然一致。

### 2.2 输入 JSON 结构

```json

```

**`executionMode` 设计说明：**

| 属性 | 值 | 含义 | 任务自愈方式 |
|------|-----|------|-------------|
| `executionMode` | `AUTO` | 自动执行节点，引擎发送执行指令后等待业务服务回调 | **回调超时检测**：引擎发送指令后启动超时计时，业务服务未在预期时间内回调则判定为中断，触发幂等重试或失败处理 |
| `executionMode` | `MANUAL` | 人工审批节点，进入 RUNNING 后挂起等待外部操作 | **超时检测**：按业务超时（如 48h）判定，超时后触发升级/转交 |

> 若节点未显式配置 `executionMode`，默认根据 `type` 推断：`service`/`receive` 默认为 `AUTO`；`userTask` 默认为 `MANUAL`。

### 2.3 编译器设计

> **实现要点**：`FlowDefinitionCompiler` 负责将 LogicFlow JSON 解析为内存 DAG 模型。核心步骤包括：节点类型注册校验 → 边构建（邻接表 + 入度表）→ 严格校验（环检测 DFS、孤立节点 BFS 可达性、条件表达式语法、executionMode 与 livenessCheck 合法性）。编译产出为不可变的 `CompiledProcessDefinition`，一次编译多次运行。

### 2.4 动态加载与场景映射

流程定义**以数据库为唯一主存储**，不依赖启动时全量加载到内存。引擎支持两种启动模式：

1. **固定流程**：直接指定 `processKey` + `version` 启动
2. **动态场景**：传入 `businessCode` + `sceneCode`，由场景映射表解析为具体的流程定义

**典型场景：**
- A 业务：`start("incident_report", 3, variables)` —— 固定流程
- B 业务：`startByScene("SECURITY", "FIRE_ALERT", variables)` —— 动态映射到 `fire_response_v2`

#### 核心模型

- `ProcessDefinitionVersion`：流程定义的版本化模型，包含 processKey、version、rawJson、status（DRAFT/PUBLISHED/DEPRECATED）
- `FlowSceneMapping`：业务编码 + 场景编码 → processKey + version 的动态映射

#### 流程定义仓库（数据库即主存储）

`ProcessDefinitionRepository` 从数据库实时加载 raw_json 并编译为内存模型。编译结果可本地缓存加速，但数据库是唯一可信源。支持固定版本加载（`load(key, version)`）和最新版本加载（`loadLatest(key)`）。

#### 场景映射管理

`SceneMappingRepository` 根据 `businessCode + sceneCode` 解析对应的流程定义版本，实现业务场景与流程的动态绑定。

#### 可选的本地二级缓存

`ProcessDefinitionCache` 作为本地二级缓存（ConcurrentHashMap），流程发布或回滚时失效对应缓存项。

**版本策略：**

| 场景 | 行为 |
|------|------|
| 编辑保存 | 生成新的 `DRAFT` 版本，存入数据库 |
| 发布 | DRAFT → PUBLISHED，可选刷新本地缓存 |
| 启动流程实例 | 实时从数据库加载（或走本地缓存），**必须校验 PUBLISHED 状态** |
| 已运行的实例 | **绑定到启动时的 version**，不受后续发布影响 |
| 场景映射变更 | 修改 `flow_scene_mapping` 指向新流程版本，新实例即时生效 |
| 回滚 | 旧版本重新标记为 PUBLISHED，新版本标记为 DEPRECATED，失效缓存 |

**核心约束：版本间不兼容，每次修改均为破坏性变更**

本引擎**不支持**同一 `processKey` 下版本间的向后兼容或平滑升级。设计原则如下：

1. **DSL 是完整流程图**：修改节点拓扑（增删节点、改连线）是常见操作，不存在语义化的"补丁"概念，任何修改都是全新流程
2. **实例强绑定版本**：流程实例启动时锁定 `processKey + version`，运行时从数据库加载对应版本的 raw_json 编译执行，与后续发布完全隔离
3. **升级即替换**：业务升级通过修改 `flow_scene_mapping` 指向新版本实现，老实例继续用老版本跑完，新实例走新版本
4. **简化心智负担**：不需要维护版本间差异、兼容性校验、实例迁移逻辑，降低引擎复杂度

#### 历史数据与版本追溯

版本升级不会影响已产生的历史数据，设计保障如下：

| 数据 | 归属 | 生命周期 | 版本升级时的处理 |
|------|------|---------|----------------|
| 流程实例 (`flow_instance`) | 绑定到 `processKey + version` | 永久保留 | **不删除**，按 `processKey + version` 可查 |
| 活动实例 (`flow_activity`) | 归属到 `instanceId` | 永久保留 | **不删除**，可完整回放节点执行轨迹 |
| 事件日志 (`flow_event_log`) | 归属到 `instanceId` | 永久保留 | **不删除**，审计与对账的依据 |
| 流程定义 (`flow_definition`) | 按 `processKey + version` 存储 | 永久保留 | **不物理删除**，标记为 `DEPRECATED` 仅禁止启动新实例 |
| Token 记录 (`flow_token`) | 归属到 `instanceId` | 流程结束后可清理 | 流程终态后可异步归档或删除 |

**版本追溯示例：**

**DEPRECATED 的含义：**
- ❌ 禁止启动新的流程实例
- ✅ 正在运行的实例不受影响，继续执行
- ✅ 历史实例的查询、审计、回放不受影响
- ✅ `raw_json` 永久保留在数据库中，支持历史追溯

## 三、运行时核心：状态机引擎

### 3.1 流程实例生命周期

```
CREATED → RUNNING → [COMPLETED / TERMINATED / FAILED]
```

- **CREATED**：实例刚创建，尚未开始执行
- **RUNNING**：至少有一个节点处于非终态
- **COMPLETED**：所有路径都到达 end 节点，正常结束
- **TERMINATED**：被外部强制终止
- **FAILED**：关键节点失败且无降级路径，流程异常结束

### 3.2 活动实例状态机

每个节点实例（ActivityInstance）有独立的状态机，这是引擎的核心。

#### 状态定义

每个节点实例（ActivityInstance）有独立的状态机，状态定义如下：

- `PENDING`：等待上游节点完成，尚未激活
- `RUNNING`：已激活，正在执行或等待执行
- `COMPLETED`：执行成功完成
- `FAILED`：执行失败（可进入重试或降级分支）
- `SKIPPED`：被条件分支跳过
- `CANCELLED` / `TERMINATED`：流程级取消或强制终止

#### 事件定义

驱动状态流转的事件：`START`、`COMPLETE`、`FAIL`、`TIMEOUT`、`RETRY`、`SKIP`、`CANCEL`、`TERMINATE`。

#### 状态转移矩阵

| 当前状态 \ 事件 | START | COMPLETE | FAIL | TIMEOUT | RETRY | SKIP | CANCEL | TERMINATE |
|--------------|-------|----------|------|---------|-------|------|--------|-----------|
| **PENDING** | RUNNING | — | — | — | — | SKIPPED | CANCELLED | TERMINATED |
| **RUNNING** | — | COMPLETED | FAILED | FAILED | RUNNING | — | CANCELLED | TERMINATED |
| **FAILED** | — | — | — | — | RUNNING | — | CANCELLED | TERMINATED |
| **COMPLETED** | — | — | — | — | — | — | CANCELLED | — |
| **SKIPPED** | — | — | — | — | — | — | CANCELLED | — |

> 注：COMPLETED 不接受 TERMINATE，是因为已完成的节点不应被强制回滚，避免下游已执行节点产生不一致。

#### 状态机实现

> **实现要点**：采用**预定义状态转移矩阵**（EnumMap 实现），`fire()` 方法接收当前状态和事件，查表得到目标状态。状态变更同步持久化到 MySQL（乐观锁防并发），然后发布状态变更事件到事件总线，驱动后续逻辑。

### 3.3 事件总线

状态机只负责"状态转移的原子性"，状态转移后产生的事件通过事件总线分发，解耦状态机与后续动作。事件总线采用进程内异步发布模式，订阅方包括：超时监听器、流程驱动器（处理下游节点）、外部系统通知器。

### 4.2 流程驱动器

流程驱动器是流程执行的"大脑"，职责是：**"收到节点完成事件后，决定下游哪些节点可以执行"**。

### 4.3 执行器与线程模型

流程驱动器决定"执行什么"，执行器负责"怎么执行"。由于引擎是**独立服务**，业务 Handler 在业务服务进程中执行，引擎内部的线程池只负责**调度、回调、检测**，不执行业务代码。

**引擎内部线程模型：**

| 场景 | 线程策略 | 原因 |
|------|---------|------|
| HTTP API 请求处理 | Netty / Tomcat 线程池 | 接收业务服务的 start / complete / callback 请求 |
| 状态机调度 | 单线程无锁调度器（每分片一个） | 驱动 DAG 拓扑和状态流转，避免并发竞态 |
| 回调业务服务 | 回调线程池 | 向业务服务内嵌 Netty HTTP Server 发起执行指令 |
| 等待型节点 | 虚拟线程（Java 21+）或 waiting 线程池 | 长阻塞等待业务回调，不占平台线程 |
| 超时 / 心跳检测 | 独立后台线程 + Redis ZSET | 与业务执行完全隔离 |

**线程池划分原则：**

- **状态机调度线程**：每个 `instance_id` 哈希分片绑定一个单线程调度器（类似 Netty EventLoop），保证同一实例的所有状态变更串行处理
- **回调线程池**：向业务服务发送执行指令，线程数按业务服务数量 × 并发度配置，有界队列防止堆积
- **等待型任务**：Java 21 优先使用虚拟线程（`Executors.newVirtualThreadPerTaskExecutor()`），百万级并发无压力；低版本使用大线程数 + `SynchronousQueue` 的线程池
- **后台检测线程**：1~2 个线程，负责心跳扫描、超时检测、升级链触发

### 4.4 执行模式与任务自愈策略

流程引擎同时支持**自动执行**（`AUTO`）和**人工审批**（`MANUAL`）两种节点执行模式。两种模式下，节点的 RUNNING 状态代表完全不同的语义，因此任务自愈策略必须区分设计。

#### 核心差异

| 维度 | 自动执行（AUTO） | 人工审批（MANUAL） |
|------|----------------|-------------------|
| **RUNNING 含义** | 引擎已向业务服务发送执行指令，等待回调 | 任务已派发给审批人，等待人工操作 |
| **执行线程** | ❌ 引擎内不占业务线程；业务服务中执行 Handler | ❌ 无，流程挂起 |
| **状态推进方式** | 业务服务执行完成后回调引擎触发 COMPLETE | 依赖外部调用 `complete(taskId, action)` |
| **任务自愈检测依据** | **回调超时**（引擎发送指令后，业务服务未在预期时间内回调） | **业务超时**（如 48h 未审批） |
| **liveness 表现** | 业务服务重启或网络中断，回调丢失，任务僵死 | 审批人遗漏/离职/通知丢失，任务无人处理 |
| **恢复动作** | 自动重试（幂等回调）或按超时失败处理 | 催办、升级上级、自动转交 |

### 4.5 外部调用失败的容错处理

流程引擎中大量节点依赖外部 HTTP 接口调用（如发送通知、查询数据、同步指令）。这类调用受网络抖动、服务端故障、限流等影响，失败处理策略必须体系化设计，不能简单抛异常中断流程。

#### 失败分类

HTTP 调用失败首先按**是否可重试**分类，这是后续一切策略的前提。

**可重试错误**：网络超时、连接断开、服务端返回 5xx、限流 429。这类错误的特征是请求可能未到达服务端，或服务端尚未处理，重试不会破坏数据一致性。

**不可重试错误**：请求参数错误 400、鉴权失败 401/403、业务规则拒绝 422。这类错误的特征是请求已被服务端正确接收并明确拒绝，重复调用结果不变，重试只会浪费资源。

分类判定由节点的 Behavior 层完成，将分类结果随失败事件一并提交给状态机。

#### 可重试失败的处理链路

可重试失败走**状态机 + 延迟重试**链路，而非 Behavior 层内循环重试。

节点 Behavior 捕获可重试异常后，调用执行上下文进入失败状态。状态机接收 FAIL 事件，将节点从 RUNNING 转移到 FAILED，同步持久化到活动实例表，记录失败原因和当前重试次数。随后事件总线发布节点失败事件，由重试调度器消费。

重试调度器不立刻触发重试，而是将任务投入延迟队列（如 Redis ZSET 或本地时间轮），按**指数退避**策略计算下次执行时间。首次失败延迟较短，随重试次数增加逐渐拉长，避免对故障服务端造成流量冲击。到达重试时间后，调度器向状态机发送 RETRY 事件，节点从 FAILED 回到 RUNNING，Behavior 被重新调度执行。

重试次数上限由流程定义中节点的 retryTimes 属性控制。达到上限仍未成功，则视为重试耗尽，转入最终失败处理。

#### 重试的幂等性保证

可重试链路的核心风险是**请求实际已成功，但响应丢失**，导致客户端误判为失败并重复调用，产生数据重复。

幂等方案按优先级选择：

1. **业务层幂等**：每次调用生成唯一幂等键，由请求头或请求体携带。下游服务端用 Redis 或数据库做去重判断。这是最可靠的方案，要求下游接口配合支持。

2. **查询补偿**：调用前或失败后，先调用下游查询接口确认目标操作是否已完成。若已生效则跳过本次调用，直接按成功处理。适用于下游提供完善查询能力的场景。

3. **流程级本地事务表**：对关键指令类调用，在流程引擎内部维护调用事务表。记录请求已发送状态，收到明确成功回执后更新为已完成。未收到回执且需重试时，复用同一事务 ID 发起调用，下游可据此去重。

无论采用哪种方案，重试时必须保证**同一节点实例、同一重试轮次**的幂等键全局唯一，且随持久化记录留存，服务重启后可恢复。

#### 不可重试失败与重试耗尽后的降级

不可重试错误，或重试次数已耗尽的可重试错误，均进入最终失败处理。此时流程不应直接异常结束，而是尝试驱动**降级分支**。

流程定义中的边支持条件表达式。节点失败后，流程驱动器遍历该节点的所有出边，评估边上的条件表达式。典型设计是在主流程边之外配置一条失败降级边，条件为失败状态或特定错误码匹配。满足条件时，Token 被传递给降级目标节点，流程继续执行补偿逻辑（如转人工介入、记录异常日志、触发兜底通知）。

若节点无配置降级分支，则流程按 FAILED 终态结束。

#### 服务重启后的恢复

可重试失败的恢复是 RecoveryManager 的重点场景之一。

**引擎服务**启动扫描时，对于状态为 RUNNING 但超过回调超时时间未收到业务服务回调的节点，判定为引擎与业务服务之间的调用中断。若该节点配置了重试，引擎直接向业务服务重新发起幂等执行指令（复用同一幂等键），不等待延迟时间。

对于状态为 FAILED 且重试次数未满的节点，检查延迟队列中是否仍存在对应的重试任务。若任务丢失（如 Redis 数据过期），则由恢复线程重新计算延迟时间并重新投递，确保重试不中断。

对于状态为 FAILED 且重试已耗尽的节点，直接进入降级分支恢复逻辑，重新评估出边条件并驱动 Token 流转。

### 5.3 服务重启恢复机制

> **实现要点**：`RecoveryManager` 在**引擎服务**启动时（`@PostConstruct`）和定时任务中扫描未完成的实例。恢复策略区分执行模式：
> - **AUTO 节点**：检查节点状态为 RUNNING 但超过回调超时时间未收到业务服务回调，判定为中断。引擎重新向业务服务发起幂等执行指令（复用同一幂等键），或按超时失败处理

> - **Token 恢复**：将数据库中活跃 Token 重新加载到各引擎节点的流程驱动器内存（按 `instance_id` 哈希路由到对应节点）

### 7.1 节点类型注册（客户端 SDK + 自动发现）

引擎以独立服务部署，业务 Handler 在业务服务进程中执行。节点类型的注册和发现由**客户端 SDK** 负责：

**注册方式：**

1. **注解注册**：业务方在 Handler 类上标注 `@FlowHandler(nodeType = "smsNotify")`，SDK 启动时自动扫描并注册到引擎
2. **手动注册**：`FlowClient.register("smsNotify", new SmsNotifyHandler())`
3. **动态脚本**：引擎内置轻量脚本引擎（Groovy / Aviator），支持无代码节点

**执行流程：**

1. 引擎状态机将节点置为 RUNNING
2. 引擎通过内网直连向业务服务的 Netty HTTP Server 发送执行指令（含 `handlerName`, `input`, `callbackToken`）
3. 业务服务的 Client SDK 接收指令，根据 `handlerName` 路由到本地注册的 Handler
4. Handler 执行完成后，SDK 自动回调引擎 `POST /api/flow/callback`

**SDK 内置治理能力：**

- 幂等去重（同一执行指令多次投递只执行一次）
- 超时熔断（Handler 执行超时时自动中断并回调 FAIL）
- 失败重试（回调引擎失败时本地排队重试）
- 服务注册（SDK 启动时向引擎注册本服务的地址和可用 Handler 列表）

## 九、部署与扩展性

### 9.1 水平扩展：按 Instance 分片

引擎以独立服务集群部署，水平扩展的核心是**同一流程实例的所有状态变更必须路由到同一引擎节点**，保证状态机串行化。

**分片策略：**

- 分片键：`instance_id` 的 hash 值
- 路由层：API 网关或负载均衡器（如 Nginx / Traefik / K8s Ingress）根据 `instance_id` 哈希将请求路由到固定引擎节点
- 粘性会话：同一实例的 start、callback、query 请求均路由到同一节点

**扩容场景：**

- 引擎节点增加时，部分实例哈希值落入新节点，需做**渐进式迁移**（新实例走新节点，老实例仍在老节点直到结束）
- 引擎节点故障时，该节点负责的实例由其他节点通过 **RecoveryManager** 接管（从数据库加载 Token 和状态继续驱动）

### 9.2 高可用设计

| 组件 | 高可用策略 |
|------|-----------|
| 引擎服务 | 多实例部署 + 无状态设计（状态在 DB / Redis），单点故障可快速切换 |
| MySQL | 主从复制 + 读写分离，状态变更写主库，查询走从库 |
| Redis | Cluster 模式，延迟队列和分布式锁高可用 |
| RocketMQ | 多 Broker + 多副本，确保回调消息不丢失 |
| API 网关 | 多实例 + 健康检查，自动摘除故障引擎节点 |

### 9.3 多租户（SaaS）

引擎独立服务天然适合 SaaS 化，所有租户共用同一引擎集群，通过数据隔离实现多租户：

**隔离方案（行级隔离为主）：**

- 核心表增加 `tenant_id` 字段：`flow_definition`、`flow_instance`、`flow_activity`、`flow_token`
- 租户认证：业务服务调用引擎 API 时携带租户 Token（如 Header `X-Tenant-Id`），网关层解析后注入到引擎上下文
- 流程定义隔离：平台提供公共模板（`tenant_id = 'platform'`），租户可复制后自定义
- 资源配额：按租户限制最大并发实例数、最大重试队列深度、回调线程池配额

**数据安全：**

- 租户间数据不可见：所有查询自动注入 `tenant_id` 过滤条件
- 审计隔离：`flow_event_log` 按租户分表或分区归档

### 9.4 客户端 SDK

业务服务接入引擎需引入轻量级 **Client SDK**（类似 XXL-JOB 的执行器）：

**SDK 职责：**

1. **服务注册**：启动时向引擎注册本服务地址（Netty HTTP Server 的 IP:Port）、可用 Handler 列表、租户信息
2. **回调监听**：内嵌 Netty HTTP Server（端口可配，默认 9999）接收引擎的执行指令
3. **Handler 路由**：根据 `handlerName` 将执行指令分发给本地注册的 Handler
4. **幂等保障**：同一执行指令多次投递时，SDK 层去重（基于 `instance_id + node_id + retry_count`）
5. **超时熔断**：Handler 执行超时时自动中断线程并回调 FAIL
6. **失败重试**：回调引擎失败时本地排队重试，避免状态丢失

**接入成本：**

- 引入 SDK（Maven 依赖）
- 在 Handler 类上加 `@FlowHandler` 注解
- 配置引擎地址和租户 Token
- 无侵入性：业务代码只关心 Handler 实现，状态机、持久化、超时检测全部由引擎托管

## 十、面试速查

### Q1：为什么自研，不用 Flowable？

> 应急通信是"指令调度流"而非"人工审批流"，核心特征是超时升级链、GIS 联动、设备可靠投递。Flowable 的 BPMN 模型过重（历史表、变量表、事件日志表），且这些领域概念无法直接映射到 BPMN 规范。我们选择**轻量级 JSON DSL + 领域专用执行引擎**，用 1/10 的代码量实现了应急场景所需的全部语义，且性能更优。

### Q2：引擎服务重启如何保证不丢状态？

> 三个机制：① **活动级快照**：每个节点状态变更同步写 MySQL（乐观锁保证并发安全）；② **Token 持久化**：节点完成后移动到下游时写库，重启后按 `instance_id` 哈希路由恢复 Token 到对应引擎节点；③ **任务自愈检测**：按节点执行模式区分策略——**自动节点**（`AUTO`）引擎发送执行指令后启动回调超时计时，若业务服务未在预期时间内回调则判定为中断，引擎重新发起幂等执行指令；**人工节点**（`MANUAL`）不依赖回调，而是按业务超时（如 48h 未审批）检测，超时后触发催办、升级或自动转交。

### Q3：引擎支持多大并发？

> 流程驱动器是**单线程无锁**的，只负责驱动 DAG 拓扑和状态流转；向业务服务发起回调由独立线程池异步执行。单机支撑 **5000+ TPS** 的流程事件，瓶颈通常在回调业务服务的 HTTP 延迟或 DB。水平扩展时引擎集群按 `instance_id` 哈希分片，同一实例的路由到同一引擎节点，保证状态机串行化。业务服务可独立扩缩容，与引擎集群解耦。

### Q4：升级链的定时任务如果挂了怎么办？

> ① 使用 **Redis ZSET** 做分布式延迟队列，多实例部署时只有一个实例能拿到到期任务（`ZREVRANGEBYSCORE + ZREM` 的原子性保证）；② 任务执行前**双重检查**节点是否仍在 RUNNING，防止误触发；③ 节点完成时**批量取消**后续升级任务，取消操作是幂等的。

### Q5：设备指令如果设备永远不回执怎么办？

> 三层兜底：① **MQ Confirm**：确保消息到达 Broker；② **本地事务表 + 定时重试**：`SENT` 状态超过 60s 未 `ACKED` 则自动重试，最多 3 次；③ **流程级超时**：重试耗尽后状态机触发 `FAIL`，流程进入预定义的降级分支（如人工介入节点）。

---

> **核心设计哲学**：编译时做尽所有校验，运行时保持极简，故障时快速恢复。所有复杂性（表达式解析、条件判断、并发控制）都推给编译期和基础设施，执行引擎本身只负责"收到事件 → 查表转移 → 驱动下游"。
