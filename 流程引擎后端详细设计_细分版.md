**分层原则：**

| 层级 | 职责 | 设计原则 |
|------|------|---------|
| 编译层 | 将外部 JSON 转换为内部可执行模型 | **一次编译，多次运行**；编译时做尽所有校验 |
| 运行时核心 | 驱动状态流转，不碰业务逻辑 | **无锁、单线程调度**；状态转移原子化 |
| 调度执行层 | 决定"什么时候执行什么"，执行业务代码 | **调度与执行分离**；业务代码异步化 |
| 基础设施 | 存储、消息、缓存 | 运行时核心**不直接依赖**基础设施，通过接口抽象 |

## 二、流程定义层：DSL 输入与编译

### 2.1 为什么选 LogicFlow 的 JSON 格式作为 DSL？

我们对比过三种输入格式：

| 格式 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **BPMN 2.0 XML** | 工业标准，开源引擎兼容 | 过于复杂，元素过多，解析慢，应急场景用不到 90% 的规范 | ❌ 过重 |
| **自定义 XML** | 结构严谨 | 前后端解析成本高，扩展需改 Schema | ❌ 不友好 |
| **LogicFlow JSON** | 轻量、前后端同构、人类可读、易扩展 | 非标准，需自行定义执行语义 | ✅ 合适 |

LogicFlow JSON 的核心结构就是 `nodes[] + edges[]`，这与图论中的 DAG 定义天然一致。

### 2.2 输入 JSON 结构

```json
{

    "processKey": "emergency_dispatch_v2",
    "version": 3,
    "name": "突发事件三级响应流程"

      "id": "start_1",
      "type": "start",

      "type": "userTask",

        "executionMode": "MANUAL",

        "livenessCheck": {
          "type": "timeout",
          "timeout": "PT48H",
          "escalation": "notifyBoss"

    },
    {
      "id": "auto_archive",
      "type": "service",
      "properties": {
        "executionMode": "AUTO",
        "handler": "ArchiveHandler",
        "timeout": 30,
        "livenessCheck": {
          "type": "heartbeat",
          "interval": "10s",
          "timeout": "60s",
          "action": "retry"
        }
      }
    },
    {
      "id": "end_1",
      "type": "end",
      "properties": {}
    }
  ],
  "edges": [
    { "id": "e1", "sourceNodeId": "start_1", "targetNodeId": "create_incident" },
    { "id": "e2", "sourceNodeId": "create_incident", "targetNodeId": "notify_sms" },
    { "id": "e3", "sourceNodeId": "notify_sms", "targetNodeId": "wait_response" },
    { "id": "e4", "sourceNodeId": "wait_response", "targetNodeId": "leader_approval" },
    { "id": "e5", "sourceNodeId": "leader_approval", "targetNodeId": "auto_archive" },
    { "id": "e6", "sourceNodeId": "auto_archive", "targetNodeId": "end_1", "properties": { "condition": "${allSuccess}" } }
  ]
}
```

**`executionMode` 设计说明：**

| 属性 | 值 | 含义 | 任务自愈方式 |
|------|-----|------|-------------|
| `executionMode` | `AUTO` | 自动执行节点，引擎提交到线程池执行 | **心跳检测**：后台线程每 10s 写心跳，重启后按心跳超时判定 |
| `executionMode` | `MANUAL` | 人工审批节点，进入 RUNNING 后挂起等待外部操作 | **超时检测**：按业务超时（如 48h）判定，超时后触发升级/转交 |

> 若节点未显式配置 `executionMode`，默认根据 `type` 推断：`service`/`receive` 默认为 `AUTO`；`userTask` 默认为 `MANUAL`。

### 2.3 编译器设计

> **实现要点**：`FlowDefinitionCompiler` 负责将 LogicFlow JSON 解析为内存 DAG 模型。核心步骤包括：节点类型注册校验 → 边构建（邻接表 + 入度表）→ 严格校验（环检测 DFS、孤立节点 BFS 可达性、条件表达式语法、executionMode 与 livenessCheck 合法性）。编译产出为不可变的 `CompiledProcessDefinition`，一次编译多次运行。

### 2.4 动态加载与场景映射

流程定义**以数据库为唯一主存储**，不依赖启动时全量加载到内存。引擎支持两种启动模式：

1. **固定流程**：直接指定 `processKey` + `version` 启动
2. **动态场景**：传入 `businessCode` + `sceneCode`，由场景映射表解析为具体的流程定义

**典型场景：**
- A 业务：`start("incident_report", 3, variables)` —— 固定流程
- B 业务：`startByScene("SECURITY", "FIRE_ALERT", variables)` —— 动态映射到 `fire_response_v2`

#### 核心模型

- `ProcessDefinitionVersion`：流程定义的版本化模型，包含 processKey、version、rawJson、status（DRAFT/PUBLISHED/DEPRECATED）
- `FlowSceneMapping`：业务编码 + 场景编码 → processKey + version 的动态映射

#### 流程定义仓库（数据库即主存储）

`ProcessDefinitionRepository` 从数据库实时加载 raw_json 并编译为内存模型。编译结果可本地缓存加速，但数据库是唯一可信源。支持固定版本加载（`load(key, version)`）和最新版本加载（`loadLatest(key)`）。

#### 场景映射管理

`SceneMappingRepository` 根据 `businessCode + sceneCode` 解析对应的流程定义版本，实现业务场景与流程的动态绑定。

#### 可选的本地二级缓存

`ProcessDefinitionCache` 作为本地二级缓存（ConcurrentHashMap），流程发布或回滚时失效对应缓存项。

**版本策略：**

| 场景 | 行为 |
|------|------|
| 编辑保存 | 生成新的 `DRAFT` 版本，存入数据库 |
| 发布 | DRAFT → PUBLISHED，可选刷新本地缓存 |
| 启动流程实例 | 实时从数据库加载（或走本地缓存），**必须校验 PUBLISHED 状态** |
| 已运行的实例 | **绑定到启动时的 version**，不受后续发布影响 |
| 场景映射变更 | 修改 `flow_scene_mapping` 指向新流程版本，新实例即时生效 |
| 回滚 | 旧版本重新标记为 PUBLISHED，新版本标记为 DEPRECATED，失效缓存 |

**核心约束：版本间不兼容，每次修改均为破坏性变更**

本引擎**不支持**同一 `processKey` 下版本间的向后兼容或平滑升级。设计原则如下：

1. **DSL 是完整流程图**：修改节点拓扑（增删节点、改连线）是常见操作，不存在语义化的"补丁"概念，任何修改都是全新流程
2. **实例强绑定版本**：流程实例启动时锁定 `processKey + version`，运行时从数据库加载对应版本的 raw_json 编译执行，与后续发布完全隔离
3. **升级即替换**：业务升级通过修改 `flow_scene_mapping` 指向新版本实现，老实例继续用老版本跑完，新实例走新版本
4. **简化心智负担**：不需要维护版本间差异、兼容性校验、实例迁移逻辑，降低引擎复杂度

#### 历史数据与版本追溯

版本升级不会影响已产生的历史数据，设计保障如下：

| 数据 | 归属 | 生命周期 | 版本升级时的处理 |
|------|------|---------|----------------|
| 流程实例 (`flow_instance`) | 绑定到 `processKey + version` | 永久保留 | **不删除**，按 `processKey + version` 可查 |
| 活动实例 (`flow_activity`) | 归属到 `instanceId` | 永久保留 | **不删除**，可完整回放节点执行轨迹 |
| 事件日志 (`flow_event_log`) | 归属到 `instanceId` | 永久保留 | **不删除**，审计与对账的依据 |
| 流程定义 (`flow_definition`) | 按 `processKey + version` 存储 | 永久保留 | **不物理删除**，标记为 `DEPRECATED` 仅禁止启动新实例 |
| Token 记录 (`flow_token`) | 归属到 `instanceId` | 流程结束后可清理 | 流程终态后可异步归档或删除 |

**版本追溯示例：**

**DEPRECATED 的含义：**
- ❌ 禁止启动新的流程实例
- ✅ 正在运行的实例不受影响，继续执行
- ✅ 历史实例的查询、审计、回放不受影响
- ✅ `raw_json` 永久保留在数据库中，支持历史追溯

## 三、运行时核心：状态机引擎

### 3.1 流程实例生命周期

```
CREATED → RUNNING → [COMPLETED / TERMINATED / FAILED]
```

- **CREATED**：实例刚创建，尚未开始执行
- **RUNNING**：至少有一个节点处于非终态
- **COMPLETED**：所有路径都到达 end 节点，正常结束
- **TERMINATED**：被外部强制终止
- **FAILED**：关键节点失败且无降级路径，流程异常结束

### 3.2 活动实例状态机

每个节点实例（ActivityInstance）有独立的状态机，这是引擎的核心。

#### 状态定义

每个节点实例（ActivityInstance）有独立的状态机，状态定义如下：

- `PENDING`：等待上游节点完成，尚未激活
- `RUNNING`：已激活，正在执行或等待执行
- `COMPLETED`：执行成功完成
- `FAILED`：执行失败（可进入重试或降级分支）
- `SKIPPED`：被条件分支跳过
- `CANCELLED` / `TERMINATED`：流程级取消或强制终止

#### 事件定义

驱动状态流转的事件：`START`、`COMPLETE`、`FAIL`、`TIMEOUT`、`RETRY`、`SKIP`、`CANCEL`、`TERMINATE`。

#### 状态转移矩阵

| 当前状态 \ 事件 | START | COMPLETE | FAIL | TIMEOUT | RETRY | SKIP | CANCEL | TERMINATE |
|--------------|-------|----------|------|---------|-------|------|--------|-----------|
| **PENDING** | RUNNING | — | — | — | — | SKIPPED | CANCELLED | TERMINATED |
| **RUNNING** | — | COMPLETED | FAILED | FAILED | RUNNING | — | CANCELLED | TERMINATED |
| **FAILED** | — | — | — | — | RUNNING | — | CANCELLED | TERMINATED |
| **COMPLETED** | — | — | — | — | — | — | CANCELLED | — |
| **SKIPPED** | — | — | — | — | — | — | CANCELLED | — |

> 注：COMPLETED 不接受 TERMINATE，是因为已完成的节点不应被强制回滚，避免下游已执行节点产生不一致。

#### 状态机实现

> **实现要点**：采用**预定义状态转移矩阵**（EnumMap 实现），`fire()` 方法接收当前状态和事件，查表得到目标状态。状态变更同步持久化到 MySQL（乐观锁防并发），然后发布状态变更事件到事件总线，驱动后续逻辑。

### 3.3 事件总线

状态机只负责"状态转移的原子性"，状态转移后产生的事件通过事件总线分发，解耦状态机与后续动作。事件总线采用进程内异步发布模式，订阅方包括：超时监听器、流程驱动器（处理下游节点）、外部系统通知器。

### 4.2 流程驱动器

流程驱动器是流程执行的"大脑"，职责是：**"收到节点完成事件后，决定下游哪些节点可以执行"**。

### 4.3 执行器与线程模型

流程驱动器决定"执行什么"，执行器负责"怎么执行"。

**线程模型设计决策：**

| 场景 | 线程策略 | 原因 |
|------|---------|------|
| 普通 service 节点 | 业务线程池 | 快速执行，快速释放 |
| receive / deviceCmd 节点 | waiting 线程池 | 可能阻塞数分钟等回执，不能用有限线程池 |
| 定时 / 延迟任务 | 调度线程（Quartz/xxl-job） | 不占用执行线程 |
| 超时检测 | 独立后台线程 + Redis ZSET | 与业务执行完全隔离 |

### 4.4 执行模式与任务自愈策略

流程引擎同时支持**自动执行**（`AUTO`）和**人工审批**（`MANUAL`）两种节点执行模式。两种模式下，节点的 RUNNING 状态代表完全不同的语义，因此任务自愈策略必须区分设计。

#### 核心差异

| 维度 | 自动执行（AUTO） | 人工审批（MANUAL） |
|------|----------------|-------------------|
| **RUNNING 含义** | 后台线程正在执行业务逻辑 | 任务已派发给审批人，等待人工操作 |
| **执行线程** | ✅ 有，占用线程池 | ❌ 无，流程挂起 |
| **状态推进方式** | 线程执行完成后自动触发 COMPLETE | 依赖外部调用 `complete(taskId, action)` |
| **任务自愈检测依据** | **心跳超时**（线程每 10s 写心跳） | **业务超时**（如 48h 未审批） |
| ** liveness 表现** | 执行线程随服务重启死亡，任务僵死 | 审批人遗漏/离职/通知丢失，任务无人处理 |
| **恢复动作** | 自动重试（幂等）或按超时失败处理 | 催办、升级上级、自动转交 |

## 五、持久化与容错

### 5.1 核心表结构设计

```sql
-- 流程定义版本表
CREATE TABLE flow_definition (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    process_key VARCHAR(64) NOT NULL COMMENT '流程标识',
    version INT NOT NULL COMMENT '版本号',
    name VARCHAR(128) COMMENT '流程名称',
    raw_json TEXT NOT NULL COMMENT '原始LogicFlow JSON',
    status VARCHAR(16) NOT NULL COMMENT 'DRAFT/PUBLISHED/DEPRECATED',
    publish_time DATETIME,
    created_by VARCHAR(64),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_process_version (process_key, version)
) COMMENT '流程定义';

-- 场景映射表（支持动态配置业务-场景-流程的映射关系）
CREATE TABLE flow_scene_mapping (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    business_code VARCHAR(64) NOT NULL COMMENT '业务编码，如 SECURITY',
    scene_code VARCHAR(64) NOT NULL COMMENT '场景编码，如 FIRE_ALERT',
    process_key VARCHAR(64) NOT NULL COMMENT '映射到的流程标识',
    version INT NOT NULL COMMENT '映射到的版本号',
    status TINYINT DEFAULT 1 COMMENT '0=DISABLED, 1=ENABLED',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_business_scene (business_code, scene_code),
    KEY idx_process_key (process_key, version)
) COMMENT '场景映射';

-- 流程实例表
CREATE TABLE flow_instance (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
```

### 5.2 持久化策略矩阵

| 数据 | 时机 | 方式 | 同步/异步 | 目的 |
|------|------|------|----------|------|
| 流程实例创建 | `start()` 调用时 | INSERT | **同步** | 实例存在性保证 |
| 活动状态变更 | `stateMachine.fire()` 内部 | UPDATE（乐观锁） | **同步** | 服务重启可恢复 |
| 活动输入/输出 | 节点执行完成时 | UPDATE | **同步** | 调试与审计 |
| 流程变量变更 | `setVariable()` 时 | UPDATE instance | **同步** | 下游节点依赖 |
| Token 状态 | 节点完成后移动到下游时 | INSERT/UPDATE | **同步** | 调度正确性 |
| 事件日志 | 每次状态转移 | INSERT | **异步批量** | 审计、对账、回放 |
| 心跳更新（AUTO） | RUNNING 的自动节点每 10s | UPDATE heartbeat_time | **异步** | 自动节点任务自愈检测 |
| 审批人/任务状态（MANUAL） | 节点进入 RUNNING 时 | UPDATE assignee + 写入 task 表 | **同步** | 人工节点待办恢复 |

### 5.3 服务重启恢复机制

> **实现要点**：`RecoveryManager` 在应用启动时（`@PostConstruct`）和定时任务中扫描未完成的实例。恢复策略区分执行模式：
> - **AUTO 节点**：读取心跳时间，若超时（默认 60s）则判定为中断，根据配置触发幂等重试或按超时失败处理
> - **MANUAL 节点**：检查待办任务是否存在，若丢失则根据流程定义重新生成
> - **Token 恢复**：将数据库中活跃 Token 重新加载到流程驱动器内存

#### 实现

**关键点：**

1. **取消机制必须可靠**：节点完成时，所有后续升级任务必须从延迟队列中移除，否则会造成误通知
2. **双重检查**：拉取到期任务后，再次查库确认节点仍在 RUNNING，防止"刚好完成但尚未取消"的竞态
3. **分布式**：Redis ZSET 保证集群多实例下只有一个实例处理同一任务（利用原子操作）

## 九、面试速查

### Q1：为什么自研，不用 Flowable？

> 应急通信是"指令调度流"而非"人工审批流"，核心特征是超时升级链、GIS 联动、设备可靠投递。Flowable 的 BPMN 模型过重（历史表、变量表、事件日志表），且这些领域概念无法直接映射到 BPMN 规范。我们选择**轻量级 JSON DSL + 领域专用执行引擎**，用 1/10 的代码量实现了应急场景所需的全部语义，且性能更优。

### Q2：如何保证服务重启不丢状态？

> 三个机制：① **活动级快照**：每个节点状态变更同步写 MySQL（乐观锁保证并发安全）；② **Token 持久化**：节点完成后移动到下游时写库，重启后恢复 Token 位置；③ **任务自愈检测**：按节点执行模式区分策略——**自动节点**（`AUTO`）由后台线程每 10s 写心跳，重启后心跳超时则判定为中断，支持自动重试或按超时失败处理；**人工节点**（`MANUAL`）不依赖心跳，而是按业务超时（如 48h 未审批）检测，超时后触发催办、升级或自动转交。

### Q3：引擎支持多大并发？

> 流程驱动器是**单线程无锁**的，只负责驱动 DAG 拓扑和状态流转；业务逻辑提交到线程池异步执行。单机支撑 **5000+ TPS** 的流程事件，瓶颈通常在业务侧的 HTTP 调用或 DB。水平扩展时按 `instance_id` 哈希分片，同一实例路由到同一节点，保证状态机串行化。

### Q4：升级链的定时任务如果挂了怎么办？

> ① 使用 **Redis ZSET** 做分布式延迟队列，多实例部署时只有一个实例能拿到到期任务（`ZREVRANGEBYSCORE + ZREM` 的原子性保证）；② 任务执行前**双重检查**节点是否仍在 RUNNING，防止误触发；③ 节点完成时**批量取消**后续升级任务，取消操作是幂等的。

### Q5：设备指令如果设备永远不回执怎么办？

> 三层兜底：① **MQ Confirm**：确保消息到达 Broker；② **本地事务表 + 定时重试**：`SENT` 状态超过 60s 未 `ACKED` 则自动重试，最多 3 次；③ **流程级超时**：重试耗尽后状态机触发 `FAIL`，流程进入预定义的降级分支（如人工介入节点）。

---

> **核心设计哲学**：编译时做尽所有校验，运行时保持极简，故障时快速恢复。所有复杂性（表达式解析、条件判断、并发控制）都推给编译期和基础设施，执行引擎本身只负责"收到事件 → 查表转移 → 驱动下游"。
