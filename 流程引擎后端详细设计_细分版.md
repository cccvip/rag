**分层原则：**

| 层级 | 职责 | 设计原则 |
|------|------|---------|
| 编译层 | 将外部 JSON 转换为内部可执行模型 | **一次编译，多次运行**；编译时做尽所有校验 |
| 运行时核心 | 驱动状态流转，不碰业务逻辑 | **无锁、单线程调度**；状态转移原子化 |
| 调度执行层 | 决定"什么时候执行什么"，执行业务代码 | **调度与执行分离**；业务代码异步化 |
| 基础设施 | 存储、消息、缓存 | 运行时核心**不直接依赖**基础设施，通过接口抽象 |

## 二、流程定义层：DSL 输入与编译

### 2.1 为什么选 LogicFlow 的 JSON 格式作为 DSL？

我们对比过三种输入格式：

| 格式 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **BPMN 2.0 XML** | 工业标准，开源引擎兼容 | 过于复杂，元素过多，解析慢，应急场景用不到 90% 的规范 | ❌ 过重 |
| **自定义 XML** | 结构严谨 | 前后端解析成本高，扩展需改 Schema | ❌ 不友好 |
| **LogicFlow JSON** | 轻量、前后端同构、人类可读、易扩展 | 非标准，需自行定义执行语义 | ✅ 合适 |

LogicFlow JSON 的核心结构就是 `nodes[] + edges[]`，这与图论中的 DAG 定义天然一致。

### 2.3 编译器设计

```java
@Component
public class FlowDefinitionCompiler {
    
    @Autowired
    private NodeTypeRegistry nodeRegistry;
    
    @Autowired
    private ExpressionParser expressionParser;
    
    /**
     * 编译入口：LogicFlow JSON → 内存 ProcessDefinition
     */
    public CompiledProcessDefinition compile(String rawJson) {
        LogicFlowGraph graph = JsonUtils.parse(rawJson, LogicFlowGraph.class);
        
        // 1. 基础结构转换
        ProcessDefinition def = new ProcessDefinition();
        def.setProcessKey(graph.getProperties().getProcessKey());
        def.setVersion(graph.getProperties().getVersion());
        
        // 2. 编译节点
        Map<String, NodeDefinition> nodeMap = new HashMap<>();
        for (LFNode lfNode : graph.getNodes()) {
            NodeDefinition node = compileNode(lfNode);
            nodeMap.put(node.getId(), node);
            def.addNode(node);
        }
        
        // 3. 编译边（构建邻接表和入度表）
        for (LFEdge lfEdge : graph.getEdges()) {
            EdgeDefinition edge = compileEdge(lfEdge);
            def.addEdge(edge);
            
            // 构建运行时所需的邻接表
            def.getAdjacencyList().computeIfAbsent(edge.getSource(), k -> new ArrayList<>()).add(edge);
            def.getInDegreeMap().merge(edge.getTarget(), 1, Integer::sum);
        }
        
        // 4. 严格校验（编译时做尽所有检查）
        validate(def, nodeMap);
        
        return new CompiledProcessDefinition(def, rawJson);
    }
    
    private NodeDefinition compileNode(LFNode lfNode) {
        NodeDefinition node = new NodeDefinition();
        node.setId(lfNode.getId());
        node.setType(lfNode.getType());
        node.setProperties(lfNode.getProperties());
        
        // 校验节点类型是否已注册（后端必须支持该类型的 Behavior）
        if (!nodeRegistry.isRegistered(lfNode.getType())) {
            throw new CompileException("未注册的节点类型: " + lfNode.getType());
        }
        
        return node;
    }
    
    private EdgeDefinition compileEdge(LFEdge lfEdge) {
        EdgeDefinition edge = new EdgeDefinition();
        edge.setId(lfEdge.getId());
        edge.setSource(lfEdge.getSourceNodeId());
        edge.setTarget(lfEdge.getTargetNodeId());
        
        // 解析条件表达式（如 ${allSuccess}）
        String conditionText = lfEdge.getProperties() != null ? lfEdge.getProperties().getCondition() : null;
        if (StringUtils.hasText(conditionText)) {
            edge.setCondition(expressionParser.parse(conditionText));
        }
        
        return edge;
    }
    
    private void validate(ProcessDefinition def, Map<String, NodeDefinition> nodeMap) {
        // 校验 1：有且仅有 1 个 start 节点和至少 1 个 end 节点
        long startCount = def.getNodes().stream().filter(n -> "start".equals(n.getType())).count();
        long endCount = def.getNodes().stream().filter(n -> "end".equals(n.getType())).count();
        if (startCount != 1) throw new CompileException("流程必须有且仅有 1 个开始节点");
        if (endCount < 1) throw new CompileException("流程至少要有 1 个结束节点");
        
        // 校验 2：所有边的 source 和 target 必须存在于 nodes 中
        for (EdgeDefinition edge : def.getEdges()) {
            if (!nodeMap.containsKey(edge.getSource())) {
                throw new CompileException("边引用了不存在的节点: " + edge.getSource());
            }
            if (!nodeMap.containsKey(edge.getTarget())) {
                throw new CompileException("边引用了不存在的节点: " + edge.getTarget());
            }
        }
        
        // 校验 3：DAG 无环检测（DFS）
        if (hasCycle(def)) {
            throw new CompileException("流程定义存在循环依赖，请检查节点连线");
        }
        
        // 校验 4：孤立节点检测（无入边且无出边，或从 start 不可达）
        Set<String> reachableFromStart = bfsReachable(def, getStartNodeId(def));
        for (NodeDefinition node : def.getNodes()) {
            if (!reachableFromStart.contains(node.getId())) {
                throw new CompileException("节点不可达: " + node.getId());
            }
        }
        
        // 校验 5：Fork/Join 配对检查
        validateForkJoinPairs(def, nodeMap);
        
        // 校验 6：条件表达式语法检查
        for (EdgeDefinition edge : def.getEdges()) {
            if (edge.getCondition() != null) {
                expressionParser.validate(edge.getCondition());
            }
        }
    }
    
    private boolean hasCycle(ProcessDefinition def) {
        Set<String> visited = new HashSet<>();
        Set<String> recStack = new HashSet<>();
        
        for (String nodeId : def.getNodeIds()) {
            if (!visited.contains(nodeId)) {
                if (hasCycleDFS(nodeId, visited, recStack, def)) {
                    return true;
                }
            }
        }
        return false;
    }
    
    private boolean hasCycleDFS(String nodeId, Set<String> visited, Set<String> recStack, ProcessDefinition def) {
        visited.add(nodeId);
        recStack.add(nodeId);
        
        List<EdgeDefinition> outgoing = def.getAdjacencyList().getOrDefault(nodeId, Collections.emptyList());
        for (EdgeDefinition edge : outgoing) {
            String neighbor = edge.getTarget();
            if (!visited.contains(neighbor)) {
                if (hasCycleDFS(neighbor, visited, recStack, def)) return true;
            } else if (recStack.contains(neighbor)) {
                return true;
            }
        }
        
        recStack.remove(nodeId);
        return false;
    }
    
    private void validateForkJoinPairs(ProcessDefinition def, Map<String, NodeDefinition> nodeMap) {
        // Fork 和 Join 必须成对出现，且路径不交叉
        // 实现：找到所有 fork，沿着出边遍历，必须遇到且仅遇到一个 join，且所有分支汇聚到同一个 join
        // 若检测到交叉（如 fork A 的分支到达 join B，fork B 的分支到达 join A），则报错
    }
}
```

**版本策略：**

| 场景 | 行为 |
|------|------|
| 编辑保存 | 生成新的 `DRAFT` 版本，或覆盖当前 draft |
| 发布 | DRAFT → PUBLISHED，生成新版本号，写入缓存 |
| 启动流程实例 | 必须使用 `PUBLISHED` 状态的最新版本 |
| 已运行的实例 | **绑定到启动时的版本号**，不受后续发布影响 |
| 回滚 | 将旧版本重新标记为 PUBLISHED，新版本降级 |

## 三、运行时核心：状态机引擎

### 3.1 流程实例生命周期

```
CREATED → RUNNING → [COMPLETED / TERMINATED / FAILED]
```

- **CREATED**：实例刚创建，尚未开始执行
- **RUNNING**：至少有一个节点处于非终态
- **COMPLETED**：所有路径都到达 end 节点，正常结束
- **TERMINATED**：被外部强制终止
- **FAILED**：关键节点失败且无降级路径，流程异常结束

### 3.2 活动实例状态机

每个节点实例（ActivityInstance）有独立的状态机，这是引擎的核心。

#### 状态定义

```java
public enum ActivityState {
    PENDING,      // 等待上游节点完成，尚未激活
    RUNNING,      // 已激活，正在执行或等待执行
    COMPLETED,    // 执行成功完成
    FAILED,       // 执行失败（可进入重试或降级分支）
    SKIPPED,      // 被条件分支跳过，或被人为跳过
    CANCELLED,    // 流程终止时连带取消
    TERMINATED    // 被人为强制终止
}
```

#### 事件定义

```java
public enum StateEvent {
    START,           // 从 PENDING 激活
    COMPLETE,        // 执行成功
    FAIL,            // 执行异常
    TIMEOUT,         // 执行超时
    RETRY,           // 手动或自动重试
    SKIP,            // 跳过该节点
    CANCEL,          // 流程级取消
    TERMINATE        // 强制终止
}
```

#### 状态转移矩阵

| 当前状态 \ 事件 | START | COMPLETE | FAIL | TIMEOUT | RETRY | SKIP | CANCEL | TERMINATE |
|--------------|-------|----------|------|---------|-------|------|--------|-----------|
| **PENDING** | RUNNING | — | — | — | — | SKIPPED | CANCELLED | TERMINATED |
| **RUNNING** | — | COMPLETED | FAILED | FAILED | RUNNING | — | CANCELLED | TERMINATED |
| **FAILED** | — | — | — | — | RUNNING | — | CANCELLED | TERMINATED |
| **COMPLETED** | — | — | — | — | — | — | CANCELLED | — |
| **SKIPPED** | — | — | — | — | — | — | CANCELLED | — |

> 注：COMPLETED 不接受 TERMINATE，是因为已完成的节点不应被强制回滚，避免下游已执行节点产生不一致。

#### 状态机实现

```java
@Component
public class ActivityStateMachine {
    
    private static final Map<ActivityState, Map<StateEvent, ActivityState>> TRANSITION_TABLE;
    
    static {
        Map<ActivityState, Map<StateEvent, ActivityState>> table = new EnumMap<>(ActivityState.class);
        
        table.put(PENDING, Map.of(
            START, RUNNING,
            SKIP, SKIPPED,
            CANCEL, CANCELLED,
            TERMINATE, TERMINATED
        ));
        
        table.put(RUNNING, Map.of(
            COMPLETE, COMPLETED,
            FAIL, FAILED,
            TIMEOUT, FAILED,
            RETRY, RUNNING,
            CANCEL, CANCELLED,
            TERMINATE, TERMINATED
        ));
        
        table.put(FAILED, Map.of(
            RETRY, RUNNING,
            CANCEL, CANCELLED,
            TERMINATE, TERMINATED
        ));
        
        table.put(COMPLETED, Map.of(
            CANCEL, CANCELLED
        ));
        
        table.put(SKIPPED, Map.of(
            CANCEL, CANCELLED
        ));
        
        TRANSITION_TABLE = Collections.unmodifiableMap(table);
    }
    
    public StateTransitionResult fire(ActivityInstance instance, StateEvent event, Object payload) {
        ActivityState current = instance.getState();
        Map<StateEvent, ActivityState> transitions = TRANSITION_TABLE.get(current);
        
        if (transitions == null || !transitions.containsKey(event)) {
            throw new IllegalStateTransitionException(
                String.format("非法状态转移: %s → 事件[%s]", current, event)
            );
        }
        
        ActivityState next = transitions.get(event);
        
        // 持久化（同步，保证不丢）
        instance.setState(next);
        instance.setUpdateTime(LocalDateTime.now());
        activityRepository.updateState(instance);
        
        // 发布状态变更事件（驱动后续逻辑）
        return new StateTransitionResult(instance.getId(), current, next, event, payload);
    }
}
```

### 3.3 事件总线

状态机只负责"状态转移的原子性"，状态转移后产生的事件通过事件总线分发，解耦状态机与后续动作。

```java
public interface EventBus {
    void publish(DomainEvent event);
    void subscribe(String eventType, EventHandler handler);
}

// 内部实现（进程内）
@Component
public class InMemoryEventBus implements EventBus {
    private final Map<String, List<EventHandler>> handlers = new ConcurrentHashMap<>();
    private final Executor executor = Executors.newCachedThreadPool();
    
    @Override
    public void publish(DomainEvent event) {
        List<EventHandler> subs = handlers.getOrDefault(event.getType(), Collections.emptyList());
        for (EventHandler handler : subs) {
            executor.execute(() -> {
                try {
                    handler.handle(event);
                } catch (Exception e) {
                    // 事件消费异常不影响主流程，记录日志后续处理
                    log.error("事件消费异常", e);
                }
            });
        }
    }
}
```

**订阅方：**

```java
@Component
public class StateTransitionEventRouter {
    
    @Autowired
    private TokenScheduler scheduler;
    
    @Autowired
    private EscalationManager escalationManager;
    
    @Autowired
    private ExternalEventPublisher externalPublisher;
    
    @PostConstruct
    public void init() {
        eventBus.subscribe("ACTIVITY_STATE_CHANGED", event -> {
            StateTransitionResult result = (StateTransitionResult) event.getPayload();
            
            switch (result.getNewState()) {
                case RUNNING:
                    // 节点开始执行，注册超时监听
                    escalationManager.onActivityStarted(result.getActivityId());
                    break;
                case COMPLETED:
                case FAILED:
                case SKIPPED:
                    // 节点到达终态，驱动调度器处理下游
                    scheduler.onActivityFinished(result);
                    // 取消超时监听
                    escalationManager.onActivityFinished(result.getActivityId());
                    break;
                default:
                    break;
            }
            
            // 异步通知外部系统（如前端监控台）
            externalPublisher.publish(result);
        });
    }
}
```

## 四、调度与执行层

### 4.1 Token 机制详解

Token 是流程执行的"控制权令牌"，调度器通过 Token 的位置来决定"现在该执行哪个节点"。

#### Token 数据结构

```java
public class Token {
    private String tokenId;           // 全局唯一
    private String instanceId;        // 所属流程实例
    private String currentNodeId;     // Token 当前所在的节点
    private Token parent;             // Fork 时生成的父 Token（Join 时用于追溯）
    private TokenType type;           // NORMAL / SUB_PROCESS
    private boolean active;           // 是否有效（Join 后子 Token 被标记为失效）
    private LocalDateTime createdAt;
}
```

#### 启动流程：生成初始 Token

```java
public class ProcessInstanceStarter {
    
    public ProcessInstance start(String processKey, Map<String, Object> variables) {
        ProcessDefinition def = definitionCache.getLatest(processKey);
        
        // 1. 创建流程实例
        ProcessInstance instance = new ProcessInstance();
        instance.setInstanceId(generateId());
        instance.setProcessKey(processKey);
        instance.setVersion(def.getVersion());
        instance.setState(InstanceState.RUNNING);
        instance.setVariables(new ConcurrentHashMap<>(variables));
        instanceRepository.insert(instance);
        
        // 2. 找到 start 节点
        String startNodeId = def.getNodes().stream()
            .filter(n -> "start".equals(n.getType()))
            .findFirst()
            .map(NodeDefinition::getId)
            .orElseThrow();
        
        // 3. 创建初始 Token，放入调度队列
        Token initialToken = new Token(instance.getInstanceId(), startNodeId);
        tokenRepository.insert(initialToken);
        
        // 4. 触发 start 节点的执行
        scheduler.schedule(instance.getInstanceId(), startNodeId, initialToken);
        
        return instance;
    }
}
```

#### Fork：并行分支的 Token 分裂

```java
public class ParallelGatewayHandler {
    
    public void handleFork(ProcessInstance instance, Token parentToken, NodeDefinition forkNode) {
        // 1. 查询该节点的所有出边（即并行分支）
        List<EdgeDefinition> outgoingEdges = instance.getDefinition()
            .getAdjacencyList()
            .getOrDefault(forkNode.getId(), Collections.emptyList());
        
        // 2. 父 Token 标记为"等待中"（不删除，用于 Join 时比对）
        parentToken.setActive(false);
        parentToken.setState(TokenState.WAITING_JOIN);
        tokenRepository.update(parentToken);
        
        // 3. 为每个分支生成子 Token
        for (EdgeDefinition edge : outgoingEdges) {
            Token childToken = new Token(
                instance.getInstanceId(),
                edge.getTarget()       // Token 初始位置放在下游节点
            );
            childToken.setParent(parentToken);
            childToken.setActive(true);
            tokenRepository.insert(childToken);
            
            // 4. 每个子 Token 驱动对应分支的下游节点进入 PENDING → RUNNING
            scheduler.schedule(instance.getInstanceId(), edge.getTarget(), childToken);
        }
    }
}
```

#### Join：并行分支的 Token 聚合

```java
public class ParallelJoinHandler {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    public void handleJoin(ProcessInstance instance, Token completedToken, NodeDefinition joinNode) {
        // 1. 找到该分支对应的父 Token
        Token parentToken = completedToken.getParent();
        if (parentToken == null) {
            throw new IllegalStateException("Join 节点的 Token 必须有父 Token");
        }
        
        String joinCounterKey = "join:" + instance.getInstanceId() + ":" + parentToken.getTokenId();
        
        // 2. 原子递增到达计数（使用 Redis 或数据库乐观锁）
        Long arrivedCount = redisTemplate.opsForValue().increment(joinCounterKey);
        
        // 3. 获取期望到达的分支数（从 Join 节点配置读取）
        int expectedCount = joinNode.getProperties().getInt("expectedTokens");
        
        if (arrivedCount >= expectedCount) {
            // 4. 所有分支到达，清理计数器，重新激活父 Token
            redisTemplate.delete(joinCounterKey);
            
            // 5. 将所有子 Token 标记为失效
            List<Token> children = tokenRepository.findByParentId(parentToken.getTokenId());
            children.forEach(c -> c.setActive(false));
            tokenRepository.batchUpdate(children);
            
            // 6. 父 Token 移动到 Join 节点，继续驱动下游
            parentToken.setActive(true);
            parentToken.setCurrentNodeId(joinNode.getId());
            parentToken.setState(TokenState.NORMAL);
            tokenRepository.update(parentToken);
            
            scheduler.schedule(instance.getInstanceId(), joinNode.getId(), parentToken);
        }
        // 否则：计数不足，当前 Token 使命完成，不触发后续
    }
}
```

**Join 的并发安全问题：**

多个分支可能同时完成，同时触发 Join 逻辑。解决方案：
- **Redis 原子递增**：适合分布式部署，利用 `INCR` 的原子性
- **数据库乐观锁**：`UPDATE token SET arrived_count = arrived_count + 1, version = version + 1 WHERE ...`，利用 version 字段防并发
- **单 JVM 队列**：如果调度器是单实例，可用内存队列串行化 Join 事件

### 4.2 调度器实现

调度器是流程执行的"大脑"，职责是：**"收到节点完成事件后，决定下游哪些节点可以执行"**。

```java
@Component
public class TokenScheduler {
    
    @Autowired
    private ActivityStateMachine stateMachine;
    
    @Autowired
    private ActivityExecutor activityExecutor;
    
    @Autowired
    private ExpressionEvaluator expressionEvaluator;
    
    /**
     * 节点执行完成后的回调入口
     */
    public void onActivityFinished(StateTransitionResult result) {
        ProcessInstance instance = instanceRepository.load(result.getInstanceId());
        ProcessDefinition def = instance.getDefinition();
        NodeDefinition completedNode = def.getNode(result.getActivityId());
        
        // 1. 判断节点类型，决定后续动作
        if ("parallelGateway".equals(completedNode.getType()) 
            && "fork".equals(completedNode.getProperties().getString("mode"))) {
            // Fork 节点：分裂 Token，不走普通下游逻辑
            return; // Fork 的处理在 ForkGatewayBehavior 中完成
        }
        
        // 2. 查询该节点的所有出边
        List<EdgeDefinition> outgoingEdges = def.getAdjacencyList()
            .getOrDefault(completedNode.getId(), Collections.emptyList());
        
        // 3. 评估每条出边的条件表达式
        for (EdgeDefinition edge : outgoingEdges) {
            boolean shouldFire = evaluateCondition(edge, instance);
            
            if (shouldFire) {
                // 4. 为下游节点创建新的 Token（或复用当前 Token）
                Token newToken = new Token(instance.getInstanceId(), edge.getTarget());
                tokenRepository.insert(newToken);
                
                // 5. 驱动下游节点从 PENDING 转为 RUNNING
                schedule(instance.getInstanceId(), edge.getTarget(), newToken);
            }
        }
        
        // 6. 检查流程是否到达终态（所有 Token 都到达 end 节点或被终止）
        checkInstanceCompletion(instance);
    }
    
    /**
     * 将节点放入执行队列
     */
    public void schedule(String instanceId, String nodeId, Token token) {
        // 1. 创建 ActivityInstance（如果尚未创建）
        ActivityInstance activity = activityRepository.find(instanceId, nodeId);
        if (activity == null) {
            activity = new ActivityInstance(instanceId, nodeId);
            activity.setState(ActivityState.PENDING);
            activityRepository.insert(activity);
        }
        
        // 2. 状态机：PENDING → RUNNING
        StateTransitionResult transition = stateMachine.fire(activity, StateEvent.START, null);
        
        // 3. 获取该节点对应的 Behavior
        NodeDefinition nodeDef = token.getInstance().getDefinition().getNode(nodeId);
        ActivityBehavior behavior = nodeRegistry.getBehavior(nodeDef.getType());
        
        // 4. 构建执行上下文
        ExecutionContext ctx = buildContext(token, nodeDef);
        
        // 5. 提交执行器
        activityExecutor.submit(activity, behavior, ctx);
    }
    
    private boolean evaluateCondition(EdgeDefinition edge, ProcessInstance instance) {
        if (edge.getCondition() == null) {
            return true; // 无条件边默认通过
        }
        return expressionEvaluator.evaluate(edge.getCondition(), instance.getVariables());
    }
    
    private void checkInstanceCompletion(ProcessInstance instance) {
        List<Token> activeTokens = tokenRepository.findActiveByInstance(instance.getInstanceId());
        
        // 所有活跃的 Token 都在 end 节点上，则流程完成
        boolean allAtEnd = activeTokens.stream().allMatch(t -> 
            "end".equals(instance.getDefinition().getNode(t.getCurrentNodeId()).getType())
        );
        
        if (allAtEnd) {
            instance.setState(InstanceState.COMPLETED);
            instance.setEndTime(LocalDateTime.now());
            instanceRepository.update(instance);
        }
    }
}
```

### 4.3 执行器与线程模型

调度器决定"执行什么"，执行器负责"怎么执行"。

```java
@Component
public class ActivityExecutor {
    
    // 通用业务执行线程池
    private final ThreadPoolExecutor businessExecutor = new ThreadPoolExecutor(
        4, 32, 60, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(2000),
        new ThreadFactoryBuilder().setNameFormat("flow-biz-%d").build(),
        new ThreadPoolExecutor.CallerRunsPolicy()
    );
    
    // 长时间等待任务专用线程池（如等 MQ 回执）
    private final ExecutorService waitingExecutor = Executors.newCachedThreadPool(
        new ThreadFactoryBuilder().setNameFormat("flow-wait-%d").build()
    );
    
    /**
     * 提交执行任务
     */
    public void submit(ActivityInstance activity, ActivityBehavior behavior, ExecutionContext ctx) {
        // 判断节点类型，决定执行策略
        if (behavior instanceof WaitingBehavior) {
            // 异步等待型（如 receive 节点），不占用业务线程池
            waitingExecutor.execute(() -> executeWithInterceptors(activity, behavior, ctx));
        } else {
            // 计算型节点，提交到业务线程池
            businessExecutor.execute(() -> executeWithInterceptors(activity, behavior, ctx));
        }
    }
    
    private void executeWithInterceptors(ActivityInstance activity, ActivityBehavior behavior, ExecutionContext ctx) {
        try {
            // 1. 执行前拦截器
            interceptorChain.beforeExecute(ctx);
            
            // 2. 执行业务逻辑
            ExecuteResult result = behavior.execute(ctx);
            
            // 3. 执行后拦截器
            interceptorChain.afterExecute(ctx, result);
            
            // 4. 根据结果驱动状态机
            if (result.isWaiting()) {
                // 异步等待：保持 RUNNING，线程释放，等外部事件回调
                return;
            } else if (result.isSuccess()) {
                stateMachine.fire(activity, StateEvent.COMPLETE, result.getOutput());
            } else {
                stateMachine.fire(activity, StateEvent.FAIL, result.getError());
            }
            
        } catch (Exception e) {
            log.error("节点执行异常", e);
            stateMachine.fire(activity, StateEvent.FAIL, e);
        }
    }
}
```

**线程模型设计决策：**

| 场景 | 线程策略 | 原因 |
|------|---------|------|
| 普通 service 节点 | 业务线程池 | 快速执行，快速释放 |
| receive / deviceCmd 节点 | waiting 线程池 | 可能阻塞数分钟等回执，不能用有限线程池 |
| 定时 / 延迟任务 | 调度线程（Quartz/xxl-job） | 不占用执行线程 |
| 超时检测 | 独立后台线程 + Redis ZSET | 与业务执行完全隔离 |

### 5.2 持久化策略矩阵

| 数据 | 时机 | 方式 | 同步/异步 | 目的 |
|------|------|------|----------|------|
| 流程实例创建 | `start()` 调用时 | INSERT | **同步** | 实例存在性保证 |
| 活动状态变更 | `stateMachine.fire()` 内部 | UPDATE（乐观锁） | **同步** | 服务重启可恢复 |
| 活动输入/输出 | 节点执行完成时 | UPDATE | **同步** | 调试与审计 |
| 流程变量变更 | `setVariable()` 时 | UPDATE instance | **同步** | 下游节点依赖 |
| Token 状态 | Fork/Join/移动时 | INSERT/UPDATE | **同步** | 调度正确性 |
| 事件日志 | 每次状态转移 | INSERT | **异步批量** | 审计、对账、回放 |
| 心跳更新 | RUNNING 节点每 10s | UPDATE heartbeat_time | **异步** | 孤儿任务检测 |

### 5.3 服务重启恢复机制

```java
@Component
public class RecoveryManager {
    
    @Scheduled(fixedDelay = 60000) // 每分钟执行一次兜底扫描
    public void recover() {
        // 1. 加载所有非终态的流程实例
        List<String> unfinishedInstances = instanceRepository.findByStates(
            Arrays.asList(InstanceState.RUNNING, InstanceState.CREATED)
        );
        
        for (String instanceId : unfinishedInstances) {
            recoverInstance(instanceId);
        }
    }
    
    @PostConstruct // 启动时立即执行
    public void recoverOnStartup() {
        recover();
    }
    
    private void recoverInstance(String instanceId) {
        ProcessInstance instance = instanceRepository.load(instanceId);
        ProcessDefinition def = definitionCache.get(instance.getProcessKey(), instance.getVersion());
        instance.setDefinition(def);
        
        // 2. 加载该实例下所有非终态的活动
        List<ActivityInstance> unfinishedActivities = activityRepository.findByInstanceAndStates(
            instanceId, Arrays.asList(ActivityState.PENDING, ActivityState.RUNNING)
        );
        
        for (ActivityInstance activity : unfinishedActivities) {
            if (activity.getState() == ActivityState.PENDING) {
                // PENDING：重新放入调度队列，等待上游触发
                // 实际上 PENDING 不需要恢复，等上游节点完成时自然会触发
                log.info("节点 {} 处于 PENDING，等待上游触发", activity.getNodeId());
                
            } else if (activity.getState() == ActivityState.RUNNING) {
                // RUNNING：需要判断是"正在执行中"还是"执行线程已丢失"
                recoverRunningActivity(instance, activity);
            }
        }
        
        // 3. 恢复 Token 状态到内存调度器
        List<Token> activeTokens = tokenRepository.findActiveByInstance(instanceId);
        for (Token token : activeTokens) {
            scheduler.restoreToken(instance, token);
        }
    }
    
    private void recoverRunningActivity(ProcessInstance instance, ActivityInstance activity) {
        // 心跳检测：若最后更新时间超过阈值（如 60s），判定为孤儿任务
        Duration elapsed = Duration.between(activity.getHeartbeatTime(), LocalDateTime.now());
        
        if (elapsed.getSeconds() > 60) {
            log.warn("检测到孤儿任务: instance={}, node={}, 已等待 {}s", 
                instance.getInstanceId(), activity.getNodeId(), elapsed.getSeconds());
            
            // 策略 1：按超时处理
            stateMachine.fire(activity, StateEvent.TIMEOUT, 
                "服务重启后检测到孤儿任务，按超时处理");
            
            // 策略 2（可选）：若该节点支持幂等重试，可自动重试
            // stateMachine.fire(activity, StateEvent.RETRY, "自动重试");
        } else {
            // 心跳仍在更新，说明执行线程存活，无需干预
            log.info("节点 {} 正在正常运行中", activity.getNodeId());
        }
    }
}
```

## 六、应急领域特化

### 6.1 超时升级链（Escalation）

#### 设计

超时升级不是简单的"超时即失败"，而是**多级通知强度递增**。

#### 配置

```json
{
  "id": "wait_response",
  "type": "receive",
  "properties": {
    "timeout": 300,
    "escalation": [
      { "delay": 300, "action": "SMS", "template": "ESC_L1", "targets": ["${leaderPhone}"] },
      { "delay": 600, "action": "PHONE_CALL", "template": "ESC_L2", "targets": ["${leaderPhone}", "${backupPhone}"] },
      { "delay": 900, "action": "BROADCAST", "template": "ESC_L3", "targets": ["@all"] }
    ]
  }
}
```

#### 实现

```java
@Component
public class EscalationManager {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private NotificationService notificationService;
    
    private static final String ESCALATION_KEY_PREFIX = "escalation:";
    
    /**
     * 节点进入 RUNNING 时调用
     */
    public void onActivityStarted(ActivityInstance activity) {
        EscalationConfig config = parseEscalationConfig(activity);
        if (config == null || config.getLevels().isEmpty()) {
            return;
        }
        
        for (EscalationLevel level : config.getLevels()) {
            String taskKey = buildKey(activity, level);
            long executeAt = System.currentTimeMillis() + level.getDelay() * 1000;
            
            // 使用 Redis ZSET 作为分布式延迟队列
            redisTemplate.opsForZSet().add(
                "escalation:queue", 
                taskKey, 
                executeAt
            );
        }
        
        // 同时记录该节点的升级任务列表，用于完成时批量取消
        redisTemplate.opsForSet().add(
            ESCALATION_KEY_PREFIX + activity.getInstanceId() + ":" + activity.getNodeId(),
            config.getLevels().stream().map(this::buildKey).toArray(String[]::new)
        );
    }
    
    /**
     * 节点到达终态时调用，取消所有待执行的升级任务
     */
    public void onActivityFinished(ActivityInstance activity) {
        String setKey = ESCALATION_KEY_PREFIX + activity.getInstanceId() + ":" + activity.getNodeId();
        Set<String> taskKeys = redisTemplate.opsForSet().members(setKey);
        
        if (taskKeys != null && !taskKeys.isEmpty()) {
            // 从延迟队列中移除
            redisTemplate.opsForZSet().remove("escalation:queue", taskKeys.toArray());
            // 删除记录集合
            redisTemplate.delete(setKey);
        }
    }
    
    /**
     * 定时拉取到期任务（独立后台线程或 xxl-job）
     */
    @Scheduled(fixedRate = 5000)
    public void processDueEscalations() {
        long now = System.currentTimeMillis();
        Set<String> dueTasks = redisTemplate.opsForZSet()
            .rangeByScore("escalation:queue", 0, now);
        
        for (String taskKey : dueTasks) {
            EscalationTask task = parseTaskKey(taskKey);
            
            // 双重检查：节点是否仍在 RUNNING
            ActivityInstance activity = activityRepository.find(task.getInstanceId(), task.getNodeId());
            if (activity != null && activity.getState() == ActivityState.RUNNING) {
                notificationService.send(task.getAction(), task.getTemplate(), task.getTargets());
                
                // 如果是最后一级升级，触发超时失败
                if (task.isLastLevel()) {
                    stateMachine.fire(activity, StateEvent.TIMEOUT, 
                        "升级链全部触发仍未收到响应");
                }
            }
            
            // 从队列移除（无论是否执行，避免重复）
            redisTemplate.opsForZSet().remove("escalation:queue", taskKey);
        }
    }
}
```

**关键点：**

1. **取消机制必须可靠**：节点完成时，所有后续升级任务必须从延迟队列中移除，否则会造成误通知
2. **双重检查**：拉取到期任务后，再次查库确认节点仍在 RUNNING，防止"刚好完成但尚未取消"的竞态
3. **分布式**：Redis ZSET 保证集群多实例下只有一个实例处理同一任务（利用原子操作）

### 6.2 可靠指令投递（Reliable Command Delivery）

#### 时序与状态

```
[流程引擎] --1.写本地事务表(SENT)--> [MySQL]
         --2.发送MQ指令--> [RocketMQ]
         --3.MQ Confirm--> [流程引擎] 更新事务表为 DELIVERED
         
[设备网关] <--4.消费MQ-- [RocketMQ]
         --5.下发设备--> [终端设备]
         <--6.设备回执-- [终端设备]
         --7.发送回执MQ--> [RocketMQ]
         
[流程引擎] <--8.消费回执-- [RocketMQ]
```

```java
@Component
public class DeviceCommandBehavior implements ActivityBehavior {
    
    @Autowired
    private CommandTxRepository txRepository;
    
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    @Override
    public ExecuteResult execute(ExecutionContext ctx) {
        String commandId = generateCommandId(ctx);
        String deviceId = ctx.getVariable("deviceId");
        
        // 1. 幂等检查：该节点是否已发送过指令
        CommandTx existing = txRepository.findByCommandId(commandId);
        if (existing != null) {
            if (existing.getState() == TxState.ACKED) {
                return ExecuteResult.success(existing.getAckPayload());
            }
            // 否则继续等待或重试
            return ExecuteResult.waiting();
        }
        
        // 2. 写本地事务表（SENT 状态）
        CommandTx tx = new CommandTx();
        tx.setCommandId(commandId);
        tx.setInstanceId(ctx.getInstanceId());
        tx.setNodeId(ctx.getCurrentActivityId());
        tx.setDeviceId(deviceId);
        tx.setCommandPayload(ctx.getVariable("cmdPayload"));
        tx.setState(TxState.SENT);
        tx.setSendTime(LocalDateTime.now());
        txRepository.insert(tx);
        
        // 3. 发送 MQ（异步 Confirm）
        mqTemplate.asyncSend("device-command-topic", 
            new DeviceCommand(commandId, deviceId, tx.getCommandPayload()),
            new SendCallback() {
                @Override
                public void onSuccess(SendResult sendResult) {
                    txRepository.updateState(commandId, TxState.DELIVERED);
                }
                @Override
                public void onException(Throwable e) {
                    log.error("指令发送失败: {}", commandId, e);
                    // 定时任务会扫描 SENT 状态超时的记录进行重试
                }
            }
        );
        
        // 4. 返回 WAITING，释放执行线程
        return ExecuteResult.waiting();
    }
}

// MQ 回执监听器
@Component
@RocketMQMessageListener(topic = "device-response-topic", consumerGroup = "flow-engine")
public class DeviceResponseListener implements RocketMQListener<DeviceResponse> {
    
    @Autowired
    private CommandTxRepository txRepository;
    
    @Autowired
    private ActivityStateMachine stateMachine;
    
    @Override
    public void onMessage(DeviceResponse response) {
        CommandTx tx = txRepository.findByCommandId(response.getCommandId());
        if (tx == null || tx.getState() == TxState.ACKED) {
            return; // 幂等：已处理则丢弃
        }
        
        if (response.isSuccess()) {
            tx.setState(TxState.ACKED);
            tx.setAckTime(LocalDateTime.now());
            tx.setAckPayload(response.getPayload());
            txRepository.update(tx);
            
            // 驱动流程继续
            ActivityInstance activity = activityRepository.find(tx.getInstanceId(), tx.getNodeId());
            stateMachine.fire(activity, StateEvent.COMPLETE, response.getPayload());
        } else {
            // 设备执行失败，进入重试或失败分支
            handleDeviceFailure(tx, response);
        }
    }
}
```

**兜底重试（定时任务）：**

```java
@Scheduled(fixedRate = 30000)
public void retryTimeoutCommands() {
    // 查询 SENT 状态超过 60s 未确认的记录
    List<CommandTx> timeoutCommands = txRepository.findByStateAndSendTimeBefore(
        TxState.SENT, LocalDateTime.now().minusSeconds(60)
    );
    
    for (CommandTx tx : timeoutCommands) {
        if (tx.getRetryCount() >= tx.getMaxRetry()) {
            // 超过最大重试，标记失败，触发状态机 FAIL
            tx.setState(TxState.FAILED);
            txRepository.update(tx);
            
            ActivityInstance activity = activityRepository.find(tx.getInstanceId(), tx.getNodeId());
            stateMachine.fire(activity, StateEvent.FAIL, "设备指令超时未响应");
        } else {
            // 重试
            tx.setRetryCount(tx.getRetryCount() + 1);
            txRepository.update(tx);
            
            mqTemplate.syncSend("device-command-topic", 
                new DeviceCommand(tx.getCommandId(), tx.getDeviceId(), tx.getCommandPayload()));
        }
    }
}
```

### 6.3 动态拦截器链

```java
public interface ActivityInterceptor {
    int getOrder(); // 优先级，越小越先执行
    void beforeExecute(ExecutionContext ctx);
    void afterExecute(ExecutionContext ctx, ExecuteResult result);
}

@Component
public class InterceptorChain {
    
    private final List<ActivityInterceptor> interceptors;
    
    public InterceptorChain(List<ActivityInterceptor> interceptors) {
        this.interceptors = interceptors.stream()
            .sorted(Comparator.comparingInt(ActivityInterceptor::getOrder))
            .collect(Collectors.toList());
    }
    
    public void beforeExecute(ExecutionContext ctx) {
        for (ActivityInterceptor interceptor : interceptors) {
            interceptor.beforeExecute(ctx);
        }
    }
    
    public void afterExecute(ExecutionContext ctx, ExecuteResult result) {
        // 反向执行，类似 Servlet Filter 的 unwind
        for (int i = interceptors.size() - 1; i >= 0; i--) {
            interceptors.get(i).afterExecute(ctx, result);
        }
    }
}
```

## 九、面试速查

### Q1：为什么自研，不用 Flowable？

> 应急通信是"指令调度流"而非"人工审批流"，核心特征是超时升级链、GIS 联动、设备可靠投递。Flowable 的 BPMN 模型过重（历史表、变量表、事件日志表），且这些领域概念无法直接映射到 BPMN 规范。我们选择**轻量级 JSON DSL + 领域专用执行引擎**，用 1/10 的代码量实现了应急场景所需的全部语义，且性能更优。

### Q2：如何保证服务重启不丢状态？

> 三个机制：① **活动级快照**：每个节点状态变更同步写 MySQL（乐观锁保证并发安全）；② **Token 持久化**：Fork/Join/移动都写库，重启后恢复 Token 位置；③ **孤儿检测**：RUNNING 节点每 10s 更新心跳，重启后扫描超时的 RUNNING 节点，按超时或重试策略处理。

### Q3：并行分支的 Join 怎么保证不丢？

> 采用 **Redis 原子递增计数器**（或数据库乐观锁）。每个分支完成时对计数器做 `INCR`，达到 `expectedTokens` 时触发 Join。Redis 的 `INCR` 是单线程原子的，分布式环境下也能保证不会漏计。

### Q4：引擎支持多大并发？

> 调度器是**单线程无锁**的，只负责驱动 DAG 拓扑和状态流转；业务逻辑提交到线程池异步执行。单机支撑 **5000+ TPS** 的流程事件，瓶颈通常在业务侧的 HTTP 调用或 DB。水平扩展时按 `instance_id` 哈希分片，同一实例路由到同一节点，保证状态机串行化。

### Q5：升级链的定时任务如果挂了怎么办？

> ① 使用 **Redis ZSET** 做分布式延迟队列，多实例部署时只有一个实例能拿到到期任务（`ZREVRANGEBYSCORE + ZREM` 的原子性保证）；② 任务执行前**双重检查**节点是否仍在 RUNNING，防止误触发；③ 节点完成时**批量取消**后续升级任务，取消操作是幂等的。

### Q6：设备指令如果设备永远不回执怎么办？

> 三层兜底：① **MQ Confirm**：确保消息到达 Broker；② **本地事务表 + 定时重试**：`SENT` 状态超过 60s 未 `ACKED` 则自动重试，最多 3 次；③ **流程级超时**：重试耗尽后状态机触发 `FAIL`，流程进入预定义的降级分支（如人工介入节点）。

---

> **核心设计哲学**：编译时做尽所有校验，运行时保持极简，故障时快速恢复。所有复杂性（表达式解析、条件判断、并发控制）都推给编译期和基础设施，执行引擎本身只负责"收到事件 → 查表转移 → 驱动下游"。
