HasHMap的扩容涉及到几个关键点

2 什么时候发生扩容, 扩容时机(存放下一个元素的时候发现容量不够>容量*负载因子)

头插入法会导致数据反转（链表反转）,因此有可能会发生Cpu死循环问题

T1的新表（已迁移A）:  A → null
T1局部变量: e = B, next = C

T2完成全部迁移（头插法反转了顺序）:  C → B → A → null
T2设置 table = T2的新表

T1恢复执行，但它的e和next还是旧指针：
  1. 处理B: B.next 指向新表头A → 新表: B → A → null
  2. 处理C: C.next 指向新表头B → 新表: C → B → A → null
  
  问题出在下一步：
  3. T1以为C.next是null（因为初始时C是最后一个），
     但实际上T2迁移时已经改了C.next = B（头插法）
     
  4. T1执行 e = C.next = B，发现B不是null，继续循环
  5. 再次处理B: B.next 指向新表头C → 新表: B → C → B → ... 成环了！


如果数据量很大, 可以预估容量。 但如果数据量上千、或者这个临时Map在循环/高并发里被频繁创建，那就有必要给初始容量，避免多次扩容的rehash开销。

5. volatile能保证原子性吗？你项目中有没有用volatile做状态标志位？如果有一个场景：多个线程同时对一个volatile变量做i++操作，最终结果会正确吗？你平时是怎么判断一个变量该用volatile还是AtomicInteger的？

常见情况下状态值用volatile,计数使用AtomicInteger

不能保证原子性,解决的是可见性问题。

不会正确。volatile 只保证可见性和有序性，不保证原子性。而 i++ 在字节码层面是三步操作：读值 → 加1 → 写回。两个线程同时读到 0，各自加1后都写回1，结果就丢了。

实际测试：两个线程各执行 10000 次 volatile i++，最终结果大概率是 10000 到 20000 之间的随机数，永远不会是 20000。

如果需要原子递增，应该用 AtomicInteger 的 incrementAndGet()，底层是 CAS 无锁操作；如果并发极高，用 LongAdder 分段累加。

总结一下volatile的i++，看得见抢不过。

> "我在项目中基于 AQS 实现过一个**并发限流器**，用来控制接口的最大并发数。
>
> 核心思路是继承 AbstractQueuedSynchronizer，重写 `tryAcquireShared` 和 `tryReleaseShared`。用 AQS 的 `state` 表示剩余许可证数，初始值设为最大并发数比如 100。`tryAcquireShared` 里用 CAS 把 state 减 1，如果减完还大于等于 0 就获取成功；如果小于 0 就返回负数，AQS 会自动把线程放进 FIFO 队列阻塞。`tryReleaseShared` 里用 CAS 把 state 加 1，加成功后 AQS 会唤醒队列里的等待线程。
>
> 这个限流器比 Semaphore 更轻量，因为我们还加了超时获取的功能，防止线程无限阻塞。实际用在订单接口上，把并发控制在数据库连接池的最大连接数以内，避免连接池耗尽。"

更熟悉G1,线上的业务场景以G1为主。
G1是物理上的分区,分为连续多个Region。
Region又细分为正常大小的Region以及humongous Region，用于存放大对象。
大对象是直接进入老年代。

内存结构
1 Region：物理上不再连续区分 Eden/Survivor/Old/Humongous，逻辑上仍存在代际
2 Humongous Region：专门存放大对象（> 0.5 个 Region），直接进老年代，可能成为内存碎片和 GC 瓶颈
3 Card Table + Remembered Set（RSet）：
RSet：每个 Region 维护一个，记录哪些其他 Region 的对象引用了本 Region 的对象
作用：Young GC 时不用扫描整个 Old 区，只需扫描 RSet，实现跨代引用的快速定位
Card Table：辅助 RSet，记录内存页（512B）的引用变化

Young GC的时候回收Eden区, 回收年轻代的Region
Mixed GC: 当老年代达到阈值的时候,并发标记回收（年轻代+部分老年代）
Full Gc: 单线程回收, 老年代资源不够存放。由于是单线程下STW耗时可能会很长

Humongous Region不仅仅是一个大对象,而且 分配策略、回收时机、内存布局"完全不同的一类 Region。

## 四、Spring生态

13. Spring的Bean循环依赖是怎么解决的？如果A依赖B，B依赖C，C又依赖A，Spring能处理吗？如果是构造器注入的循环依赖，Spring怎么处理？你们项目中有没有因为循环依赖导致启动失败的案例？

Spring 通过三级缓存解决单例 Bean 的 Setter/Field 注入循环依赖。当一个 Bean 完成实例化（构造函数执行完、对象已在堆上创建）但还未完成属性注入和初始化时，Spring 会通过第三级缓存 singletonFactories 中的 ObjectFactory 提前暴露一个引用（必要时生成 AOP 代理），放入二级缓存 earlySingletonObjects。这样当循环依赖的另一方需要注入它时，能拿到这个半成品引用，避免死循环。但如果是构造器注入，因为对象还未实例化，无法提前暴露，Spring 是解决不了的，需要改用 @Lazy 或重构代码。


构造器注入的循环依赖,Spring怎么处理。
构造注入发生在Bean的生命周期的第一步,实例化。
A——>B B->A。则会报错

Setter注入,对象已经New,第一步阶段实例化完成


在重构代码的时候,容易遇到循环依赖的问题。

Spring官方推荐使用构造器注入的原因
我认为最核心的点,提前暴露设计缺陷。使用Setter注入,由于引入的半成品对象。即使有循环依赖的问题，问题也被隐藏了。

18. MySQL的RR隔离级别下，幻读到底解决了没有？如果有一个场景：事务A先查询id>100的记录（返回10条），事务B插入了一条id=105的记录并提交，事务A再次查询id>100，结果是什么？InnoDB是怎么解决这个问题的？间隙锁在实际业务中带来过死锁问题吗？

"InnoDB 的 REPEATABLE READ 没有完全解决幻读。它的 MVCC 机制可以保证纯快照读（普通 SELECT）不出现幻读，因为读取的是事务开始时的 ReadView 一致性快照。但一旦事务中出现了当前读（FOR UPDATE、UPDATE、DELETE），就会读取最新已提交数据，幻读就可能发生。
最典型的案例是：事务 A 先快照查询 id=105 不存在，事务 B 插入 id=105 并提交，事务 A 再用 UPDATE ... WHERE id=105 去更新，不仅能更新成功，后续再查询还能读到这条'凭空出现'的记录。如果业务对幻读零容忍，应该使用 SERIALIZABLE 隔离级别，或者所有查询都使用 FOR UPDATE 加 Next-Key Lock。"

如果当前查询的记录,命中了唯一索引,使用记录锁。

如果有一个场景：要求严格保证消息的顺序性，同时又要高吞吐，你会怎么设计？

1 要去听上去是很扯,但拆解问题可以分为两点
如果全局有序,并且还要高吞吐的时候,那么最终退化为单进程一个分区的List。
因此顺序性就必须是相对的,保证高吞吐,需要引入多分区的概念。

首先，生产者端用业务 Key（如订单号）做哈希路由，保证同一个 Key 的消息始终进入同一个 Partition。Partition 内部是追加写日志，天然有序。

其次，消费者端一个 Partition 只能被一个线程顺序处理，避免多线程乱序。如果消费逻辑有 IO 阻塞，我会在消费者内部用内存队列 + Key 哈希分发，做到线程间并行、线程内串行。

最后，为了提升吞吐，我会用批量发送减少网络 RTT，零拷贝降低 CPU 开销，消息压缩减少带宽。副本同步保证 ISR 内的顺序一致性。

这样设计的吞吐可以随分区数线性扩展，同时保证业务关心的 Key 级别有序。如果是金融撮合这种必须全局有序的场景，只能退化成单队列，靠硬件加速扛吞吐。"
