## 一、集合框架

1. 你项目中HashMap用得很多，假设现在有一个高频写入的场景，你评估后决定用HashMap（单线程场景）。上线后发现**单次写入偶尔有比较明显的延迟毛刺**，排查后发现是扩容导致的rehash。你能详细描述一下HashMap扩容的完整流程吗？为什么JDK 1.8之后改用尾插法？如果数据量很大，初始容量和负载因子你会怎么设置来减少扩容频率？




头插入法会导致数据反转（链表反转）,因此有可能会发生Cpu死循环问题

T1的新表（已迁移A）:  A → null
T1局部变量: e = B, next = C

T2完成全部迁移（头插法反转了顺序）:  C → B → A → null
T2设置 table = T2的新表

T1恢复执行，但它的e和next还是旧指针：
  1. 处理B: B.next 指向新表头A → 新表: B → A → null
  2. 处理C: C.next 指向新表头B → 新表: C → B → A → null
  
  问题出在下一步：
  3. T1以为C.next是null（因为初始时C是最后一个），
     但实际上T2迁移时已经改了C.next = B（头插法）
     
  4. T1执行 e = C.next = B，发现B不是null，继续循环
  5. 再次处理B: B.next 指向新表头C → 新表: B → C → B → ... 成环了！


如果数据量很大, 可以预估容量。 但如果数据量上千、或者这个临时Map在循环/高并发里被频繁创建，那就有必要给初始容量，避免多次扩容的rehash开销。


2. 有一个需求：需要维护一个全局的ID生成器缓存，要求线程安全且读多写少。你会选ConcurrentHashMap还是Collections.synchronizedMap？还是有更好的方案？如果ID数量达到千万级，内存占用成为瓶颈，你会怎么优化？

使用CurrentHashMap,Cas锁+Syncnorized关键字。多线程情况下,读写性能较高。
Collections.synchronizedMap。是 使用syncorized关键字上锁。


ConcurrentHashMap 写性能更高的核心原因不是"增加了CAS"，而是"锁粒度变细"。CAS 只是它在特定场景下避免加锁的手段。


如果ID数量达到千万级，内存占用成为瓶颈，你会怎么优化？
1 考虑这个存储的ID是什么,用途是什么,使用场景是否合理
2 如果ID需要判重,使用布隆过滤器判断,存储方式从Map修改Set
3 如果ID自增,考虑更换数据结构
3 如果存储的内容没有办法生成，并且必须保持在内存中。考虑使用堆外内存,减少GC压力



1. 你在项目里有没有遇到过HashMap在并发环境下导致死循环或数据丢失的问题？如果有，具体场景是什么？如果没有，你能构造一个复现场景吗？1.8之后死循环问题解决了，但数据丢失还存在吗？
