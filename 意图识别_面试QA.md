# 意图识别 — 面试深度 QA

> 面试题：意图识别有哪些主流技术方案？在不同业务场景下如何选型？
> 候选人背景：8年 Java 后端，智能运维 Agent（规则匹配 + LLM 推理）

---

## 一、标准答案框架

意图识别是 Agent 系统的第一道关卡——判断用户"想干什么"，决定后续走哪条处理链路。主流方案有四类，从简单到复杂：

### 方案一：基于规则 / 关键词匹配

用正则、关键词词典、模板匹配。适合意图集合固定、表达方式有限的场景。

- 优势：零延迟（< 1ms）、零成本、可解释、易调试
- 劣势：覆盖不了同义表达，维护成本随规则数量指数增长
- 适用：意图 < 20 个、表达方式固定的封闭域

### 方案二：传统分类模型（BERT / TextCNN）

把意图识别当文本分类任务，用标注数据训练分类器。BERT-base 在中文意图分类上 F1 通常能到 92-95%。

- 优势：泛化能力强、推理延迟可控（BERT-base 约 10-20ms）
- 劣势：需要标注数据、新增意图要重新训练、对 OOV 表达敏感
- 适用：意图 20-200 个、有标注数据、对延迟敏感的生产系统

### 方案三：LLM 直接判断

用大模型做 zero-shot 或 few-shot 意图分类。Prompt 里列出所有意图定义，让模型直接输出意图标签。

- 优势：零标注数据、新增意图只改 Prompt、天然支持多语言
- 劣势：延迟高（200-500ms）、成本高、对 Prompt 设计敏感
- 适用：意图频繁变化、冷启动阶段、需要理解复杂语义的场景

### 方案四：混合级联架构（生产系统首选）

实际生产系统通常用级联策略：

```
用户输入 → 规则层（高置信度直接命中）
         → 分类模型层（中等置信度）
         → LLM 兜底层（低置信度 / 模糊意图）
```

每一层设阈值，高置信度直接返回，低置信度下沉到下一层。80% 的请求在规则层和模型层就解决了，只有 20% 需要调 LLM，成本和延迟都可控。

### 选型决策框架

| 维度 | 规则 | 分类模型 | LLM | 混合级联 |
|------|------|---------|-----|---------|
| 意图规模 | < 20 | 20-200 | 不限 | 不限 |
| 标注数据 | 不需要 | 需要 | 不需要 | 少量 |
| 延迟 | < 1ms | 10-20ms | 200-500ms | 分层 |
| 新意图上线 | 改规则 | 重新训练 | 改 Prompt | 改配置 |
| 适用阶段 | MVP | 成长期 | 冷启动 | 生产 |

## 二、结合项目的真实回答（推荐用这个）

"我们的运维 Agent 处理的是**重要且紧急**的线上告警，场景非常固定——MQ 堆积、DB 慢查询、Redis 热点 Key、服务宕机这几类。用户输入也不是自然语言，而是结构化的告警事件。

所以意图识别我们**没有用 LLM，直接用规则匹配**——告警类型字段映射到对应的处理模板。这不是偷懒，是刻意的工程选择：

1. **延迟要求**：告警响应要快，规则匹配 < 1ms，LLM 推理要 200ms+，运维场景等不起
2. **准确性要求**：误判的代价很高——错判为'服务宕机'可能触发错误重启。规则匹配是确定性的，不会幻觉
3. **场景不需要泛化**：告警类型是有限枚举，不需要理解自然语言的多样性

LLM 用在**推理层**——匹配到告警类型后，用 LLM 做根因分析和处置建议，而不是做意图识别。**让 LLM 做它擅长的事（推理），而不是做规则能做好的事（分类）。**"

## 三、加分项（体现架构师思维深度）

1. **不只讲方案，讲选型决策框架。** "意图识别没有银弹，选型取决于三个维度：意图规模、标注数据量、延迟预算。"

2. **主动提级联架构。** 生产系统不会只用一种方案，而是分层降级。能讲出"规则层兜高置信度、模型层兜中等、LLM 兜底"的级联设计，说明做过真实的生产系统。

3. **提到意图识别的可观测性。** "意图识别的置信度分布是系统健康度的重要指标——如果低置信度请求占比突然升高，说明用户行为变了或者意图集合需要扩展。"

4. **结合自己项目举例（用真实数据）。** "我在运维 Agent 中告警场景是固定枚举，直接用规则匹配，高频告警类型覆盖了大部分请求。LLM 只负责推理层——根因分析和处置建议。人工介入率从 100% 降到了 15%，MTTR 从 45 分钟压缩到 8 分钟。"

5. **提到意图层次结构。** "实际系统中意图不是扁平的，而是树状的——一级意图分类（查询/操作/投诉），二级意图细化（查什么/操作什么）。分层识别比扁平分类更可维护。"

## 四、追问应对

### Q1：多意图消歧怎么处理？

"多意图分两种情况：

**可拆分型**——用户一句话包含两个独立意图，比如'查一下余额再转 500 块给张三'。处理方式是 LLM 做意图拆分，输出结构化结果，按优先级顺序串行执行，前一个完成后再执行后一个。

**不可拆分型**——用户表达模糊，同时匹配多个意图且置信度接近。触发澄清机制，列出 Top-2 意图让用户选择。

**关键设计点**：多意图拆分和执行必须共享同一个 session context，前一个意图的执行结果可能影响后一个意图的 slot 填充。"

> **结合项目**：运维 Agent 的告警场景不存在多意图问题——一条告警就是一个明确的事件类型。但可以补充："如果未来扩展到自然语言交互（比如用户说'帮我看看 MQ 有没有堆积，顺便重启一下服务'），就需要多意图拆分机制。"

### Q2：模糊意图的澄清策略是什么？

"三步走：

**第一步，置信度检测。** 意图识别模型输出置信度分数，低于阈值（比如 0.7）就触发澄清流程，不硬猜。

**第二步，渐进式引导。** 用漏斗式提问，每轮收窄范围，2-3 轮内收敛到明确意图。

**第三步，兜底退出。** 如果 3 轮还无法收敛，提供人工客服入口或让用户重新表述。

**工程化要点**：澄清轮次要设上限（最多 3 轮），每轮 prompt 都要携带已确认的上下文。澄清对话本身也要记录，用于后续优化意图识别模型。"

> **结合项目**：运维 Agent 用的是结构化告警，不存在模糊意图。但可以补充："我们的模糊性体现在另一个层面——告警收敛。同一个故障可能触发 100 条告警，我们的 Agent 需要把它们收敛成一个'故障事件'，这其实也是一种消歧。"

### Q3：意图识别的准确率怎么评估？你们的线上指标是多少？

"评估分三个层面：

**离线评估**——用标注测试集算 Precision / Recall / F1。关注整体 F1 和每个意图类别的 F1。

**在线指标**——更关键：
- 端到端任务完成率：用户最终是否完成了他想做的事
- 澄清触发率：触发澄清的比例越低越好，正常系统在 5-15%
- 人工转接率：兜底到人工的比例，反映系统能力边界

**关键认知**：意图识别的准确率不能只看离线数字。线上场景分布和离线测试集往往不一致。要用在线数据持续补充训练集，形成数据飞轮。"

> **结合项目**（用真实数据）："我们的场景更简单——告警类型是结构化字段，不存在识别准确率的问题。我们的评估指标是**端到端的**：人工介入率从 100% 降到 15%，MTTR 从 45 分钟压缩到 8 分钟。评估闭环是通过 Bad Case 彽集——每次人工介入的案例都会归档分析，看是推理层的问题还是预定义动作表覆盖不全。"

## 六、面试策略总结

| 场景 | 怎么说 |
|------|--------|
| 面试官问通用方案 | 讲四类方案 + 选型框架，展示知识广度 |
| 面试官问你的项目 | 讲规则匹配 + LLM 分层，强调是刻意的工程选择 |
| 面试官追问为什么不用 LLM 做意图识别 | 讲三个原因：延迟、准确性、场景不需要泛化 |
| 面试官问准确率 | 用真实指标：人工介入率 15%、MTTR 8min，不编造 F1 数字 |
| 面试官问未来扩展 | 讲如果扩展到自然语言交互，会引入混合级联架构 |

**核心原则：不要为了显得"高级"而过度设计。简单场景用简单方案，把 LLM 的能力留给真正需要推理的环节。这本身就是架构师最重要的判断力。**
