### 📌 阶段二：Agent系统架构（第2周）

> **目标**：Agent系统的每个核心模块都能讲清架构设计、选型依据和约束权衡。

| 模块 | 学习重点 | 检验标准 |

| **记忆管理** | 短期记忆（对话窗口/滑动窗口/摘要压缩）、长期记忆（向量检索）、外部记忆（知识库/数据库） | 能讲清一个Agent请求的全链路记忆交互 |

### 📌 阶段三：大模型应用与算法基础（第3周）

> **目标**：不需要懂算法细节，但要理解模型应用的核心范式和基础概念，能和算法工程师对话。

#### 3.1 Prompt Engineering & Context Engineering

| 技术 | 核心要点 | 面试表达 |
|------|---------|---------|
| Prompt Engineering | 角色设定、少样本示例（Few-shot）、思维链提示（CoT）、自一致性（Self-Consistency） | "Prompt不是魔法，是约束模型输出空间的结构化指令。核心是让模型知道：你是谁、任务是什么、输出格式是什么、边界条件是什么。" |
| Context Engineering | 上下文窗口管理、长文本压缩、RAG作为Context补充、多轮对话历史截断策略 | "Context Engineering比Prompt Engineering更重要，因为模型能看到的上下文是有限的。关键是：给模型什么信息、不给什么信息、怎么组织信息优先级。" |
| Function Calling | 工具描述Schema设计（OpenAI格式 vs 智谱格式）、工具选择策略、并行工具调用 | 能讲清Function Calling的完整链路：用户输入→模型判断→工具参数生成→后端执行→结果回传→模型生成回复 |

#### 3.2 模型微调基础（概念级，不要深入数学）

| 技术 | 核心理解 | 面试表达 |
|------|---------|---------|
| SFT（监督微调） | 用标注好的（指令，输出）对继续训练模型，让模型学会特定任务 | "SFT是Agent落地最常用的微调方式。不需要从头训练，只需要准备高质量的业务指令数据，对基座模型进行继续训练。" |
| RLHF（人类反馈强化学习） | 训练奖励模型→用PPO算法优化策略模型，让模型输出更符合人类偏好 | "RLHF是让模型'对齐'人类价值观的方法。需要三阶段：SFT→奖励模型训练→PPO优化。成本高，适合对安全性要求极高的场景。" |
| DPO（直接偏好优化） | 跳过奖励模型，直接用偏好数据优化模型，更简单高效 | "DPO是RLHF的简化版，不需要训练奖励模型，直接用人类偏好数据优化。效果接近RLHF但训练成本更低，是目前更实用的方案。" |

**面试策略**：
- 承认自己是工程背景，算法细节不深
- 但要讲清：**什么场景该微调、什么场景用Prompt就够了、微调的投入产出比怎么评估**
- 加分点："我有SFT的实践经验（或学习计划），准备了XX条业务指令数据，在XX模型上做了微调实验，效果是..."

#### 3.3 推理优化（高并发低延迟）

| 技术 | 核心要点 |
|------|---------|
| 推理框架 | vLLM（PagedAttention）、TensorRT-LLM、TGI、智谱自研推理引擎 |
| 批处理优化 | Dynamic Batching（动态批处理）、Continuous Batching（持续批处理，提高GPU利用率） |
| KV Cache | 缓存已计算的Key/Value，避免重复计算；内存优化策略 |
| 模型量化 | INT8/INT4量化、AWQ、GPTQ，降低显存占用和推理延迟 |
| 流式输出 | SSE（Server-Sent Events）实现流式响应，降低首字延迟（TTFT） |
| 多模型路由 | 根据任务复杂度选择不同模型（简单任务用轻量模型，复杂任务用大模型） |

**面试表达**：
"模型推理优化和传统服务优化有两个关键区别：
1. **计算密集**：GPU利用率是核心指标，Continuous Batching能显著提升吞吐
2. **内存瓶颈**：KV Cache占显存大头，需要通过量化、分页策略优化
3. **延迟敏感**：流式输出是必选项，用户感知延迟比总延迟更重要"

#### 3.4 Python生态认知（不要求实操，要求概念理解）

| 框架 | 核心概念 | 与Java的对应 |
|------|---------|------------|
| LangChain | Chains（链式调用）、Agents（推理循环）、Tools（工具）、Memory（记忆）、Retrievers（检索器） | Spring AI 对标 LangChain，提供类似的抽象层 |
| LangGraph | 状态机（StateGraph）、节点（Node）、边（Edge）、条件分支（Conditional Edge） | 类似工作流引擎，但更适配LLM的异步推理特性 |
| Spring AI | Java生态的LLM应用框架，支持智谱/ OpenAI / Ollama | 面试重点：Spring AI的ChatClient、ChatMemory、VectorStore抽象 |

**面试策略**：
- "我的主语言是Java，但我理解LangChain/LangGraph的设计思想。Spring AI是Java生态的对应方案，我在实际项目中使用Spring AI对接智谱API，实现了Function Calling和RAG集成。"
- 如果面试官追问Python细节，坦诚说明："Python我不是主力语言，但我能阅读和理解LangChain的源码逻辑。"

### 📌 阶段六：业务场景与面试冲刺（第5-6周）

#### 6.1 业务场景深度（结合JD中的5大场景）

准备每个场景的核心设计方案：

**场景1：智能自然语言对话取数（NL2SQL/NL2API）**
- 核心链路：自然语言→意图识别→Schema选择→SQL/API生成→执行→结果解读
- 难点：Schema理解（表结构太复杂模型看不懂）、SQL准确性校验、权限控制
- 你的TOB经验优势："我做过很多数据报表和开放API，对Schema设计和权限控制有深刻理解。"

**场景2：智能数据分析**
- 核心链路：用户问题→数据检索→分析思路生成→代码/工具执行→结论生成+可视化建议
- 难点：分析思路的合理性、代码执行安全（沙箱）、结果可信度

**场景3：智能营销/文案生成**
- 核心链路：用户画像→场景识别→Prompt模板选择→内容生成→效果反馈→模型迭代
- 难点：品牌一致性控制、多客户定制化、A/B测试
- 你的TOB经验优势："多租户定制化正是SaaS的核心能力，可以复用配置化模板体系。"

**场景4：Agent驱动的自动化运营**
- 核心链路：触发条件（定时/事件）→任务拆解→工具执行→结果校验→通知/回滚
- 难点：长周期任务的可靠性、异常处理、人工介入点设计

#### 6.2 项目叙事重构（STAR+技术选型论证）

**项目A：TOB SaaS平台（主项目，体现平台化+业务抽象能力）**

```

T（任务）：负责XX模块的架构设计与核心开发

   1. 多租户数据隔离与性能（一套代码服务不同规模客户）
```

## 八、给8年经验工程师的特别提醒

### 8.1 不要犯的错误

1. **不要炫耀Java基础**：8年经验还讲HashMap扩容，面试官会觉得你没深度。基础部分被问到才展开。
2. **不要回避Agent经验不足**：坦诚说明，但强调"工程化能力是Agent落地的瓶颈，我有这个能力，正在快速补足AI部分"。
3. **不要只讲做了什么**：高级岗位要讲的是"为什么这样设计"、"如果重来怎么优化"、"这个业务用Agent怎么做"。
4. **不要忽视业务思维**：这个JD特别强调"业务敏感度"和"跨团队协作"。你的TOB SaaS经验是巨大优势，要讲清楚。
5. **不要说Python不会**："能阅读和理解，但不是主力语言。Spring AI可以完成同样的Agent开发。"

### 8.2 必须展现的能力

1. **系统架构能力**：能从0到1设计Agent系统的每个核心模块。
2. **工程落地能力**：从架构设计到代码实现到上线运营的全链路经验。
3. **业务抽象能力**：把模糊的业务需求转化为明确的AI解决目标。
4. **持续学习能力**：主动关注Agent/LLM/MCP等新方向，有实际行动。
5. **跨团队协作能力**：能和算法、产品、运营团队有效沟通，推动项目落地。

### 8.3 薪资谈判建议

- **锚定**：8年经验 + 高级Agent工程师 + 算法工程一体化，参考市场合理报价。
- **策略**：先问清职级和薪资带宽。如果特别喜欢这个方向，可以适度降低短期现金要求，换取成长空间。
- **强调**："我更看重Agent方向的技术成长和业务影响力，薪资期望在合理范围内可以灵活沟通。"
