# 分布式事务 — 本地消息表方案

> **核心结论**：本地消息表是 BASE 理论的工程化实践，将分布式事务拆分为**本地事务** + **可靠消息投递**，通过异步重试保证最终一致性。

---

## 一、核心架构

```
┌─────────────┐         ┌──────────────┐         ┌─────────────┐
│  业务服务 A  │         │   消息表      │         │  业务服务 B  │
│  （订单服务）│◄───────►│  (MySQL)     │◄───────►│  （库存服务）│
└──────┬──────┘         └──────┬───────┘         └──────┬──────┘
       │                       │                        │
       │  ① 下单+写消息表（同一事务）                    │
       │                       │                        │
       │                       ▼                        │
       │                 ┌──────────┐                   │
       └────────────────►│ 定时任务  │                   │
                         │(1min/次) │◄──────────────────┘
                         └────┬─────┘   ④ 拉取/消费消息
                              │
                              ▼
                         ┌──────────┐
                         │  消息队列  │
                         │ (RocketMQ │
                         │ /Kafka)   │
                         └────┬─────┘
                              │
                              ▼
                         ③ 投递消息
                              │
                              ▼
                         ┌──────────┐
                         │  消费方   │
                         │  业务服务B │
                         └──────────┘
```

**核心思想**：
- 业务操作和消息记录必须在**同一个本地事务**中，利用数据库 ACID 保证原子性
- 定时任务扫描消息表，将待发送消息投递到 MQ
- 消费方幂等处理，保证消息至少被消费一次（at-least-once）

## 二、消息表设计

```sql
CREATE TABLE local_message (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    biz_id          VARCHAR(64) NOT NULL COMMENT '业务唯一标识，如订单号',
    biz_type        VARCHAR(32) NOT NULL COMMENT '业务类型：ORDER/PAYMENT',
    destination     VARCHAR(128) NOT NULL COMMENT '目标队列/topic',
    payload         TEXT NOT NULL COMMENT '消息体（JSON）',
    status          TINYINT NOT NULL DEFAULT 0 COMMENT '0:待发送 1:发送中 2:已发送 3:消费成功 4:消费失败',
    retry_count     INT NOT NULL DEFAULT 0 COMMENT '重试次数',
    max_retry       INT NOT NULL DEFAULT 5 COMMENT '最大重试次数',
    next_retry_time DATETIME COMMENT '下次重试时间（指数退避）',
    create_time     DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    
    UNIQUE KEY uk_biz (biz_type, biz_id)
) ENGINE=InnoDB COMMENT='本地消息表';
```

**字段说明**：

| 字段 | 说明 |
|------|------|
| `biz_id` + `biz_type` | 联合唯一键，防止重复记录，也是消费方幂等校验的依据 |
| `status` | 状态机驱动：待发送 → 发送中 → 已发送 → 消费成功 |
| `retry_count` / `max_retry` | 控制重试上限，防止无限重试打满资源 |
| `next_retry_time` | 支持指数退避，避免失败消息集中爆发重试 |

## 三、核心流程

### 3.1 生产者（业务服务 A）

```java
@Service
public class OrderService {
    
    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private LocalMessageMapper messageMapper;
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    @Transactional
    public void createOrder(OrderDTO order) {
        // 1. 保存订单（业务操作）
        orderMapper.insert(order);
        
        // 2. 写消息表（同一本地事务）
        LocalMessage msg = new LocalMessage();
        msg.setBizId(order.getOrderNo());
        msg.setBizType("ORDER");
        msg.setDestination("stock_deduct_topic");
        msg.setPayload(JSON.toJSONString(
            new StockDeductCmd(order.getOrderNo(), order.getSkuId(), order.getQty())
        ));
        msg.setStatus(0); // 待发送
        msg.setMaxRetry(5);
        msg.setNextRetryTime(LocalDateTime.now());
        
        messageMapper.insert(msg);
        
        // 3. 事务提交后，尝试即时发送（可选优化）
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronizationAdapter() {
                @Override
                public void afterCommit() {
                    asyncSend(msg.getId());
                }
            }
        );
    }
}
```

**关键**：`orderMapper.insert()` 和 `messageMapper.insert()` 必须在同一个 `@Transactional` 中。

### 3.2 定时任务（消息投递）

```java
@Component
public class MessageScanner {
    
    @Autowired
    private LocalMessageMapper messageMapper;
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    /**
     * 每分钟扫描一次
     */
    @Scheduled(fixedRate = 60_000)
    public void scanAndSend() {
        List<LocalMessage> msgs = messageMapper.selectByStatusAndNextRetryTime(
            Arrays.asList(0, 1, 4),  // 待发送 / 发送中 / 消费失败
            LocalDateTime.now(),
            100  // 每次批量拉取 100 条
        );
        
        for (LocalMessage msg : msgs) {
            try {
                // CAS 更新状态为"发送中"，防止多节点并发投递
                int updated = messageMapper.casUpdateStatus(
                    msg.getId(), msg.getStatus(), 1
                );
                if (updated == 0) continue;
                
                // 发送到 MQ
                rocketMQTemplate.syncSend(msg.getDestination(), msg.getPayload());
                
                // 更新为"已发送"
                messageMapper.updateStatus(msg.getId(), 2);
                
            } catch (Exception e) {
                // 更新重试计数和下次重试时间（指数退避）
                messageMapper.increaseRetry(
                    msg.getId(), 
                    calculateNextRetryTime(msg.getRetryCount())
                );
                log.error("消息投递失败, msgId={}", msg.getId(), e);
            }
        }
    }
    
    private LocalDateTime calculateNextRetryTime(int retryCount) {
        // 指数退避：1min, 2min, 4min, 8min, 16min
        long delayMinutes = 1L << retryCount;
        return LocalDateTime.now().plusMinutes(delayMinutes);
    }
}
```

**为什么用定时任务而不是即时发送？**
- 简单、可控、不依赖 MQ 事务特性
- 天然具备"兜底"能力：即使服务重启，消息也不会丢失
- 延迟通常是秒级~分钟级，对大多数异步场景可接受

### 3.3 消费者（业务服务 B）

```java
@Service
@RocketMQMessageListener(
    topic = "stock_deduct_topic", 
    consumerGroup = "stock_consumer"
)
public class StockConsumer implements RocketMQListener<String> {
    
    @Autowired
    private StockService stockService;
    @Autowired
    private DeductLogMapper deductLogMapper;
    
    @Override
    public void onMessage(String payload) {
        StockDeductCmd cmd = JSON.parseObject(payload, StockDeductCmd.class);
        
        // ========== 幂等校验（核心）==========
        if (deductLogMapper.exists(cmd.getOrderNo())) {
            log.warn("重复消费, orderNo={}", cmd.getOrderNo());
            return; // 直接返回成功，不抛异常
        }
        
        try {
            // 执行业务：扣库存
            stockService.deduct(cmd.getSkuId(), cmd.getQty());
            
            // 记录消费日志（幂等表）
            deductLogMapper.insert(
                new DeductLog(cmd.getOrderNo(), cmd.getSkuId(), cmd.getQty())
            );
            
        } catch (BusinessException e) {
            // 业务异常（如库存不足）：记录失败状态，不无限重试
            log.error("业务执行失败, orderNo={}", cmd.getOrderNo(), e);
            // 根据策略：转入死信队列或直接标记失败
            
        } catch (Exception e) {
            // 系统异常：抛出异常让 MQ 重试
            throw new RuntimeException("扣库存失败", e);
        }
    }
}
```

**幂等表设计**：
```sql
CREATE TABLE deduct_log (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no    VARCHAR(64) NOT NULL UNIQUE COMMENT '订单号',
    sku_id      BIGINT NOT NULL,
    qty         INT NOT NULL,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB;
```

## 四、异常场景处理（面试重点）

### 场景 1：本地事务提交后，服务 A 宕机

| 现象 | 消息状态 = 0（待发送） |
|------|----------------------|
| 结果 | 定时任务下次扫描到会继续投递 |
| 结论 | **消息不会丢失**，最多延迟一个扫描周期 |

---

### 场景 2：MQ 投递成功，但更新消息状态失败

| 现象 | 消息实际已到 MQ，但本地表状态还是"发送中" |
|------|------------------------------------------|
| 结果 | 定时任务会**重复投递** |
| 结论 | **消费方必须幂等**（靠 `biz_id` + 幂等表保证） |

---

### 场景 3：服务 B 消费成功，但 ACK 失败（或 B 宕机）

| 现象 | MQ 未收到 ACK，会重新投递 |
|------|--------------------------|
| 结果 | 消费方幂等表拦截重复消费 |
| 结论 | **at-least-once 语义，靠幂等实现 exactly-once 效果** |

---

### 场景 4：服务 B 业务执行失败（如库存不足）

```
消费失败
    │
    ├── 重试次数 < max_retry ──► MQ 重试
    │
    └── 重试次数 >= max_retry ──► 转入死信队列（DLQ）
                                    │
                                    ▼
                              人工介入 或 自动补偿（如取消订单）
```

**注意**：业务失败和系统失败要区分处理：
- **业务失败**（库存不足、参数错误）：不应无限重试，应转入补偿流程
- **系统失败**（DB 连接超时、网络抖动）：可以重试

---

### 场景 5：消息一直发送失败，超过最大重试次数

```sql
-- 达到 max_retry 的消息标记为"已死亡"
UPDATE local_message 
SET status = 5  -- 5: 已死亡/人工处理
WHERE retry_count >= max_retry AND status != 3;
```

- 触发告警通知运维/开发
- 人工核查后：修复问题重发，或回滚业务（如关闭订单、退款）

## 五、高级优化

### 5.1 即时发送 + 定时兜底（双保险）

```java
@Transactional
public void createOrder(OrderDTO order) {
    // ... 保存订单 + 消息表
    
    // 事务提交后，立即触发投递
    TransactionSynchronizationManager.registerSynchronization(
        new TransactionSynchronizationAdapter() {
            @Override
            public void afterCommit() {
                asyncSend(msg.getId());
            }
        }
    );
}
```

- **正常情况**：事务提交后毫秒级发出，无需等待定时任务
- **异常情况**（发送失败）：定时任务兜底

---

### 5.2 并发控制（多实例部署）

多节点同时扫描消息表时，需要用乐观锁防止并发投递：

```sql
UPDATE local_message 
SET status = 1, update_time = NOW()
WHERE id = ? AND status = 0;
```

或使用分布式锁（Redisson）保证单节点执行扫描。

---

### 5.3 消息顺序性保障

如果业务要求顺序消费（如订单状态变更：创建 → 支付 → 发货）：

- 用 `biz_id`（如订单号）做分片键，保证同一订单的消息发到 MQ 同一队列
- 消费方单线程消费该队列
- 或者：消息体里带 `version` / `event_time`，消费方按版本号处理

---

### 5.4 消息清理策略

消息表会无限增长，需要定期清理：

```sql
-- 只保留最近 7 天的已消费消息
DELETE FROM local_message 
WHERE status = 3 AND create_time < DATE_SUB(NOW(), INTERVAL 7 DAY)
LIMIT 1000;
```

- 已死亡消息（status = 5）单独归档，供人工核查
- 清理操作放在低峰期执行

## 六、与 MQ 事务消息对比

| 维度 | 本地消息表 | RocketMQ 事务消息 |
|------|-----------|------------------|
| **实现复杂度** | 高（需自建消息表 + 定时任务） | 低（MQ 自带事务回查） |
| **耦合度** | 与业务数据库耦合 | 与 MQ 耦合 |
| **延迟** | 定时任务轮询间隔（通常秒级~分钟级） | 更实时 |
| **数据库依赖** | 需要业务数据库 | 不需要 |
| **事务回查** | 自己实现（定时任务相当于回查） | MQ 自动回查 |
| **适用场景** | 已有稳定 DB，不想引入复杂 MQ；使用 Kafka | 已用 RocketMQ，追求低延迟 |

**本质关系**：RocketMQ 事务消息的思想与本地消息表**完全一致**，都是"先记录/半消息，再确认/回滚"。区别在于 RocketMQ 把定时扫描和回查逻辑内置到 Broker 了。

---

## 七、适用场景与局限

| 适用场景 | 不适用场景 |
|----------|-----------|
| 异步场景：下单后发券、扣库存、通知物流 | 同步强一致：转账、余额扣减 |
| 容忍秒级延迟的业务 | 要求实时一致性（用 TCC 或 Seata AT） |
| 高并发写，下游消费能力有限 | 两阶段提交（2PC）的性能敏感场景 |
| 使用 Kafka（不支持原生事务消息） | 简单同步调用即可满足的场景 |

## 八、面试口述版（2 分钟）

> "本地消息表是我实践过的一种分布式事务方案，核心是 **BASE 理论**，把分布式事务拆成本地事务 + 可靠消息投递。
>
> 流程上，业务操作和写消息表在同一个本地事务里，保证原子性。事务提交后，通过定时任务扫描消息表，把待发送的消息投递到 MQ。消费者拿到消息后，先查幂等表去重，再执行业务。
>
> 异常处理方面，服务宕机靠定时任务兜底，重复投递靠消费方幂等，消费失败走死信队列人工处理。这个方案的优点是实现简单、不依赖特定 MQ；缺点是轮询有延迟，需要维护消息表和幂等表。
>
> 和 RocketMQ 事务消息对比，思想是一样的，只是本地消息表把回查逻辑交给了自己的定时任务。我们当时选这个方案，是因为团队用的是 Kafka，不支持原生事务消息，而且业务对秒级延迟是可接受的。"

## 九、高频追问准备

| 追问 | 回答要点 |
|------|----------|
| **为什么不用 2PC/TCC？** | 2PC 性能差、阻塞资源；TCC 开发成本高，需要为每个操作写补偿逻辑。本地消息表更适合异步最终一致场景。 |
| **消息表数据量太大怎么办？** | 已消费消息定期清理（如保留 7 天）；已死亡消息归档；按时间分表。 |
| **怎么保证消息不丢失？** | 本地事务保证业务+消息原子写；定时任务兜底；MQ 持久化；消费方先幂等再执行业务。 |
| **如果消费方一直失败怎么办？** | 达到最大重试次数后转入死信队列，人工介入或自动补偿（如关单退款）。 |
| **本地消息表 vs Saga？** | Saga 是长事务的编排/协调模式，适合微服务链路长、需要逐步提交的场景。本地消息表更适合单向通知（如 A 通知 B 做某事）。 |
