# 模型微调：SFT / RLHF / DPO — 面试深度 QA

> 面试题：SFT、RLHF、DPO 三种模型微调方式，各自的原理、适用场景和投入产出比是什么？
> 候选人背景：8年 Java 后端，本地部署 Qwen 模型，结构化 Prompt 模板，**无正式微调经验**
> **定位：概念级理解 + 工程判断力，不深入算法数学。重点讲"什么场景该用什么"和"投入产出比怎么评估"。**

---

## 一、最佳答案

### 1.1 一句话定位

| 方式 | 一句话 |
|------|--------|
| **SFT** | 用标注好的"指令-回答"对，教模型学会做某件具体的事 |
| **RLHF** | 训练一个"打分器"（奖励模型），再用强化学习让模型学会拿高分 |
| **DPO** | 跳过打分器，直接用"好回答 vs 坏回答"的偏好数据训练模型 |

### 1.2 SFT（Supervised Fine-Tuning，监督微调）

**原理**：

准备一批高质量的 (instruction, output) 数据对，在基座模型上继续训练。模型学会"看到这种指令，就输出这种回答"。

```
训练数据:
  输入: "MQ 堆积告警，请分析根因"
  输出: "1. 检查消费端线程池是否打满\n2. 检查消息生产速率是否异常\n..."

模型学习: P(output | instruction, 基座权重) → 调整权重使输出匹配标注
```

**适用场景**：

- 模型需要学会特定的**输出格式**（JSON、表格、固定模板）
- 模型需要掌握**领域知识**（医疗、法律、运维的专业术语和推理路径）
- 模型需要学会**特定任务**（NL2SQL、代码生成、告警诊断）

**投入产出比**：

| 维度 | 评估 |
|------|------|
| 数据准备 | 中等——需要几百到几千条高质量标注数据 |
| 训练成本 | 低——LoRA 微调一张 A100 几小时 |
| 效果提升 | 显著——格式和领域知识提升明显 |
| 维护成本 | 中——每次业务变更可能需要补充数据重新训练 |
| 适用频率 | 高——最常用的微调方式 |

### 1.3 RLHF（Reinforcement Learning from Human Feedback，人类反馈强化学习）

**原理**：

三阶段训练：

```
阶段 1: SFT
  用标注数据先做一轮 SFT，让模型具备基本能力

阶段 2: 训练奖励模型 (Reward Model)
  让模型对同一个问题生成多个回答
  人工标注: 回答 A > 回答 B > 回答 C (偏好排序)
  训练一个打分器: 输入(问题, 回答) → 输出分数

阶段 3: PPO 强化学习
  模型生成回答 → 奖励模型打分 → PPO 算法根据分数调整模型权重
  目标: 让模型学会生成"拿高分"的回答
```

**适用场景**：

- 模型需要学会**人类偏好**（有用、无害、诚实——HHH 原则）
- 对话型应用需要**对齐人类价值观**（不能输出有害内容、不能编造信息）
- 对安全性要求极高的场景（金融、医疗、法律）

**投入产出比**：

| 维度 | 评估 |
|------|------|
| 数据准备 | 高——需要大量偏好标注数据（同一问题多个回答 + 人工排序） |
| 训练成本 | 高——三阶段训练，PPO 不稳定，需要大量 GPU 和调参经验 |
| 效果提升 | 高——模型的"对齐"质量最好 |
| 维护成本 | 高——奖励模型需要持续更新 |
| 适用频率 | 低——通常只有大模型厂商才做 |

### 1.4 DPO（Direct Preference Optimization，直接偏好优化）

**原理**：

跳过 RLHF 的"训练奖励模型 + PPO"两步，直接用偏好数据训练模型：

```
RLHF:  标注数据 → 训练奖励模型 → PPO 强化学习 → 对齐后的模型
DPO:   标注数据 → 直接优化模型权重 → 对齐后的模型

DPO 的核心洞察:
  奖励模型的最优解可以用数学公式直接表达
  不需要真的训练一个打分器，把偏好数据直接塞进 loss function 就行
```

训练数据格式：(问题, 好回答, 坏回答) 三元组

**适用场景**：

- 和 RLHF 相同的目标（对齐人类偏好），但资源有限
- 团队没有强化学习经验，想用更简单的方式做对齐
- 快速迭代——数据准备好就能训练，不需要调 PPO

**投入产出比**：

| 维度 | 评估 |
|------|------|
| 数据准备 | 中——和 RLHF 一样需要偏好数据，但不需要那么大量 |
| 训练成本 | 低——和 SFT 相当，不需要训练额外的奖励模型 |
| 效果提升 | 接近 RLHF——论文显示效果差距在 5% 以内 |
| 维护成本 | 低——和 SFT 相当 |
| 适用频率 | 中——目前更主流的对齐方案 |

### 1.5 三者对比总结

```
训练复杂度:    SFT ◀─────────────── DPO ───────────────▶ RLHF
               简单                   中等                   复杂

数据需求:      SFT ◀─────────────── DPO ───────────────▶ RLHF
               (指令, 回答)          (指令, 好, 坏)         (指令, 好, 坏) × 大量

效果上限:      SFT ◀─────────────── DPO ───────────────▶ RLHF
               学会做事              学会做好               学会做到最好
```

| 维度 | SFT | DPO | RLHF |
|------|-----|-----|------|
| 训练阶段 | 1 阶段 | 1 阶段 | 3 阶段 |
| 数据格式 | (指令, 回答) | (指令, 好回答, 坏回答) | (指令, 好回答, 坏回答) × 大量 |
| 是否需要奖励模型 | 否 | 否 | 是 |
| 是否需要 RL | 否 | 否 | 是 (PPO) |
| 训练难度 | 低 | 低-中 | 高 |
| GPU 需求 | 1 张 A100 | 1-2 张 A100 | 4-8 张 A100 |
| 典型训练时间 | 几小时 | 几小时 | 几天 |
| 效果 | 格式+知识 | 格式+知识+偏好 | 格式+知识+偏好（最好） |
| 适用角色 | 工程团队 | 工程团队 | 算法团队 |

## 二、工程化视角：什么场景该选什么

### 2.1 决策树

```
你需要微调吗？
  │
  ├─ 不需要 → 用 Prompt Engineering + RAG（成本最低）
  │    │
  │    └─ 效果不够？→ 考虑微调
  │
  └─ 需要 → 你的目标是什么？
       │
       ├─ 学会特定格式/领域知识 → SFT
       │    │
       │    └─ 效果不够？模型输出风格不好？→ 加 DPO
       │
       └─ 对齐人类偏好/安全性 → DPO（工程团队）或 RLHF（算法团队）
```

### 2.2 投入产出比评估框架

"评估微调的投入产出比，我会问四个问题：

1. **Prompt Engineering 能解决吗？** 如果 90% 的问题可以通过优化 Prompt 解决，微调的 ROI 太低。
2. **数据够吗？** SFT 至少需要 200-500 条高质量数据，DPO/RLHF 需要偏好数据，数据准备是最耗时的环节。
3. **效果提升能量化吗？** 微调前后的指标对比——准确率、格式合规率、人工评估分数。不能量化就不知道值不值。
4. **维护成本能接受吗？** 每次业务变更、模型升级，微调数据可能需要更新。如果业务变化快，微调的维护成本可能超过收益。"

## 三、结合项目的实际回答

"我的运维 Agent 目前**没有做微调**，用的是本地部署的 Qwen 模型 + 结构化 Prompt 模板。这是一个刻意的工程选择。

**为什么没微调**：

1. 告警诊断场景的输出格式是固定的（根因、影响、处置建议），Prompt 模板约束就够了
2. 领域知识通过 RAG 注入——安全文档检索回来塞进 context，不需要模型'记住'
3. 团队 6 个人，没有专职算法工程师，微调的维护成本承受不起

**但如果要微调，我的选型是 SFT**：

- 目标：让模型学会'运维工程师的诊断思维路径'——不只是给出答案，而是按步骤分析
- 数据来源：Bad Case 归集系统已经积累了 500+ 条人工标注的'告警→诊断结论'数据
- 训练方式：LoRA 微调 Qwen-14B，一张 A100 几小时搞定
- 效果预期：诊断格式合规率从 85% 提升到 95%+，减少 Prompt 中的格式约束指令

**什么情况下会加 DPO**：

如果 SFT 之后发现模型的'诊断风格'不好——比如太啰嗦、重点不突出、或者总是先说废话再给结论——就需要用 DPO 做偏好对齐。数据来源是人工评估：同一个告警，模型生成 3 个诊断结论，人工标注哪个最好、哪个最差，构成 (好, 坏) 偏好对。"

## 四、加分项

1. **区分"工程团队"和"算法团队"的微调策略。** "SFT 和 DPO 是工程团队能搞定的——数据准备 + 训练 + 评估，不需要强化学习经验。RLHF 需要算法团队——PPO 的训练不稳定，调参是门手艺。作为 Agent 架构师，我更关注 SFT 和 DPO。"——说明你有清晰的能力边界认知。

2. **提到 LoRA/QLoRA。** "全参微调成本太高——Qwen-14B 全参需要 8 张 A100。实际生产中用 LoRA（低秩适配）或 QLoRA（量化 + LoRA），一张 A100 就能微调 14B 模型，效果损失在 2-3% 以内。"——说明你了解工程化的训练方案。

3. **用 Bad Case 闭环讲微调数据来源。** "微调数据不需要从零开始标——Agent 系统的 Bad Case 归集机制天然就是数据来源。每次人工介入的案例，都是高质量的 (输入, 正确输出) 训练对。系统跑得越久，数据越多，模型越好——这就是数据飞轮。"

4. **提到微调 vs Prompt vs RAG 的选型框架。** "三者不是替代关系，是递进关系：先试 Prompt，效果不够加 RAG，还不够才微调。微调是最后手段，不是第一选择。"——说明你有工程化的决策思维。

5. **提到模型蒸馏。** "如果成本是第一优先级，还有一个思路：用 GPT-4 的输出蒸馏到本地小模型。先用 GPT-4 生成高质量的诊断结论，再用这些数据 SFT 本地 Qwen。效果接近 GPT-4，成本只有 1/10。"——结合了你 RAG 项目的降本经验。

## 五、追问应对

### Q1：什么场景用 Prompt Engineering 就够了，什么时候必须微调？

"用一个决策框架来判断：

**Prompt Engineering 够用的场景**：
- 输出格式可以通过模板约束（JSON、Markdown、固定结构）
- 领域知识可以通过 RAG 注入（文档检索、知识库）
- 任务描述可以通过 Few-shot 示例教会模型
- 评估指标已经达标（准确率、格式合规率、人工评分）

**必须微调的场景**：
- **格式约束成本太高**：Prompt 里 50% 的 Token 在描述输出格式，模型还是偶尔出错 → SFT 学格式
- **推理路径无法用 Prompt 表达**：你想让模型'像运维工程师一样思考'，但 Prompt 写不出这种隐性知识 → SFT 学推理
- **延迟要求苛刻**：Prompt 太长导致推理慢，微调后可以用更短的 Prompt 达到同样效果 → SFT 省 Token
- **安全对齐**：模型输出不符合人类偏好（太啰嗦、重点不突出、偶尔有害）→ DPO 做偏好对齐

**量化判断**：如果 Prompt 优化 3 轮后效果提升 < 5%，就该考虑微调了。"

> **结合项目**："我的运维 Agent 目前 Prompt 优化了 3 轮，诊断格式合规率 85%，还有 15% 的输出不符合预期。如果再优化 Prompt 收益不大，下一步就是用 Bad Case 数据做 SFT。"

### Q2：SFT 的训练数据怎么准备？质量怎么保证？

"数据准备是 SFT 最关键的环节——garbage in, garbage out。

**数据来源（三个渠道）**：

1. **人工标注**：领域专家直接写 (指令, 回答) 对。质量最高，但成本也最高。适合冷启动。
2. **Bad Case 归集**：Agent 系统运行中的人工介入案例，天然就是高质量训练数据。成本几乎为零。
3. **模型生成 + 人工审核**：用 GPT-4 生成候选回答，人工审核筛选。适合快速扩充数据量。

**质量保证（四道关卡）**：

```
① 格式校验: 回答是否符合预期格式（JSON 合法、字段齐全）
② 一致性检查: 同一问题的多个标注是否一致（Kappa 系数 > 0.8）
③ 专家审核: 领域专家抽样审核（至少 10%）
④ 效果验证: 用留出的测试集评估微调后的效果
```

**数据量参考**：
- 格式学习：200-500 条
- 领域知识：500-2000 条
- 复杂推理：2000-5000 条

**关键细节**：数据的**多样性**比**数量**更重要。1000 条全是 MQ 堆积的训练数据，不如 500 条覆盖 10 种告警类型的数据。"

### Q3：DPO 相比 RLHF 省掉了什么？效果差距大吗？

"省掉了两样东西：

**① 省掉了奖励模型的训练。** RLHF 需要先训练一个打分器——输入 (问题, 回答)，输出一个分数。这个打分器本身需要大量偏好数据训练，而且训练不稳定。DPO 直接把偏好数据塞进 loss function，数学上证明了最优解等价于训练奖励模型再做 PPO。

**② 省掉了 PPO 强化学习。** PPO 是 RLHF 中最不稳定的环节——需要同时维护 4 个模型（策略模型、参考模型、奖励模型、价值模型），超参数敏感，训练经常崩。DPO 只需要 2 个模型（训练模型 + 参考模型），和 SFT 的训练复杂度相当。

**效果差距**：

论文数据显示，DPO 的效果**接近 RLHF**，差距在 5% 以内。在某些任务上甚至更好，因为 PPO 的训练不稳定性可能导致 RLHF 训练出次优解。

**实际选型**：

| 场景 | 选什么 | 原因 |
|------|--------|------|
| 大模型厂商（OpenAI、Anthropic） | RLHF | 有算法团队、有 GPU 资源、追求极致效果 |
| 企业级应用团队 | DPO | 工程团队能搞定、训练稳定、效果够用 |
| 快速验证 | SFT | 先做 SFT 看效果，不够再加 DPO |

**一句话总结**：DPO 是 RLHF 的'工程友好版'——效果接近，但训练难度降了一个数量级。"

## 六、架构图

### 三种微调方式的训练流程对比

```
=== SFT: 一步到位 ===

  高质量标注数据 (指令, 回答)
          │
          ▼
  ┌──────────────┐
  │  基座模型     │
  │  (Qwen-14B)  │──▶ SFT 训练 ──▶ 微调后模型
  └──────────────┘
       1 阶段      LoRA: 1 张 A100, 几小时


=== RLHF: 三阶段 ===

  阶段 1: SFT
  标注数据 ──▶ 基座模型 ──▶ SFT 模型

  阶段 2: 训练奖励模型
  SFT 模型生成多个回答 ──▶ 人工排序 ──▶ 奖励模型

  阶段 3: PPO 强化学习
  SFT 模型 ──▶ 生成回答 ──▶ 奖励模型打分 ──▶ PPO 更新权重
       │                                          │
       └──────────── 循环迭代 ◀───────────────────┘

  总计: 3 阶段, 4-8 张 A100, 几天


=== DPO: 简化版 ===

  偏好数据 (指令, 好回答, 坏回答)
          │
          ▼
  ┌──────────────┐     ┌──────────────┐
  │  训练模型     │     │  参考模型     │
  │  (可训练)     │     │  (冻结)       │
  └──────┬───────┘     └──────┬───────┘
         │                    │
         └────────┬───────────┘
                  │
                  ▼
           DPO Loss 计算
           (好回答概率 ↑, 坏回答概率 ↓)
                  │
                  ▼
           对齐后模型

  总计: 1 阶段, 1-2 张 A100, 几小时
```

## 七、面试策略总结

| 场景 | 怎么说 |

| 面试官问原理 | 讲三者的训练流程，重点讲 DPO 为什么比 RLHF 简单 |
