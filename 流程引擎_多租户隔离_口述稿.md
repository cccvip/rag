# 流程引擎多租户资源隔离 —— 面试口述稿

> 适用场景：面试中回答"流程引擎怎么做多租户隔离？""如何保证坏邻居不影响其他客户？""共享服务和独立部署怎么选？"
> 建议时长：3~5 分钟

---

## 一、开场：问题背景

我们的流程引擎是以 SaaS 形态对外提供服务的，多个客户共用同一套引擎集群。

一开始项目少、客户少，其实没太在意隔离。但随着接入的业务方越来越多，出现过一个真实问题：某个客户的流程配置出了 Bug，短时间内循环创建了上万个流程实例，直接把引擎的回调线程池和 MySQL 连接池占满了，导致其他客户的流程也跑不动了。

那次之后，我就牵头做了一版**多租户资源隔离方案**，核心目标是：**在不做独立部署的前提下，让单个客户的异常不会扩散成全服故障。**

---

## 二、核心思路：分层隔离

我没有选择给每个客户单独部署一套服务，因为那样版本碎片化、运维成本和资源浪费都太大。

我的思路是：**同一套服务，但在入口处做准入、在调度层做公平、在执行层做配额、在持久化层做限流、在内存层做配额，最后再加一层熔断兜底。**

可以简单记成五层隔离 + 一个开关。

---

## 三、逐层展开

### 第一层：入口限流

每个租户在 API 网关层有独立的令牌桶。

- 普通客户默认每秒 100 次创建请求
- VIP 客户可以配到 1000
- 超限时直接返回 429，不进引擎

这样即使某个客户出了死循环或者遭到异常流量，请求连门都进不来。

### 第二层：调度层公平

我们的引擎调度核心是类 Netty 的单线程 EventLoop。

这里有个容易误解的点：**这个 EventLoop 不是 `while(true)` 空转吃 CPU，而是基于阻塞队列的事件循环**。调度线程调用 `queue.take()` 挂起等待，有事件（用户创建流程、业务回调、超时触发）时才被唤醒，没事件时 CPU 占用接近零。

事件来源主要有三个：
- 用户调用 `start()` 创建流程实例
- 业务服务执行完节点后回调引擎
- 超时检测任务触发节点升级或重试

原本如果用一个大队列，某个租户堆积几万实例，其他租户都得跟着等。我改成**每个租户一个独立子队列，调度器轮询消费**，保证每人每次都有调度机会，不会被单个租户垄断。

每个队列还设了深度上限，超过的实例进延迟队列，避免内存被撑爆。

### 第三层：执行层隔离

引擎向业务服务发起 Dubbo 回调时，按租户限制最大并发数。

- 单个租户最多同时占 50 个线程
- 超过直接拒绝，走降级

同时每个租户、每种 Behavior 类型有独立的熔断器。比如租户 A 的短信通知下游挂了，只熔断 A 的短信节点，不影响 A 的其他节点，更不影响其他租户。

### 第四层：持久化限流

MySQL 连接池是全局资源。我给每个租户限制了每秒最大写操作数，比如 500，超过就排队或批量合并。

另外事件日志这种非关键路径的数据，改成异步批量刷盘，减少 DB 压力。

### 第五层：内存配额

流程实例是长生命周期的，可能挂起好几天。我给每个租户设了活跃实例上限，比如 1 万个，超过就不让新建。

同时 MANUAL 节点挂起超过 7 天没人处理，会自动归档，释放内存。

### 兜底：租户级熔断开关

最后还有一道保险：当某个租户错误率 1 分钟内超过 90%，或者创建速率超过基线 10 倍时，系统会自动熔断这个租户；运维也可以在后台一键熔断。

熔断后，该租户的新流程创建会被拒绝，但已经运行的实例不受影响，继续跑到结束。

---

## 四、监控

所有这些指标都带 tenant_id 标签，Prometheus 上可以直接看：

- 各租户调度队列深度
- 线程池并发占用
- DB 操作速率
- 活跃实例数
- 错误率

告警阈值一般设在配额的 80%，提前介入。

---

## 五、效果与收益

这套方案上线后，再也没出现过单客户拖垮全服的情况。

- 成本上：不用给每个客户买服务器，资源共享
- 运维上：一套代码、一套镜像、统一升级
- 稳定性上：单个租户异常被限制在本租户范围内

## 六、常见追问应答

### Q1：为什么不直接给每个客户独立部署？

独立部署隔离性最强，但代价也大：
- 版本容易碎片化，修一个 Bug 要发 N 个包
- 运维监控工作量成倍增长
- 很多客户夜间流量很低，资源大量闲置

我的原则是：**先用共享服务 + 分层隔离解决 90% 的问题，只有合规要求数据不出机房、或者客户流量是其他客户百倍以上时，才考虑独立部署。**

### Q2：租户级熔断会不会误伤？

熔断触发有两个条件：错误率 > 90% 或者创建速率超基线 10 倍，阈值比较严格。而且熔断只拒绝新创建请求，已经运行的流程不受影响。同时支持人工一键恢复。

### Q3：如果某个租户就是大客户，流量真的很高怎么办？

可以给他独立资源配额，比如 QPS 提到 5000、并发提到 200、活跃实例提到 5 万。但代码和部署还是共享的，只是在 K8s 层面给他独占一部分 Pod 或节点，实现"共享代码 + 独立资源"。

### Q4：单线程 EventLoop 会不会成为瓶颈？

调度器只做内存 DAG 遍历和状态机推进，真正的耗时都在下游业务调用。我们是按 instance_id 哈希分片的，每个分片一个 EventLoop，水平扩展时分片数增加，调度能力跟着线性扩展。

### Q5：调度器线程是不是一直在后台死循环？会不会把 CPU 跑满？

不是空转。调度线程内部用 `BlockingQueue.take()` 阻塞等待事件，没事件时线程挂起，CPU 占用接近零。这个模型和 Netty 的 EventLoop 是一样的。循环只是为了持续消费事件，而不是为了空跑。

## 七、方案可行性与边界

### 7.1 这套方案可行性如何？

**这套方案在 SaaS 多租户领域是非常成熟的做法，不是过度设计。**

核心思想都是：**共享物理资源，但在逻辑上给每个租户划定边界。**

具体可参考这些业界实践：

| 我们的设计 | 业界对应实践 | 参考链接 |
|-----------|------------|---------|
| 共享服务 + 分层隔离，不做独立部署 | AWS Lambda Tenant Isolation 模式 | https://aws.amazon.com/blogs/aws/streamlined-multi-tenant-application-development-with-tenant-isolation-mode-in-aws-lambda |
| 单租户并发数限制（Semaphore） | Resilience4j Bulkhead 信号量隔离 | https://resilience4j.readme.io/docs/bulkhead |
| 按租户/资源维度限流 | Sentinel 并发线程数限流 | https://github.com/alibaba/Sentinel/wiki/Flow-Control |
| 租户级资源配额 | Kubernetes ResourceQuota | https://kubernetes.io/docs/tasks/administer-cluster/manage-resources/quota-memory-cpu-namespace/ |
| Serverless 函数级并发限制 | 阿里云 FC 并发实例数上限 | https://help.aliyun.com/zh/functioncompute/fc/user-guide/overview-of-configuring-the-maximum-number-of-on-demand-instances |
| 单线程 EventLoop 调度 | Netty / Reactor Netty 事件循环模型 | https://projectreactor.io/docs/netty/1.1.17/reference |

### 7.2 最可能被质疑的 6 个点及应答

#### 质疑 1：为什么不直接给每个租户独立部署？隔离性不是更强吗？

**应答**：独立部署隔离性最强，但代价是版本碎片化、运维成本线性增长、资源利用率低。我们的原则是**分层隔离解决 90% 的问题，独立部署只留给合规强制或流量百倍以上差异的极端场景**。SaaS 行业的标准做法是共享 + 隔离，而不是一客一部署。

#### 质疑 2：Redis 限流有网络开销，性能会不会成为瓶颈？

**应答**：Redis 是内存数据库，一次限流判断 RT 通常在 1ms 以内。相对于流程引擎内部后续的 Dubbo 调用、MySQL 写入，这个开销占比很低。如果追求极致性能，可以在本地做一层短周期缓存 + Redis 异步同步，但项目初期没必要。

#### 质疑 3：单线程 EventLoop + 租户队列，会不会太复杂？

**应答**：单线程调度是为了保证状态机串行化，避免分布式锁冲突，这是流程引擎本身的特性决定的。租户队列只是在原有队列基础上按租户分桶，复杂度可控，代码量不大。收益是彻底消除了单租户阻塞全服调度的风险。

#### 质疑 4：Semaphore 隔离不够彻底，A 还是可能影响 B 吧？

**应答**：理论上如果全局线程池满，所有租户都受影响。但有两重保护：
1. 单租户 Semaphore 上限设计为远小于线程池总大小
2. 线程池容量按 "租户数 × 单租户上限 × 峰值系数" 预留 buffer

如果业务要求**绝对隔离**，那确实需要独立部署或 K8s 级别的 CPU/内存隔离。但我们评估后认为，流程引擎属于"共享服务可接受偶尔排队"的场景，这套方案性价比最高。

#### 质疑 5：参数这么多，运维调优成本是不是很高？

**应答**：可以**渐进式落地**：
1. 第一期只做入口 Redis 限流 + 线程池隔离（两周）
2. 第二期加调度层租户队列（一周）
3. 第三期加 DB 限流 + 熔断（一周）

所有参数都走配置中心，支持热更新。上线后通过 Prometheus 监控各租户指标，根据实际数据调整，不需要拍脑袋。

#### 质疑 6：为什么不用 Sentinel / Hystrix 现成的框架？

**应答**：我们内部确实复用了 Resilience4j 的熔断能力。但流程引擎有两点是通用框架覆盖不到的：
1. **按 instance_id 哈希分片**导致的本地调度隔离需求
2. **流程实例长生命周期**导致的内存配额管理需求

所以通用框架解决 60%，剩下 40% 需要结合引擎特性自研。

#### 质疑 7：既然单租户并发限制可以用 Resilience4j Bulkhead，为什么不直接用？

**应答**：Resilience4j Bulkhead 擅长的是"按下游依赖隔离并发"，比如限制对 paymentService 的并发调用数。但我们的场景是"按租户动态隔离"，有三点它覆盖不了：

1. **租户是动态增长的**：Resilience4j 需要预先配置每个 bulkhead instance（如 `paymentService`、`inventoryService`），不可能为每个租户都写 YAML。而自研 Semaphore 可以运行时 `computeIfAbsent` 动态创建。

2. **租户需要分级配额**：大客户 200、普通客户 20、试用客户 5，Resilience4j 虽然能配，但管理成本高。自研方案可以从配置中心按租户等级动态读取。

3. **全局线程池共享**：Resilience4j 的 ThreadPoolBulkhead 是为每个资源单独分配线程池，如果每个租户一个线程池，资源浪费严重。我们的方案是共享一个全局线程池，用 Semaphore 控制每个租户最多同时占多少线程。

所以我们的技术选型是：
- **Behavior 级熔断** → Resilience4j CircuitBreaker（直接复用）
- **租户级 Semaphore 并发隔离** → 自研（动态租户 + 分级配额）
- **入口限流** → Redis + Lua 自研（跨机器共享状态）
- **调度层租户队列** → 自研（流程引擎特有）

#### 质疑 8：那用 Sentinel 做并发线程数限流行不行？

**应答**：Sentinel 的并发线程数限流本质上也是 Semaphore，但它有三个问题让我们没有选择它：

1. **资源维度是方法/接口，不是租户**：Sentinel 要对每个租户每个方法配规则，租户动态增长时规则爆炸。我们的自研 Semaphore 是运行时按 `tenantId` 动态创建的。

2. **需要额外基础设施**：生产环境用 Sentinel 通常要部署控制台 + 规则中心 + 集群限流 Token Server。我们只有 2 台引擎服务，引入这些组件架构太重。自研方案只依赖现有 Redis。

3. **控制粒度不匹配**：Sentinel 控制的是"单个方法的并发数"，我们要的是"租户在所有行为上的总并发数"。如果每个方法都配 Sentinel，一个租户可能在多个方法上各占满配额，总并发反而失控。

但 Sentinel 在入口 QPS 限流、热点参数限流、熔断降级上是可以用的。如果团队已经熟悉 Sentinel，可以把它用在入口层，租户级并发隔离仍然建议自研。

### 7.3 这套方案的边界

| 场景 | 是否适合共享 + 分层隔离 | 建议方案 |
|------|----------------------|---------|
| 租户数量 < 20 | ✅ 非常适合 | 共享 + 分层隔离 |
| 租户数量 > 100 | ⚠️ 需简化 | 优先网关限流，弱化本地隔离 |
| 租户间有强合规隔离要求 | ❌ 不适合 | 独立部署或独立数据库 |
| 大客户流量是其他客户 100 倍以上 | ⚠️ 部分适合 | 大客户走独立 Pod/节点 |
| 初创期流量小 | ❌ 过度设计 | 只做全局限流即可 |

### 7.4 如果面试官说"你还是独立部署更稳"

**标准应答**：
> 独立部署确实更稳，但稳定性不是唯一目标，成本、运维复杂度、版本一致性也要考虑。我们的策略是**共享服务 + 分层隔离作为默认方案，独立部署作为可选项**。当客户提出合规要求、或者流量规模达到共享方案边界时，我们会把他切到独立部署。这样既能服务大多数客户，又能满足特殊客户。

---

## 八、一句话总结

> 多租户隔离的核心不是复制服务，而是在共享服务内部建立租户边界：**入口限流防冲击、调度公平防阻塞、执行隔离防占满、持久化限流防拖垮、内存配额防爆仓、熔断兜底保全局。**
