# Function Calling — 面试深度 QA

> 面试题：Function Calling 的完整链路是什么？从用户输入到最终回复，每一步发生了什么？工具 Schema 设计有哪些最佳实践？
> 候选人背景：8年 Java 后端，智能运维 Agent（Function Calling 调用 ELK / Prometheus / 告警收敛）

---

## 一、标准答案：Function Calling 完整链路

Function Calling 不是"模型调代码"这么简单——它是一个涉及模型侧、网关侧、业务侧三方协作的完整链路。拆开来看有 7 步：

### 第一步：工具注册（启动时）

Agent 启动时，把所有可用工具的 Schema 注册到工具注册中心。每个工具包含：

- **name**：工具的唯一标识，如 `query_elk_logs`
- **description**：自然语言描述，告诉模型这个工具干什么、什么时候该用
- **parameters**：JSON Schema 格式的参数定义，包含类型、必填、枚举值、默认值

模型侧拿到这些 Schema 后，在推理时就知道"有哪些工具可用、每个工具需要什么参数"。

### 第二步：用户输入 → 模型推理（请求阶段）

用户输入（自然语言或结构化事件）连同工具 Schema 一起发给 LLM。模型做一次推理，输出有两种可能：

- **直接回复**：模型判断不需要调用工具，直接生成文本回复
- **工具调用请求**：模型判断需要调用工具，输出结构化的工具调用指令：

```json
{
  "tool_calls": [
    {
      "id": "call_001",
      "function": {
        "name": "query_elk_logs",
        "arguments": "{\"service\": \"order-svc\", \"time_range\": \"last_1h\", \"level\": \"ERROR\"}"
      }
    }
  ]
}
```

### 第三步：参数解析与校验（网关侧）

拿到模型输出的 `arguments`（JSON 字符串），需要做：

1. **JSON 解析**：模型有时输出格式不合法（缺引号、多逗号），需要容错解析
2. **Schema 校验**：检查必填参数是否齐全、类型是否正确、枚举值是否合法
3. **安全过滤**：检查参数中是否有注入攻击（比如 SQL 注入、命令注入）

### 第四步：工具执行（业务侧）

根据工具名路由到对应的执行器，执行实际的业务逻辑。执行器需要：

- **超时控制**：设合理超时（如 5s），防止工具 hang 住整个链路
- **限流**：防止模型疯狂调用同一个工具（循环调用防护）
- **结果格式化**：把工具返回的原始数据（可能很大）压缩成模型能理解的摘要

### 第五步：结果回传 → 二次推理（模型侧）

把工具执行结果作为 `tool` 角色消息追加到对话历史，再调一次 LLM：

```
messages = [
  { role: "system", content: "你是运维助手..." },
  { role: "user", content: "order-svc 报错了，帮我查一下" },
  { role: "assistant", tool_calls: [{...}] },     // 模型决定调工具
  { role: "tool", content: "{\"logs\": [...]}" }   // 工具返回结果
]
```

模型拿到工具结果后，可能：
- **直接总结**：把工具结果翻译成自然语言回复用户
- **继续调工具**：一个工具的结果不够，需要再调另一个（比如先查日志，再查指标）
- **调用同一工具多次**：换个参数再查一次（比如查不同时间段）

### 第六步：循环控制

如果模型继续请求工具调用，重复第三到五步。需要设**最大循环次数**（通常 5-10 次），防止死循环。

### 第七步：最终生成

模型判断信息足够了，生成最终的自然语言回复，结束链路。

## 三、工具 Schema 设计最佳实践

### 3.1 Schema 核心要素

```json
{
  "type": "function",
  "function": {
    "name": "query_elk_logs",
    "description": "查询 ELK 日志系统。用于排查服务异常、错误日志分析、请求链路追踪。当用户询问服务报错、日志查询、异常排查时调用。",
    "parameters": {
      "type": "object",
      "properties": {
        "service": {
          "type": "string",
          "description": "服务名称，如 order-svc、payment-svc"
        },
        "time_range": {
          "type": "string",
          "enum": ["last_5m", "last_15m", "last_1h", "last_24h"],
          "description": "查询时间范围",
          "default": "last_1h"
        },
        "level": {
          "type": "string",
          "enum": ["ERROR", "WARN", "INFO"],
          "description": "日志级别过滤",
          "default": "ERROR"
        },
        "keyword": {
          "type": "string",
          "description": "关键词搜索，支持模糊匹配"
        },
        "limit": {
          "type": "integer",
          "description": "返回条数上限",
          "default": 20,
          "maximum": 100
        }
      },
      "required": ["service"]
    }
  }
}
```

### 3.2 六条最佳实践

**① description 决定调用准确率——这是最重要的字段**

模型靠 description 判断"什么时候该调这个工具"。写得模糊，模型就乱调。

```
❌ "查询日志"
✅ "查询 ELK 日志系统。用于排查服务异常、错误日志分析。当用户询问服务报错、日志查询时调用。"
```

要写清楚：**干什么** + **什么时候用** + **和别的工具的区别**。

**② 参数名要自解释，description 补充边界**

```
❌ { "q": { "type": "string" } }           // q 是什么？
✅ { "keyword": { "type": "string",         // 清晰
      "description": "关键词搜索，支持模糊匹配，不超过 200 字符" } }
```

**③ 用 enum 约束有限选项，减少模型幻觉**

模型容易"编造"参数值。用 enum 把可选值锁死：

```
❌ "time_range": { "type": "string" }       // 模型可能输出 "last_3_days"
✅ "time_range": { "type": "string",
     "enum": ["last_5m", "last_15m", "last_1h", "last_24h"] }
```

**④ 设 default 值，降低模型填参负担**

不是所有参数都需要模型填。有合理默认值的参数，模型可以不传，后端用默认值。

**⑤ required 只标真正必填的**

参数越少，模型填错的概率越低。可选参数尽量不要标 required。

**⑥ limit 类参数必须设上限**

防止模型传 `limit: 10000` 导致后端 OOM。Schema 里用 `maximum` 约束，后端再兜底截断。

## 四、结合项目的实际设计

### 4.1 运维 Agent 的工具集设计

```json
{
  "tools": [
    {
      "function": {
        "name": "query_elk_logs",
        "description": "查询 ELK 日志聚合。用于排查服务异常日志、错误堆栈分析、请求链路追踪。当告警涉及日志类问题时优先调用。"
      }
    },
    {
      "function": {
        "name": "query_prometheus_metrics",
        "description": "查询 Prometheus 监控指标。用于查看 CPU/内存/磁盘/网络等系统指标，以及服务级别的 QPS/RT/错误率。当告警涉及性能类问题时调用。"
      }
    },
    {
      "function": {
        "name": "converge_alerts",
        "description": "告警收敛。将同一故障的多条告警合并为一个故障事件，提取影响范围和严重程度。当短时间内出现大量相关告警时调用。"
      }
    }
  ]
}
```

**设计要点**：

- 每个工具的 description 明确写了"什么时候调用"，帮助模型做路由决策
- 工具之间有隐含的调用顺序：先 converge（收敛），再 query（查询），再 reason（推理）
- 工具返回结果做了压缩——ELK 原始返回可能上万条，只取 Top 20 条摘要

### 4.2 为什么不用一个"万能工具"？

面试时可能会被问："为什么不做一个 `execute_query` 工具，让模型自己拼查询？"

"三个原因：

1. **安全性**：万能工具等于给模型直接操作数据库的能力，一个参数拼错可能查出全量数据或拖垮数据库。专用工具封装了查询逻辑，参数空间是受限的。

2. **准确性**：工具越多越细，description 越精确，模型选对工具的概率越高。一个万能工具的 description 很难写清楚。

3. **可维护性**：每个工具独立实现、独立测试、独立限流。万能工具的逻辑会越来越复杂，最终变成一个难以维护的怪物。"

## 五、加分项

1. **主动提参数校验和安全过滤。** "模型输出的 arguments 是 JSON 字符串，不能直接信任——要做 Schema 校验、类型检查、注入检测。模型可能输出非法 JSON、缺失必填参数、甚至构造恶意参数。"——说明你有安全意识。

2. **提到结果压缩。** "工具返回的原始数据可能很大——ELK 返回上万条日志，Prometheus 返回几百个数据点。直接塞进 context 会撑爆 Token 预算。需要在后端做结果压缩，只保留模型推理需要的关键信息。"——说明你理解 Token 经济学。

3. **讲到循环控制和死循环防护。** "模型可能陷入循环——调工具 A 得到结果，判断不够，再调 A，永远不收敛。必须设最大循环次数，同时监控循环深度，超过阈值告警。"

4. **结合自己项目讲。** "我的运维 Agent 整合了 ELK、Prometheus、告敛为工具集。每个工具的 Schema 都精心设计了 description，帮助模型判断什么时候该查日志、什么时候该查指标。工具返回结果做了压缩——原始日志可能上万条，只取 Top 20 异常摘要回传给模型。"

5. **提到工具版本管理。** "工具 Schema 是有版本的——v1 的 `query_logs` 只支持关键词搜索，v2 增加了正则匹配。版本升级时要向后兼容，旧版本的调用不能因为 Schema 变了就报错。"

6. **对比不同平台的 Schema 格式。** "OpenAI、智谱、Anthropic 的 Function Calling Schema 格式略有差异——参数定义用 JSON Schema 还是自定义格式、是否支持 strict mode、parallel tool calls 的支持程度。做平台化的话需要一个统一的 Schema 适配层。"

## 六、追问应对

### Q1：并行工具调用怎么处理？结果聚合的策略是什么？

"两种情况：

**场景一：模型一次输出多个 tool_calls。** 比如同时查日志和查指标——这是模型判断两个查询是独立的，可以并行。处理方式是并发执行所有工具调用，用 `CompletableFuture.allOf` 等待全部完成，结果按 tool_call_id 对应回传。

**场景二：后端主动并行。** 有些工具调用虽然模型是串行请求的，但后端可以识别出它们之间没有数据依赖，主动并行化。

**结果聚合策略**：

```
工具 A 结果 (日志)  ──┐
                      ├──▶ 按 tool_call_id 顺序追加到 messages
工具 B 结果 (指标)  ──┘
                      │
                      ▼
               LLM 推理（同时看到两个工具的结果）
```

**关键点**：结果聚合不是后端做的——后端只是把所有工具结果按顺序追加到 messages 里，最终的'聚合'是 LLM 自己做的。后端要保证的是：每个工具结果的格式一致（都是 JSON 字符串），方便模型解析。

**我的运维 Agent 的实践**：告警来的时候，日志查询和指标查询是并行发的——ELK 查错误日志的同时，Prometheus 查 CPU/内存。两个结果都回来后一起传给模型，模型综合两方面信息做根因分析。"

### Q2：工具调用超时或失败时，降级策略怎么设计？

"三层降级：

**第一层：重试。** 网络抖动导致的瞬时失败，指数退避重试 2 次（间隔 1s、2s）。只对幂等操作重试——查日志可以重试，执行重启不能重试。

**第二层：降级返回。** 重试仍然失败，返回降级结果而不是报错：

```
正常: { logs: [...], total: 47 }
降级: { logs: [], total: 0, error: "ELK 查询超时，建议稍后重试或人工排查" }
```

把降级结果传给模型，模型会基于不完整信息给出建议，同时说明'日志查询暂不可用'。比整个 Agent 挂掉好。

**第三层：工具熔断。** 同一个工具连续失败 N 次（比如 5 次），触发熔断，后续请求直接跳过该工具，返回预设的降级结果。熔断窗口（比如 30s）后半开探测。

**我的运维 Agent 的实践**：告警场景对可用性要求很高——不能因为 ELK 挂了，Agent 就完全不工作。我们的降级策略是：日志查不到就只看指标，指标查不到就只看日志，两个都查不到就返回'建议人工排查'并自动升级告警等级。"

### Q3：工具的版本管理和权限控制怎么做？

"两个维度：

**版本管理**：

```
工具注册时带版本号:
tools/v1/query_elk_logs   → 基础查询
tools/v2/query_elk_logs   → 增加正则匹配

策略:
- 新版本发布后，旧版本保留至少 2 个版本周期
- 工具路由层根据调用方的 API 版本选择对应工具版本
- Schema 变更必须向后兼容：只能加可选参数，不能删参数或改类型
```

更实际的做法是**不暴露版本号给模型**——模型始终调用最新版本的工具，版本兼容在后端工具执行层处理。

**权限控制**：

```
工具权限 = f(租户角色, 工具敏感级别)

敏感级别:
  L0 (只读查询): query_logs, query_metrics    → 所有角色可用
  L1 (写操作):   update_config                → 管理员可用
  L2 (危险操作):  restart_service, scale_up   → 需二次确认

实现:
  ① 工具注册时标注 sensitive_level
  ② 调用前检查当前租户/角色是否有权限
  ③ L2 级操作返回确认请求而不是直接执行:
     { action: "confirm_required",
       message: "即将重启 order-svc，确认？",
       confirm_token: "xxx" }
```

**我的运维 Agent 的实践**：我们的自愈动作有明确的分级——服务重启、流量切换是 L2 级，需要人工确认（保留了人工介入兜底）；查日志、查指标是 L0 级，自动执行。权限配置存在租户级配置表里，不同租户可以自定义哪些动作允许自动执行。"

## 七、核心认知：LLM 只是冰山一角

面试时有一个判断能让你和"只会调 API"的候选人区分开来：

"很多人觉得 Agent 的核心是 LLM，其实不是。LLM 在整个 Function Calling 链路里只做两件事：**判断调哪个工具**、**把工具结果翻译成自然语言**。这两步加起来可能就 20% 的工程量。

剩下 80% 的工程量在后端：

| 环节 | LLM 做的事 | 后端做的事 |
|------|-----------|-----------|
| 调用前 | 选工具、填参数 | 参数校验、安全过滤、权限检查、限流 |
| 执行中 | 无 | 超时控制、并发调度、熔断降级、重试策略 |
| 调用后 | 总结结果 | 结果压缩、格式标准化、日志记录、指标上报、Bad Case 归档 |
| 整条链路 | 无 | Token 预算管理、循环次数控制、成本统计、链路追踪 |

**LLM 是大脑，但后端是骨骼和肌肉。** 大脑决定做什么，骨骼和肌肉决定能不能做成、做多快、做多稳。

这也是为什么我认为这个岗位需要的不是纯算法背景，而是**工程化能力**——Agent 从 Demo 到生产，差距全在后端这 80%。"

### 工具调用后的四层处理

**第一层：结果压缩**

工具返回的原始数据通常很大，不能直接塞给模型：

| 工具 | 原始返回 | 压缩方式 | 压缩后 |
|------|---------|---------|--------|
| ELK 日志 | 10000 条 | 按时间排序取 Top 20 + 去重相似堆栈 | 20 条摘要 |
| Prometheus | 几百个数据点 | 只保留异常区间（> 阈值的）+ 趋势描述 | "CPU 从 30%→85%，持续 15 分钟" |
| 告警收敛 | 100 条告警 | 按 service 聚合 + 去重 | "3 个服务共 47 条告警" |

原则：只保留模型推理需要的信息，丢掉噪音。

**第二层：结果校验**

工具返回的不一定靠谱，需要处理三种异常：

- 工具正常返回但数据异常（日志 0 条但服务确实报错）→ 标注可能原因
- 工具返回错误信息（ES timeout）→ 触发降级
- 工具返回格式异常（HTML 错误页而非 JSON）→ 兜底解析

**第三层：结果注入 context**

格式影响模型理解：

```json
// 好的格式：结构化、有摘要、有元数据
{
  "summary": "order-svc 过去 1 小时有 47 条 ERROR 日志",
  "pattern": "NullPointerException at OrderService.createOrder:142",
  "count": 47,
  "time_range": "2026-06-16T10:00~11:00",
  "top_5_logs": [...]
}

// 差的格式：原始日志直接拼接
"[2026-06-16 10:01:23] ERROR ... \n[2026-06-16 10:01:24] ERROR ..."
```

好的格式帮模型快速抓住重点，差的格式让模型自己去大海捞针。

**第四层：后处理**

模型生成最终回复后：

- 日志记录：本次调了哪些工具、每次耗时、Token 消耗
- 指标上报：工具调用成功率、延迟 P99、降级触发率
- Bad Case 标记：如果需要人工介入，归档整个链路供后续分析

## 八、面试策略总结

| 场景 | 怎么说 |
|------|--------|
| 面试官问完整链路 | 讲 7 步链路，重点讲参数校验、结果压缩、循环控制 |
| 面试官问 Schema 设计 | 讲 6 条最佳实践，重点讲 description 的写法 |
| 面试官问你的项目 | 讲运维 Agent 的三个工具 + 结果压缩 + 降级策略 |
| 面试官追问并行调用 | 讲 CompletableFuture 并发执行，结果按 id 对应回传 |
| 面试官追问超时降级 | 讲三层降级：重试 → 降级返回 → 熔断 |
| 面试官追问权限控制 | 讲工具敏感级别分级 + L2 操作需人工确认 |
| 面试官问"Agent 的核心是什么" | 讲"LLM 是大脑，后端是骨骼和肌肉"，强调 80% 工程量在后端 |

**核心原则：Function Calling 的难点不在"模型怎么调工具"，而在"调完之后怎么处理"——参数校验、超时降级、结果压缩、循环控制、权限管控。LLM 只占 20% 的工程量，剩下 80% 在后端。这才是工程化背景候选人的核心竞争力。**
