## 核心定位速查（面试前必看）

**你的 Agent 是什么：** 单 Agent + 工具调用 + 结构化 Prompt + 行动执行 + 人工兜底（Level 1）

**不是什么：** 不是 Multi-Agent，不是 ReAct 循环，不是简单 LLM 封装

**为什么选这个架构：** 运维场景需要可解释性和可控性，不能让 LLM 自己决定重启哪个服务

**核心叙事：** "Agent 落地的瓶颈不在架构有多复杂，而在工程化有多扎实"

**三个模块的关系：** 流程引擎（确定性编排）+ RAG 知识库（长期记忆）+ 智能运维 Agent（非确定性推理）= 一个完整的应急安全智能平台

### 1.1 智能运维 Agent（最高优先级）

| 预测问题 | 准备要点 |
|:---|:---|
| Agent 的完整链路是什么？从告警触发到自愈执行，每一步怎么走的？ | 画一张时序图：告警触发 → Agent 感知 → 调用 ELK/Prometheus 工具获取数据 → LLM 推理根因 → 匹配自愈动作 → 执行 → 结果校验 → 人工兜底 |
| Function Calling 具体怎么实现的？调的谁的 FC API？ | 说明是基于 Qwen 模型的 Function Calling 能力，工具 Schema 怎么定义的，参数怎么传的，结果怎么回传的 |
| Prompt 模板长什么样？举一个具体例子 | 准备一个 MQ 堆积场景的 Prompt 模板：输入是什么（告警信息 + 日志片段 + 指标数据），输出格式是什么（根因分析 + 建议动作） |
| 有没有遇到过 LLM 幻觉/误判？怎么处理的？ | 必须准备一个真实案例。比如 LLM 把正常的服务重启误判为故障，怎么发现的，怎么修复 Prompt 的 |
| 和传统规则引擎相比，Agent 方案的优势在哪？ | 规则引擎只能处理预设场景，Agent 能处理没见过的故障类型（通过 LLM 泛化能力）。但规则引擎更稳定，所以保留了规则引擎作为兜底 |
| 人工介入率从 100% 降到 15%，那 85% 自动处理的成功率是多少？ | 准备具体数字：自动处理中成功自愈的比例、误判的比例、需要人工纠正的比例 |
| 为什么选 Qwen 而不是 GPT-4/GLM？ | 本地化部署保证数据不出域（TOB 客户安全合规要求），Qwen 开源可私有化部署，推理延迟可控 |
| 这个系统跑了多久？监控多少服务？ | 准备具体数字：上线时间、监控的服务/机器数量、日均告警量 |

#### Agent 定位：诚实且有判断力

**你的 Agent 是什么层级：**

```
简单 LLM 封装 ←————[你的项目]————→ 完整多 Agent 调架
    |                    |                    |
  一个 Prompt        单 Agent + 工具调用      多 Agent 协作
  + API 调用         + 结构化推理链路         + 自主规划
                     + 行动执行层             + 动态决策
                     + 人工兜底
```

**实际架构链路：**

```
告警触发
  ↓
规则引擎预处理（过滤、聚合、去重）  ← 不是 LLM，是确定性规则
  ↓
Prompt 模板拼接（告警信息 + 日志片段 + 指标数据）  ← 结构化模板
  ↓
Qwen 模型推理（输出诊断结论）  ← LLM 在这一层
  ↓
结论解析 + 动作匹配（预定义的自愈动作表）  ← 规则匹配，不是 LLM 动态决策
  ↓
执行自愈动作 + 结果校验  ← 确定性执行
  ↓
人工兜底（置信度不够时）
```

**不要过度包装，面试官是懂行的。正确的定位话术：**

> "我的智能运维 Agent 是一个**单 Agent + 工具调用**的架构。LLM 负责故障根因分析（非确定性的部分），工具调用和行动执行是确定性的（通过 Function Calling 触发预定义的自愈动作）。
>
> 我没有做 Multi-Agent 或完全自主的 ReAct 循环，原因是运维场景需要**可解释性和可控性**——你不能让 LLM 自己决定重启哪个服务，必须有预定义的动作表和置信度阈值。
>
> 但我做了完整的工程化：工具编排、Prompt 模板管理、Bad Case 归集、人工兜底机制、全链路可观测。这正是我认为 Agent 落地的关键——不是让 Agent 更自主，而是让 Agent 更可靠。"

**面试官追问"有没有想过做 Multi-Agent"时的回答：**

> "想过。未来可以拆成三个 Agent 协作：**感知 Agent** 负责从 ELK/Prometheus 采集数据并预处理，**诊断 Agent** 负责根因分析，**执行 Agent** 负责自愈动作的选择和执行。三个 Agent 通过消息队列通信，各自有独立的 Prompt 模板和工具集。
>
> 但当前阶段我选择单 Agent，原因是：第一，运维场景的工具集有限（ELK/Prometheus/告警），不需要动态工具发现；第二，故障诊断的推理链路短（采集→分析→动作），不需要多步规划；第三，单 Agent 的可观测性和调试难度远低于 Multi-Agent。
>
> 等工具集扩展到 20+、推理链路超过 5 步、需要跨系统协作时，再拆 Multi-Agent 更合理。"

**为什么这个定位是加分的：**

| 面试官的判断 | 为什么加分 |
|:---|:---|
| 诚实 | 你没有吹成 Multi-Agent，说明你真正理解 Agent 的层级 |
| 有判断力 | 你解释了为什么选单 Agent 而不是 Multi-Agent（运维需要可控性），说明你做了架构决策 |
| 工程化扎实 | 你强调的是工程化能力而不是 Agent 的花哨程度，匹配 JD 要求的"稳定性与工程效能" |
| 有演进规划 | Multi-Agent 的演进方向说明你不是只会做当前方案，还想过未来怎么演进 |

**核心原则：面试官想听到的不是"你的 Agent 有多智能"，而是"你的 Agent 有多可靠"。**

#### 防御话术：当面试官说"这不是 Agent"时

**这是你最可能被挑战的点，必须提前准备好。**

**第一层反击：重新定义问题**

> "我理解的 Agent 不是'越自主越好'，而是'在合适的环节引入 LLM 的非确定性能力'。传统规则引擎只能处理 if-else 预设的场景，我的系统用 LLM 做根因分析——给它一个没见过的故障组合，它能推理出可能的原因，这是规则引擎做不到的。工具调用和行动执行用确定性逻辑，是因为这些环节不需要'创造力'，需要的是'可靠性'。在正确的环节用正确的范式，这才是架构判断力。"

**第二层反击：用结果说话**

> "不管它叫不叫 Agent，它在生产环境跑了 [X 个月]，夜间告警人工介入率从 100% 降到 15%，MTTR 从 45 分钟压缩到 8 分钟。这个结果是传统规则引擎做不到的，也是纯 LLM 封装做不到的。它在规则引擎和 LLM 之间找到了正确的平衡点。"

**第三层反击：拿定义说话**

> "OpenAI 对 Agent 的定义是'能使用工具完成任务的 LLM 系统'。我的系统用 Function Calling 调用 ELK/Prometheus 工具集，基于 LLM 推理结果触发行动，完全符合这个定义。Agent 不等于全自主，不等于 Multi-Agent，不等于 ReAct。"

**第四层反击：反问面试官**

> "在运维场景下，你会让 LLM 自己决定重启哪个服务吗？如果答案是'不会'，那预定义动作表 + 置信度阈值就是正确的架构选择。"

**最后兜底：承认差距，但把"差距"定义为"选择"**

> "如果按 Multi-Agent + 自主规划的标准，我的项目确实不够格。但我做的是生产级系统，不是研究原型。在生产环境下，可解释性、可控性、可观测性比自主性更重要。我选择在 LLM 推理层引入非确定性，在执行层保持确定性，这是经过权衡的架构决策，不是因为我不懂 Multi-Agent。"

**面试官质疑的本质是考察两件事：**

| 考察点 | 你的回答需要证明的 |
|:---|:---|
| 你对 Agent 的理解深度 | 你能讲清楚 Level 0/1/2/3 的区别、每个层级的适用场景、为什么选 Level 1 而不是 Level 3 |
| 你的架构判断力 | 你能解释"为什么在运维场景下选单 Agent 而不是 Multi-Agent"，并给出有说服力的理由 |

**这两点比"你的 Agent 有多高级"重要得多。**

### 1.2 RAG 知识库（高优先级）

| 预测问题 | 准备要点 |
|:---|:---|
| RAG 的完整链路是什么？ | 文档摄入 → 分块 → BGE Embedding → Qdrant 存储 → 查询时 HyDE 重写 → Dense + BM25 双轨检索 → RRF 融合 → LLM 生成 → 溯源 |
| 为什么选 Qdrant 不选 Milvus？ | Milvus 需要 Etcd+MinIO+Pulsar，运维成本高；Qdrant 单机 Docker 一行启动，零外部依赖；单租户几百份文档，Qdrant 单机完全够用 |
| 为什么用 Flat 不用 HNSW？ | 单租户 200~2000 份文档，Flat 查询 1ms，零参数零维护；超过 5000 份才切 HNSW，目前只有 3 个大客户触发 |
| 三层缓存具体怎么设计的？ | Query 答案缓存（API Gateway 层，命中率 55%）→ HyDE 缓存（查询重写后，30%）→ LLM 响应缓存（LLM 调用前，40%）。三层叠加降本 82% |
| 新租户冷启动问题怎么解决的？ | 租户级同义词库（自动提取高频术语）+ 冷启动保护期（BM25 权重提升）+ 文档质量检测（覆盖率不足时提醒补充） |
| 精确信息准确率 99.8% 怎么保证的？ | 结构化路径兜底：楼层/区域/设备编号走 SQL 查询空间资产库，不走 RAG；后处理加楼层一致性校验 |
| 为什么不微调 Embedding 模型？ | 通用 BGE-large 在应急安全领域 Recall 已经 97%，微调可能提到 98%，1% 的提升不值得 3 万块的投入 |

### 1.3 流程引擎（中优先级）

| 预测问题 | 准备要点 |
|:---|:---|
| 分片 EventLoop 和 Netty EventLoop 的区别？ | 核心模型一致（BlockingQueue.take() 阻塞等待），但我们的 EventLoop 内部有租户公平轮询机制，Netty 没有 |
| 为什么不用分布式锁保证实例级串行化？ | 按 instance_id 哈希分片，同一实例永远路由到同一个 EventLoop，天然串行化，不需要锁 |
| 五层隔离每一层的作用？ | 入口限流（拦截超限流量）→ 分片调度（公平轮询）→ 分层执行（大客户独立 Lane + 中小客户按套餐分组）→ 实例配额（防死实例累积）→ 租户熔断（兜底异常租户） |
| 主键更新 + 乐观锁的性能怎么样？ | InnoDB 主键更新走聚簇索引，O(log n)，单机几万 TPS；`AND status = currentStatus` 是天然乐观锁 |
| "快速失败 + 人工介入"和"自动重试"怎么选？ | 业务体量可预估，超出预期 = 失败。自动重试掩盖问题，人工介入根治。但 Agent 场景下需要区分：确定性任务快速失败，非确定性推理可以重试 |

## 二、Agent 架构知识（Java.md 要求的核心能力）

### 2.1 你的 Agent 架构层级定位

**先搞清楚自己在哪，再决定讲什么。**

```
Level 0  简单 LLM 封装：一个 Prompt + API 调用，没有工具，没有行动
Level 1  单 Agent + 工具调用：结构化 Prompt + Function Calling + 行动执行 + 人工兜底  ← 你在这里
Level 2  单 Agent + ReAct 循环：动态工具选择 + 多步推理 + 自我反思
Level 3  Multi-Agent 协作：多个 Agent 分工 + 自主规划 + 动态决策
```

**你的智能运维 Agent 是 Level 1**：LLM 负责根因分析（非确定性），工具调用和行动执行是确定性的（预定义动作表 + 置信度阈值）。不是 Level 3 的 Multi-Agent，也不是 Level 0 的简单封装。

**为什么选 Level 1 而不是 Level 3**：运维场景需要可解释性和可控性，不能让 LLM 自己决定重启哪个服务。工具集有限（ELK/Prometheus/告警），推理链路短（采集→分析→动作），不需要多步规划和多 Agent 协作。

**演进方向**：等工具集扩展到 20+、推理链路超过 5 步、需要跨系统协作时，拆三个 Agent（感知 Agent / 诊断 Agent / 执行 Agent）通过消息队列通信。

### 2.2 必须能口述 15 分钟的内容

**通用 Agent 架构图（面试时画白板用）：**

```
用户输入
  ↓
意图识别（规则/分类模型/LLM 直接判断）
  ↓
任务拆解（ReAct / Plan-and-Execute / ToT）
  ↓
记忆管理
├── 短期记忆：对话窗口 / 滑动窗口 / 摘要压缩
├── 长期记忆：向量检索（RAG）
└── 外部记忆：知识库 / 数据库
  ↓
推理策略（CoT / Self-Consistency / Reflexion）
  ↓
工具编排（Function Calling → 工具执行 → 结果回传）
  ↓
结果生成 + 效果评测
```

**你的智能运维 Agent 实际架构（面试时重点讲）：**

```
告警触发
  ↓
规则引擎预处理（过滤、聚合、去重）  ← 确定性规则，不是 LLM
  ↓
Prompt 模板拼接（告警信息 + 日志片段 + 指标数据）  ← 结构化模板
  ↓
Qwen 模型推理（Function Calling 调用 ELK/Prometheus）  ← LLM 核心能力
  ↓
结论解析 + 动作匹配（预定义自愈动作表 + 置信度阈值）  ← 确定性匹配
  ↓
执行自愈动作 + 结果校验  ← 确定性执行
  ↓
人工兜底（置信度不够时）
```

### 2.3 每个模块的面试表达

| 模块 | 一句话定义 | 你的项目映射 |
|:---|:---|:---|
| 意图识别 | 判断用户想做什么 | 智能运维 Agent：规则引擎判断告警类型（MQ堆积/DB慢查询/Redis热点），不是 LLM 做意图识别 |

### 2.4 必须准备的对比表

| 对比项 | A 方案 | B 方案 | 你的选择和理由 |
|:---|:---|:---|:---|
| 结构化 Prompt vs ReAct 循环 | 预定义 Prompt 模板，输出可控 | 动态推理+行动循环，更灵活 | 运维 Agent 用结构化 Prompt（需要可控性和可解释性），不是 ReAct |
| Function Calling vs 自研工具调用 | 模型原生支持，标准化 | 自定义灵活，可控性强 | 用 Qwen 的 Function Calling（标准化），但工具 Schema 自己定义 |
| 单 Agent vs Multi-Agent | 简单可控，可观测性好 | 分工明确，适合复杂场景 | 当前单 Agent（工具集有限、推理链路短），未来演进 Multi-Agent |
| 本地部署 vs API 调用 | 数据不出域，延迟可控 | 模型能力更强，无需运维 | 本地部署 Qwen（TOB 客户安全合规要求） |
| Flat vs HNSW | 零参数零维护，小数据量快 | 大数据量性能好 | 单租户 < 5000 走 Flat，> 5000 切 HNSW |

## 三、大模型应用基础（概念级，不要深入算法）

### 3.1 必须能讲清的概念

| 概念 | 一句话解释 | 和你的项目的关系 |
|:---|:---|:---|
| Prompt Engineering | 结构化指令约束模型输出空间 | 智能运维 Agent 的故障诊断 Prompt 模板 |
| Function Calling | 模型输出结构化工具调用请求，后端执行后回传结果 | Agent 调用 ELK/Prometheus 的机制 |
| RAG | 检索相关文档作为上下文，减少幻觉 | 应急安全知识库的核心架构 |
| SFT | 用标注数据微调模型，让模型学会特定任务 | 了解即可，没有实际做过 |
| DPO | 直接用偏好数据优化模型，比 RLHF 简单 | 了解即可 |
| HyDE | 先让 LLM 生成假设性答案，用答案做检索，提高召回率 | RAG 查询重写策略 |

### 3.2 高频问题话术

**"你没有 1 年 Agent 经验，怎么胜任？"**

> "我的智能运维 Agent 已经在生产环境跑了 [X 个月]，服务 120 个租户（含 4 家世界 500 强），7×24 小时运行。这不是 Demo，是真实的生产系统——夜间告警人工介入率从 100% 降到 15%，MTTR 从 45 分钟压缩到 8 分钟。同时我的 RAG 知识库和流程引擎也是 Agent 系统的核心基础设施——RAG 是 Agent 的长期记忆，流程引擎是 Agent 的工作流编排基座。我坦诚说，我的 Agent 是单 Agent + 工具调用架构，不是 Multi-Agent。但我认为 Agent 落地的瓶颈不在架构有多复杂，而在工程化有多扎实：多租户隔离、可观测性、稳定性保障、成本管控，这些是我 8 年积累的。"

**"Agent 和传统自动化工作流有什么区别？"**

> "传统工作流是确定性的 DAG，每个节点做什么事先定义好。Agent 的核心区别是引入了 LLM 的推理能力，能处理事先没预见到的情况。但 Agent 不是非黑即白的——我的智能运维 Agent 在 LLM 推理层是非确定性的，在工具调用和行动执行层是确定性的（预定义动作表 + 置信度阈值）。这不是偷工减料，是运维场景需要可控性。两者不是替代关系，是互补关系——确定性流程用引擎，非确定性推理用 Agent。"

**"模型幻觉怎么解决？"**

> "三个手段。第一，结构化 Prompt 约束输出格式，减少自由发挥空间；第二，RAG 提供事实依据，让模型基于检索结果回答而不是凭空生成；第三，关键决策保留人工兜底，比如自愈动作执行前需要校验诊断结论的置信度。"

**"你会 Python 启吗？"**

> "能阅读和理解，但不是主力语言。Java 生态有 Spring AI，可以完成同样的 Agent 开发。我的智能运维 Agent 和 RAG 知识库都是 Java 实现的。"

## 五、场景设计题（结合 JD 中的业务场景）

### 5.1 智能对话取数（NL2SQL）

```
用户："上个月 3 楼的消防演练合格率是多少？"
  ↓
意图识别：查询类意图，涉及消防演练数据
  ↓
Schema 选择：fire_drill 表 + floor 表
  ↓
SQL 生成：SELECT pass_rate FROM fire_drill WHERE floor=3 AND month='2025-05'
  ↓
权限校验：当前用户是否有 3 楼数据的访问权限
  ↓
执行 + 结果解读：返回 "3 楼消防演练合格率 95.2%"
```

**你的优势**：做过开放 API 和数据报表，对 Schema 设计、权限控制、查询优化有 8 年经验。

### 5.2 Agent 驱动的自动化运营

```
触发条件：定时巡检 / 事件触发
  ↓
任务拆解：检查消防设备 → 检查逃生通道 → 检查人员签到
  ↓
工具执行：调用 IoT 传感器 API → 调用摄像头 API → 调用签到系统 API
  ↓
结果校验：对比安全标准
  ↓
异常处理：发现隐患 → 自动生成整改通知 → 推送给安全管理员
  ↓
人工介入：复杂隐患需要人工确认处理方案
```

**你的优势**：流程引擎就是做任务编排的，DAG + 状态机 + 人工审批节点，直接复用。
