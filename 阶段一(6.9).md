## 一、集合框架

### 第1题（递进式）

> **你项目中有一个高并发的计数器，QPS 约 5万，需要统计接口调用次数。**
>
> 1. 你会用 `ConcurrentHashMap` 还是 `LongAdder`？为什么？  
> 2. 如果计数器需要按用户ID维度统计（`Map<String, AtomicInteger>`），用户量 1000万，内存暴涨，你会怎么优化？  
> 3. 如果要求计数结果最终落库，但允许秒级延迟，你会怎么设计整个链路？

**【考察方向】**
- 第一层：并发计数器的选型（`AtomicInteger` vs `LongAdder` vs `ConcurrentHashMap`）
- 第二层：高 Cardinality 场景下的内存优化（`Long2IntOpenHashMap`、`RoaringBitmap`、分桶统计）
- 第三层：架构思维（本地聚合 → 批量刷盘 → 异步落库，减少锁竞争和网络IO）

**【引导提示】**
> 先回答第一问时，不要只说 "用 LongAdder"。要讲清楚：`AtomicInteger` 是全局单点 CAS，`LongAdder` 是空间换时间的分段累加，5w QPS 下 `LongAdder` 的吞吐量是 `AtomicInteger` 的 10 倍以上。  
> 第二问是陷阱题：`ConcurrentHashMap<String, AtomicInteger>` 在 1000万 Key 下内存占用超过 10GB，要考虑用**堆外内存**或**近似算法**（Count-Min Sketch）。  
> 第三问考察你的系统观：计数结果不需要实时精确，可以用**环形数组分桶** + **定时刷盘**，参考 Linux `vmstat` 的实现思路。

---
第一个答案：单纯的计数可以使用LongAddr或者是AtomicInter都可以。当QPS到底5W甚至更高的时候,推荐使用LongAddr。原因是因为LongAddr的结构,使用空间换时间的做法。计数=base+Cell[]
而AtomicInteger使用CAS方案实现,可能导致缓存一致性流量爆炸. 如果追求计数统计的精准,可以使用AtomicInteger。

2 应该将存储进行优化,再考虑使用压缩算法尽可能的减少,最终再考虑存放在堆内还是堆外

## 二、并发编程

### 第2题（场景式）

> **你重构一个 IM 系统的消息推送模块，发现线上 CPU 偶尔飙到 90%，`top -Hp` 看到多个线程状态是 `RUNNABLE`，但火焰图上 `java.util.concurrent.atomic` 的占比很高。**
>
> 1. 你会怎么进一步确认根因？`RUNNABLE` 但不 `BLOCKED`，说明不是锁竞争，那会是什么？  
> 2. 如果发现是 `AtomicInteger` 在高并发下的 CAS 自旋导致，除了换成 `LongAdder`，还有什么优化手段？  
> 3. 如果让你重新设计这个计数器模块，完全避免 CAS 自旋，你会怎么设计？

**【考察方向】**
- 第一层：CPU 排查工具链（`top -Hp` → `jstack` → Async-Profiler 火焰图 → `perf` 看 CPU 缓存命中率）
- 第二层：CAS 自旋的底层代价（CPU 缓存一致性协议 MESI、总线风暴、伪共享）
- 第三层：无锁化设计（ThreadLocal 累加 + 定时汇总、环形队列、Disruptor 的 Sequence）

**【引导提示】**
> 第一问的关键是：`RUNNABLE` + `atomic` 占比高 = **CAS 自旋**。不是锁竞争，但比锁竞争更隐蔽。要讲清楚怎么通过火焰图定位到具体代码行。  
> 第一步 jstack -l pid temp.dump . 第二步 grep -A 20 `java.util.concurrent.atomic` temp.dump 从dump文件中找到可疑的代码片段
> 
> 第二问不要只答 `LongAdder`。可以提 **伪共享（False Sharing）** 问题：`LongAdder` 的 `Cell` 数组如果不对齐缓存行（64B），多个线程修改相邻 Cell 会导致缓存失效。可以提 `@Contended`（JDK 8）或手动 padding。  
> 第三问是分水岭：完全无锁 = **ThreadLocal 预聚合**。每个线程本地维护计数器，每 1s 批量汇总到一个环形数组，后台线程刷盘。参考 `Netty` 的 `Recycler` 设计。

### 第3题（原理+设计）

> **AQS 的 `state` 是 `volatile int`，但 `ReentrantLock` 的 `tryAcquire` 用的是 `compareAndSetState`（CAS），而不是简单的 `setState`。为什么？**
>
> 1. 如果 `state` 已经是 `volatile`，直接 `setState(1)` 再判断不就行了吗？为什么必须用 CAS？  
> 2. 如果让你用 `synchronized` 而不是 AQS 实现一个独占锁，什么场景下 `synchronized` 性能反而更好？  
> 3. AQS 的 `CLH` 队列是用双向链表实现的，那线程入队时的 `CAS` 操作是在竞争什么资源？

**【考察方向】**
- 第一层：`volatile` 的可见性 ≠ 原子性，`setState` + `if` 不是原子操作
- 第二层：AQS vs `synchronized` 的性能权衡（低竞争时 `synchronized` 的偏向锁无 CAS，高竞争时 AQS 的队列减少上下文切换）
- 第三层：CLH 队列的 `CAS` 竞争的是 `tail` 指针，入队时只用一次 CAS，出队时不需要 CAS（因为只有一个线程能获取到锁）

**【引导提示】**
> 第一问是经典陷阱：`volatile` 保证的是**单个变量的读写可见性**，但 `if (state == 0) { state = 1; }` 是**先读后写**，中间可以被其他线程打断。必须用 CAS 把 "判断 + 赋值" 变成原子操作。  
> 第二问要讲 JDK 15 废弃偏向锁的背景，但也要承认：**单线程反复获取锁**时，`synchronized` 的偏向锁比 AQS 的 CAS 更快（无自旋、无队列）。  
> 第三问考查源码深度：CLH 队列的 `enq()` 方法中，`compareAndSetTail(pred, node)` 竞争的是 `tail`；`acquireQueued()` 中 `shouldParkAfterFailedAcquire()` 把前驱节点的 `waitStatus` 改为 `SIGNAL` 时，用的是普通的 `set`，因为只有一个线程能修改自己的前驱。

## 三、JVM

### 第4题（故障排查）

> **你负责的一个服务，每天凌晨 2:00 准时触发 Full GC，STW 时间 3-5 秒，导致监控告警。但白天流量更大，却没有 Full GC。**
>
> 1. 你会怎么排查？G1 的哪些参数或日志关键字能帮你定位？  
> 2. 如果发现是 `System.gc()` 被某个定时任务触发，你怎么找到具体代码位置？  
> 3. 如果排除 `System.gc()`，发现是 Humongous Region 分配失败导致的 `Evacuation Failure`，你会怎么优化？

**【考察方向】**
- 第一层：G1 GC 日志分析（`-Xloggc:` 或 JDK 9+ 的 `-Xlog:gc*`）
- 第二层：`System.gc()` 的排查手段（`jstat -class`、Arthas 的 `stack java.lang.System gc`、BTrace）
- 第三层：Humongous Region 的分配策略（> 0.5 Region 直接进老年代）、`G1HeapRegionSize` 调优、业务层面拆分大对象

**【引导提示】**
> 第一问要讲 G1 日志的关键字：`Pause Full (System.gc())`、`Pause Full (G1 Evacuation Pause)`、`to-space exhausted`、`Humongous (reclaim)`。白天没有 Full GC 但凌晨有 = **定时任务**。  
> 第二问 `System.gc()` 的定位是实战技能：Arthas `trace java.lang.System gc` 可以直接抓到调用栈；或者 JVM 参数 `-XX:+DisableExplicitGC` 先禁用，观察 Full GC 是否消失。  
> 第三问 Humongous Region 是 8年经验的区分点：G1 的 `G1HeapRegionSize` 默认根据堆大小计算（1MB ~ 32MB），但 `Humongous` 的阈值是 `> 0.5 RegionSize`。如果业务频繁分配 1MB 数组，而 RegionSize = 1MB，那么任何 > 512KB 的对象都是 Humongous，直接进老年代，导致频繁 Full GC。**解决方案**：调大 `G1HeapRegionSize`（如 16MB），或业务层面拆分大对象。

### 第5题（设计权衡）

> **你的服务堆内存 32G，目前用 G1，Full GC 频率尚可，但 Young GC 的 STW 时间偶尔超过 200ms，业务要求 P99 < 50ms。**
>
> 1. 你会先调优 G1 的哪些参数？`MaxGCPauseMillis` 设成 20ms 有用吗？  
> 2. 如果调优后 Young GC 仍然无法满足，你会考虑 ZGC 吗？迁移的风险是什么？  
> 3. 如果不换 GC，从架构层面（而不是 JVM 参数）有什么手段可以降低 GC 延迟？

**【考察方向】**
- 第一层：G1 调优参数（`MaxGCPauseMillis`、`G1NewSizePercent`、`G1MaxNewSizePercent`、`G1HeapRegionSize`）
- 第二层：ZGC 的染色指针 + 读屏障 + 吞吐量 trade-off
- 第三层：架构层面的 GC 优化（堆外内存、对象池、无GC设计、分代隔离）

**【引导提示】**
> 第一问：`MaxGCPauseMillis=20` 不一定有用，G1 只是**目标**，不是**承诺**。Young GC 时间长通常是因为新生代太大或对象晋升太快。可以调 `G1NewSizePercent` 和 `G1MaxNewSizePercent` 限制新生代大小，或调大 `G1HeapRegionSize` 减少 Region 数量。  
> 第二问 ZGC 的迁移风险：JDK 11 实验特性，JDK 17 生产可用；吞吐量比 G1 低 5-10%；JNI/native 代码可能不兼容染色指针；运维团队的学习成本。  
> 第三问是架构师思维：如果 JVM GC 优化到极限还不够，考虑**堆外内存**（DirectBuffer、MappedByteBuffer）、**对象池**（Netty 的 Recycler、 Commons Pool）、甚至**无 GC 语言**（Rust 写关键模块，通过 JNI 调用）。或者把大对象、长生命周期对象剥离到独立服务，本服务保持小堆、频繁 Young GC。

## 四、Spring 生态

### 第6题（原理+坑）

> **你们项目里有一个 `@Service` 类，内部方法 A 调用了方法 B，方法 B 上有 `@Transactional`。**
>
> 1. 方法 A 没有 `@Transactional`，方法 B 有，A 调用 B 时，B 的事务会生效吗？为什么？  
> 2. 如果方法 A 也有 `@Transactional(propagation = Propagation.REQUIRED)`，方法 B 的传播行为是 `REQUIRES_NEW`，A 调用 B 时，B 抛异常了，A 会回滚吗？  
> 3. 如果让你从架构层面避免这类事务问题，你会怎么设计？（不是代码层面的 @Lazy 或注入自身）

**【考察方向】**
- 第一层：AOP 代理机制，同类方法调用不走代理，@Transactional 失效
- 第二层：事务传播行为的嵌套规则（REQUIRED 沿用外层事务，REQUIRES_NEW 挂起外层事务开新事务）
- 第三层：架构设计（事务边界应在 Service 层的最外层方法，内部方法不应独立加事务；或采用 CQRS 分离读写，写操作统一走事务门面）

**【引导提示】**
> 第一问是送分题：同类方法调用 `this.methodB()` 不走代理，@Transactional 失效。但要讲清底层：Spring AOP 代理对象 intercept 的是外部调用，内部调用是 `this` 指针直接调用，绕过了代理。  
> 第二问是陷阱题：`REQUIRES_NEW` 会挂起 A 的事务，开一个新事务。B 抛异常 → B 回滚，但 A 的事务**继续执行**（因为 B 的新事务已挂起，异常被抛出到 A 时，如果 A 没有 catch，A 也会回滚；如果 A catch 了，A 不回滚）。要讲清**异常传播链**。  
> 第三问考察架构洁癖：事务应该是**粗粒度**的，在 Service 的入口方法统一加 `@Transactional`，内部方法不加。如果必须拆分，用**事务门面模式**（Transaction Facade），把事务控制集中到一层，避免散落在各个方法上。

### 第7题（源码深度）

> **Spring Boot 的自动配置不是魔法，但很多人说不清楚 `spring.factories` 到 `AutoConfiguration` 的完整链路。**
>
> 1. `spring.factories` 里的类是在 Bean 生命周期的哪个阶段被加载的？`BeanFactoryPostProcessor` 和 `BeanPostProcessor` 的区别是什么？  
> 2. 如果两个 Starter 都定义了同一个 `@Bean`（比如 `DataSource`），Spring Boot 怎么决定用哪个？`@ConditionalOnMissingBean` 和 `@Primary` 的优先级是什么？  
> 3. 如果让你写一个 Starter，支持根据配置文件动态开关某个功能，你会怎么设计？（不能用 `@ConditionalOnProperty` 之外的方案）

**【考察方向】**
- 第一层：Spring Boot 自动配置的加载时机（`SpringFactoriesLoader` → `BeanDefinitionRegistryPostProcessor` → `ConfigurationClassPostProcessor`）
- 第二层：`@Conditional` 家族 + `@Primary` 的优先级（条件装配先于 `@Primary`）
- 第三层：Starter 设计模式（Enable 注解 + ImportSelector + 配置属性绑定）

**【引导提示】**
> 第一问要讲清时序：`spring.factories` 的类在 `refresh()` 的 `invokeBeanFactoryPostProcessors()` 阶段被加载，此时 Bean 还没实例化，只是注册 `BeanDefinition`。`BeanFactoryPostProcessor` 操作 `BeanDefinition`，`BeanPostProcessor` 操作 Bean 实例（AOP 代理在这里发生）。  
> 第二问 `@ConditionalOnMissingBean` 和 `@Primary` 的优先级：`@Conditional` 家族先执行，如果条件不满足，Bean 根本不注册；`@Primary` 是在多个候选 Bean 中做选择。所以 `@ConditionalOnMissingBean` 优先级高于 `@Primary`。  
> 第三问是设计题：可以用 `@Import(MyImportSelector.class)` + `ImportSelector.selectImports()` 动态根据配置类决定导入哪些配置类。比 `@ConditionalOnProperty` 更灵活，支持复杂的开关逻辑。

## 五、数据存储

### 第8题（实战排查）

> **线上 MySQL 出现一条慢 SQL，执行时间 8s，业务超时。**
>
> 1. 你的排查第一步是什么？`EXPLAIN` 的哪些字段最关键？`type=range` 但 `rows=100000`，索引用上了吗？  
> 2. 如果发现是 `WHERE create_time > '2024-01-01' ORDER BY id DESC LIMIT 10`，`create_time` 有索引，但仍然慢，为什么？怎么优化？  
> 3. 如果这条 SQL 的数据量再涨 10 倍，你现有的优化方案还够用吗？从架构层面你会做什么？

**【考察方向】**
- 第一层：`EXPLAIN` 分析（`type`、`key`、`rows`、`Extra`）
- 第二层：索引下推（ICP）、覆盖索引、文件排序（Using filesort）优化、LIMIT 深分页问题
- 第三层：架构演进（归档、分表、异构查询 ES/ClickHouse、冷热分离）

**【引导提示】**
> 第一问关键字段：`type`（ALL/index/range/ref）、`key`（实际使用的索引）、`rows`（扫描行数，估算值）、`Extra`（Using filesort / Using temporary / Using index）。`type=range` 但 `rows=100000` = **索引范围太大**，虽然走了索引，但扫描了 10万行，再回表取数据，IO 爆炸。  
> 第二问是经典场景：`create_time` 索引 + `ORDER BY id DESC` → **文件排序**。因为 `create_time` 索引的叶子节点按 create_time 排序，但你需要按 id 排序，MySQL 必须额外排序。优化方案：**联合索引** `(create_time, id)` 实现覆盖索引 + 消除 filesort；或者**改写 SQL** 用子查询先查 id，再回表。  
> 第三问是分水岭：单表优化到极限后，必须做**归档**（3个月前数据移到历史库）、**分表**（按时间或 hash）、**异构查询**（ES/ClickHouse 做复杂查询）。不要再说"升服务器配置"。

### 第9题（锁机制深度）

> **InnoDB 的 RR 隔离级别下：**
>
> 1. `SELECT * FROM order WHERE id = 100`（id 是唯一索引），会加什么锁？  
> 2. `SELECT * FROM order WHERE id > 100 FOR UPDATE`，事务 A 执行后，事务 B `INSERT INTO order (id) VALUES (105)`，会被阻塞吗？为什么？  
> 3. 如果事务 A 先执行 `SELECT * FROM order WHERE id = 100`（普通 SELECT），然后事务 B `INSERT id=105`，然后事务 A 再执行 `UPDATE order SET ... WHERE id = 105`，能更新成功吗？这是幻读吗？

**【考察方向】**
- 第一层：唯一索引等值查询的锁类型（记录锁 Record Lock）
- 第二层：范围查询的 Next-Key Lock（记录锁 + 间隙锁 Gap Lock），阻塞插入
- 第三层：快照读 vs 当前读的幻读问题（RR 的"半解决"状态）

**【引导提示】**
> 第一问：`id` 是唯一索引且记录存在 → **记录锁（Record Lock）**，不是间隙锁。  
> 第二问：`id > 100 FOR UPDATE` 是范围当前读，加 **Next-Key Lock**（`(100, +∞)` 的间隙锁 + 记录锁）。事务 B 插入 `id=105` 会命中间隙锁，被阻塞。  
> 第三问是分水岭：`SELECT`（快照读）看不到事务 B 插入的 105，但 `UPDATE ... WHERE id = 105`（当前读）能看到，并且能更新成功。然后事务 A 再次 `SELECT` 就能看到 105 了——**这就是 RR 隔离级别下的幻读！** 因为 MVCC 只保证快照读一致，当前读会读取最新数据。

## 六、Redis

### 第10题（设计题）

> **你们系统有一个热点配置数据，QPS 50万，读多写少，要求 P99 < 5ms。**
>
> 1. 你会怎么设计缓存架构？本地缓存（Caffeine）+ Redis 还是直接 Redis？  
> 2. 如果写操作是运营后台修改配置，你怎么保证本地缓存和 Redis 的一致性？延迟双删的延迟时间怎么定？  
> 3. 如果 Redis 挂了，本地缓存还在，但配置已经更新了，你怎么发现本地缓存是脏数据？

**【考察方向】**
- 第一层：多级缓存架构选型（本地缓存抗热点 + Redis 做共享）
- 第二层：缓存一致性方案（延迟双删、Canal Binlog 监听、版本号校验）
- 第三层：故障降级和脏数据发现（TTL 兜底、版本号对比、主动推送失效）

**【引导提示】**
> 第一问：50w QPS 纯走 Redis 需要 500+ 连接，且网络 RTT 至少 1ms，P99 很难压到 5ms 以内。必须加 **Caffeine 本地缓存**（读取 < 1μs），Redis 做一致性保障。  
> 第二问：延迟双删的延迟时间 = **业务读操作的最大耗时 + 主从同步延迟**。通常 200ms ~ 500ms，可以用业务 P99 读延迟作为参考。更好的方案是 **Canal 监听 Binlog**，异步推送缓存失效，延迟更低。  
> 第三问是架构师思维：Redis 挂了 ≠ 本地缓存一定脏。可以在本地缓存的 Value 里藏一个 **版本号（或最后更新时间戳）**，Redis 恢复后，对比本地版本号和 Redis 版本号，不一致就失效本地缓存。或者更简单：**本地缓存设短 TTL（如 30s）**，Redis 挂了也能靠 TTL 自然过期收敛。

### 第11题（源码级）

> **Redisson 的看门狗（Watchdog）机制，面试常问但很多人说不清细节。**
>
> 1. 看门狗是守护线程吗？它是在哪个线程池里跑的？  
> 2. 如果业务线程执行了 15s，看门狗已经续期了 2 次（每 10s 一次），然后业务线程宕机了，锁会永远存在吗？  
> 3. 如果 Redis 主从切换期间，看门狗往新主节点续期，但新主节点上没有这个锁（主从同步延迟），会发生什么？

**【考察方向】**
- 第一层：Redisson Watchdog 的实现（Netty HashedWheelTimer，不是每个锁一个线程）
- 第二层：Watchdog 的活性保证 vs 安全性边界（客户端死亡 = 续期停止 = TTL 兜底）
- 第三层：Redis 主从切换下的锁丢失问题（RedLock 的争议、业务幂等兜底）

**【引导提示】**
> 第一问：Redisson 用 **Netty 的 HashedWheelTimer**（时间轮），不是守护线程池。一个 HashedWheelTimer 管理所有锁的续期任务，避免线程爆炸。  
> 第二问：不会永远存在。Redisson 的锁有 **TTL（默认 30s）**，看门狗只是续期。如果业务线程宕机（JVM 挂了），看门狗线程也死了，不再续期，30s 后锁自动释放。**看门狗解决的是活性问题（业务还没完锁就过期），不是安全性问题（客户端挂了怎么办）。**  
> 第三问是分水岭：主从切换时锁丢失是 Redis 分布式锁的**固有缺陷**。看门狗续期失败（新主没有锁 key），Redisson 的 `renewExpiration()` 方法会返回 0，业务线程应该感知到锁丢失。但 Redisson 默认不会抛异常，只是停止续期。**正确做法**：业务代码在执行业务前校验锁是否仍持有，或采用 **RedLock**（多主节点），但 RedLock 有 Martin Kleppmann 的争议。

## 七、分布式基础

### 第12题（方案设计）

> **你们有一个下单后扣库存的场景，需要保证：下单成功 → 扣库存成功，且最终一致。你会怎么设计？**
>
> 1. 如果用本地消息表方案，消息表和业务表必须在同一个数据库里吗？为什么？  
> 2. 如果库存服务消费消息失败了，重试 5 次后仍然失败，你会怎么处理？直接回滚订单吗？  
> 3. 如果让你从本地消息表升级到 Seata AT，你会评估哪些利弊？什么场景下坚决不用 Seata？

**【考察方向】**
- 第一层：本地消息表的核心设计（同库事务、异步投递、幂等消费）
- 第二层：死信队列（DLQ）+ 人工介入 + 自动补偿（关单退款）
- 第三层：Seata AT 的侵入性（代理数据源、undo_log、全局锁）、性能损耗、与本地消息表的 trade-off

**【引导提示】**
> 第一问：**必须同库**。本地消息表的本质是利用**本地事务的 ACID** 保证"业务操作 + 消息记录"原子性。如果分库，无法保证原子性，方案就退化了。  
> 第二问：重试 5 次后进入 **DLQ（死信队列）**，触发告警。处理方式分两种：**系统异常**（网络抖动）可以人工修复后重投；**业务异常**（库存不足）应该走补偿流程（自动关单 + 退款 + 短信通知用户）。**不要无限重试**，否则消息堆积会拖垮系统。  
> 第三问是架构师权衡：Seata AT 的优点是**对业务代码无侵入**（自动代理数据源），缺点是**性能损耗**（全局锁、undo_log、TC 协调器单点）。**坚决不用 Seata 的场景**：超高并发（> 1w TPS）、长事务（> 30s）、跨异构数据库（MySQL + Oracle）。本地消息表更轻量、更可控。

### 第13题（系统设计）

> **设计一个秒杀系统的库存扣减模块，要求：高并发、不超卖、用户体验好。**
>
> 1. 你会用 Redis 做预扣库存还是直接走数据库？Redis 扣减后，怎么保证最终和数据库一致？  
> 2. 如果 Redis 扣减成功，但用户最终没支付，库存怎么回退？回退的时效性怎么保证？  
> 3. 如果秒杀流量是平时的 1000 倍，除了缓存和异步，你在架构层面（不是代码层面）还有什么手段？

**【考察方向】**
- 第一层：Redis 预扣（DECR）+ Lua 脚本保证原子性，异步同步数据库
- 第二层：库存回退（延时队列 / TTL Key / 定时任务），支付超时释放
- 第三层：架构层面（库存拆分、分桶竞争、热点隔离、流量错峰、CDN 静态化）

**【引导提示】**
> 第一问：Redis 做预扣（`Lua: if stock > 0 then decr else return 0 end`），**异步消息队列**（RocketMQ/Kafka）削峰后同步到数据库。一致性靠**最终一致**，数据库是兜底。  
> 第二问：回退用 **RocketMQ 延迟消息**（18级延迟，如 15min 后未支付自动释放），或 Redis **TTL Key**（设置库存 key 15min 过期，过期时监听 Keyspace Notification 回退）。  
> 第三问是架构分水岭：**库存分桶**（100个库存桶，用户 hash 到不同桶，分散热点）、**流量错峰**（验证码/预约机制，把 1s 的流量摊到 10min）、**CDN + 静态化**（秒杀页面静态化，不走后端）、**答题机制**（人为制造漏斗，过滤刷单）。

## 八、综合设计题（跨模块串联）

### 第14题（架构师级）

> **你们要设计一个"实时用户行为分析平台"，需求如下：**
>
> - 接入 APP/小程序/Web 的点击、浏览、下单事件，日活 1000万，峰值 QPS 10万
> - 实时计算每个用户的实时标签（如"最近30天浏览了3次手机"）
> - 运营后台可以实时查询用户标签，用于精准营销
> - 标签数据可以容忍秒级延迟
>
> 1. 数据采集到存储的完整链路你会怎么设计？（从客户端到数据库）  
> 2. 实时标签计算用 Flink 还是 Spark Streaming？为什么？如果团队不会 Flink，你会怎么妥协？  
> 3. 用户标签存储用 MySQL、Redis、还是 HBase？如果运营后台要按标签组合查询（如"近7天浏览手机且未下单的用户"），存储层怎么设计？

**【考察方向】**
- 第一层：高并发数据采集（客户端埋点 → Kafka → Flink/Spark → 存储）
- 第二层：流处理选型（Flink 的 Exactly-Once、低延迟 vs Spark Streaming 的微批、高吞吐）
- 第三层：标签存储设计（HBase 宽表 / Redis Bitmap / Elasticsearch / ClickHouse，按查询场景选型）

**【引导提示】**
> 第一问：客户端埋点 → **OpenResty/Nginx** 接收（本地聚合）→ **Kafka**（削峰）→ **Flink**（窗口聚合）→ **HBase/ClickHouse**（存储）。不要直接打日志到文件再收集，延迟太高。  
> 第二问：Flink 是**事件驱动**（来一条处理一条，延迟 < 1s），Spark Streaming 是**微批**（几百 ms ~ 几秒一批）。实时标签计算要求低延迟，**首选 Flink**。如果团队不会 Flink，妥协方案：**Kafka Streams**（轻量，Java 友好）或 **Flink 的 SQL API**（降低学习成本）。  
> 第三问是存储选型分水岭：
> - **HBase**：宽表设计，适合写密集、按用户ID查，但不适合多条件组合查
> - **Redis Bitmap**：适合简单标签（0/1），位运算快，但标签维度多时报废
> - **ClickHouse**：列式存储，适合 OLAP 复杂查询，写入吞吐高，**推荐**
> - **Elasticsearch**：倒排索引，适合文本检索，但聚合性能不如 ClickHouse
>
> 推荐方案：**HBase 存明细**（用户ID → 行为序列）+ **ClickHouse 存聚合标签**（用户ID → 标签向量），运营查询走 ClickHouse。
