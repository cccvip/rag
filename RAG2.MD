# 检索优化：BM25 vs Dense vs Hybrid；Re-ranker 原理；ColBERT 晚期交互

---

## 1 BM25（Best Match 25）

### 原理

BM25 是经典的信息检索排序函数，基于 TF-IDF 做了两处关键改进：

```
score(Q, D) = Σ IDF(q_i) · [ f(q_i, D) · (k1 + 1) ] / [ f(q_i, D) + k1 · (1 - b + b · |D| / avgDl) ]
```

**三个核心设计**：

| 设计 | 作用 | 直观理解 |
|------|------|---------|
| **TF 饱和** | 词频增加到一定阈值后贡献递减 | "充电桩"出现 10 次和 100 次，相关性差距不大 |
| **文档长度归一化** | 长文档天然词频高，需要惩罚 | 防止长篇大论占便宜 |
| **IDF 降权常见词** | 在越多文档中出现的词，区分度越低 | "的"、"是"这类词几乎不得分 |

### BM25 的本质

> **精确的词项匹配**。它只关心 query 中的词是否在文档中出现，以及出现的频率和分布。

**优点**：
- 零训练成本，开箱即用
- 对专业术语、ID、代码、人名、型号等**精确匹配**效果极好
- 可解释性强，能告诉用户"因为文档里出现了这些词"

**缺点**：
- 无法理解**同义词和语义变体**
  - "充电桩安装费用" 和 "EV charger installation cost" 在 BM25 眼里完全无关
- 对**长尾查询**和**口语化表达**效果差
- 没有上下文理解能力

## 2 Dense Retrieval（稠密向量检索）

### 原理

用深度神经网络（Transformer-based Encoder）把 query 和 document 分别编码成固定维度的稠密向量，通过**向量空间中的距离**（余弦相似度或内积）来判断相关性。

```
Query:  "充电桩安装费用"   → Encoder → [0.12, -0.05, 0.33, ..., 0.89]  (768维)
Doc:    "2024年家用充电桩安装收费标准..." → Encoder → [-0.03, 0.11, ..., 0.92]

Similarity = cos(v_q, v_d) = 0.87
```

### 为什么 Dense 能找到 BM25 找不到的东西？

| 用户查询 | 文档中的表达 | BM25 | Dense |
|---------|-------------|------|-------|
| "充电桩安装费用" | "家用充电设备报装价格" | ❌ 词完全不重叠 | ✅ 语义相近 |
| "电动车充电贵不贵" | "7kW 充电桩每度电 0.5 元" | ❌ | ✅ |
| "特斯拉 supercharger" | "特斯拉超级充电桩" | ❌ | ✅ |

**优点**：
- 语义泛化能力强，解决**词汇鸿沟**问题
- 对同义词、近义词、口语化表达天然友好
- 训练好的模型可以跨语言检索

**缺点**：
- 需要训练/微调，领域外效果可能崩
- 对**精确术语、ID、型号、代码**的匹配不如 BM25
  - "Model-X-2024" 和 "Model-Y-2024" 在向量空间里可能非常接近
- 向量构建和存储有额外成本

## 3 Hybrid Search（混合检索）

### 为什么需要混合？

| 查询类型 | BM25 | Dense |
|---------|------|-------|
| "第 3.2 条规定了什么" | ✅ 精确命中章节号 | ⚠️ "3.2"语义信息弱 |
| "充电桩安装费用" | ❌ 需词完全匹配 | ✅ 语义泛化强 |
| "GB/T 18487.1 标准" | ✅ 型号精确匹配 | ❌ 可能和 18487.2 混淆 |
| "充电贵不贵" | ❌ 口语化 | ✅ 理解意图 |

> **BM25 和 Dense Retrieval 是互补的，不是互斥的。**

### 融合策略

#### 策略 1：RRF（Reciprocal Rank Fusion）— 最常用

```
score = Σ 1 / (k + rank_i)

k = 60（经验值，防止低排名项得分过高）
```

**示例**：

```
BM25 召回:   [Doc-A, Doc-C, Doc-B, Doc-D]  → ranks: A=1, C=2, B=3, D=4
Dense 召回:  [Doc-B, Doc-A, Doc-E, Doc-C]  → ranks: B=1, A=2, E=3, C=4

Doc-A: 1/(60+1) + 1/(60+2) = 0.0325
Doc-B: 1/(60+3) + 1/(60+1) = 0.0323
Doc-C: 1/(60+2) + 1/(60+4) = 0.0317
Doc-E: 0         + 1/(60+3) = 0.0159

最终: Doc-A > Doc-B > Doc-C > Doc-E
```

**RRF 的优点**：
- 无需调参，不需要给 BM25 和 Dense 分配权重
- 对各路召回的分数分布不敏感（不需要归一化）
- 鲁棒性极强，工程上最省心

#### 策略 2：加权分数融合

```
final_score = α · score_BM25 + (1-α) · score_Dense
```

需要调 α，对数据分布敏感，不推荐作为默认方案。

### 一句话总结

> **BM25 是"查字典"——精确、快速、无训练；Dense Retrieval 是"联想记忆"——语义泛化、理解意图；Hybrid Search 是"左右脑并用"——用 RRF 把两者的长处结合起来，是生产环境的最优默认选择。**

## 4 Re-ranker：Cross-Encoder 为什么比 Bi-Encoder 更准但更慢

### 4.1 架构差异

**Bi-Encoder（双塔）**：
```
Query → Encoder → v_q (768维)
Doc   → Encoder → v_d (768维)  ← 离线预计算，只算一次

相似度 = cos(v_q, v_d) = 点积
```

**Cross-Encoder（交互）**：
```
Query: "充电桩安装费用"
Doc:   "2024年家用充电桩安装标准..."

拼接: [CLS] 充电桩安装费用 [SEP] 2024年家用充电桩安装标准... [SEP]
      ↓
[Transformer × 12层]  ← 每层 Query 的每个 token 都要 attend 到 Doc 的每个 token
      ↓
[CLS] → MLP → 相关性得分
```

### 4.2 为什么更准？

Bi-Encoder 的问题：**Q 和 D 在编码时彼此不可见**。

```
Bi-Encoder:
  "充电桩" 在 Query 中的向量 = f("充电桩", 上下文只有 Query)
  "充电桩" 在 Doc 中的向量   = g("充电桩", 上下文只有 Doc)
  → 两个向量是独立计算的，信息损失大

Cross-Encoder:
  "充电桩" 在 concat(Q,D) 中的表示 = h("充电桩", 能看到完整的 Q 和 D)
  → 注意力机制捕获 "充电桩" 与 "安装"、"费用"、"标准" 的细粒度关系
  → 能处理 "not"、"but" 等否定和转折语义
```

### 4.3 为什么更慢？

| 指标 | Bi-Encoder | Cross-Encoder |
|------|-----------|---------------|
| Query 编码 | 1 次 | N 次（每对一次） |
| Doc 编码 | 1 次（离线预计算） | N 次（每次重新算） |
| Attention | Q 和 D 各自独立 | Q 和 D 的 token 两两交互 |
| 复杂度 | O(\|Q\|) + O(1) 点积 | O(N · \|Q+D\|² · d) |
| 100万文档 | ~10ms | ~数小时 |

**核心原因**：Cross-Encoder 不可预计算。

> 文档中 "充电桩" 的表示，取决于它搭配的是 "安装费用" 还是 "充电速度"——这个语义是 Query-Dependent 的，所以每对 (Q,D) 都要重新过模型。

### 4.4 工程解法：两阶段架构

```
Stage 1: Bi-Encoder 召回（粗筛）
  目标：从 100万 → 100
  耗时：~10ms

Stage 2: Cross-Encoder 精排（细筛）
  目标：从 100 → 5
  耗时：~500ms

Bi-Encoder 解决"找得到"，Cross-Encoder 解决"找得准"。
```

## 5 ColBERT 的晚期交互机制（Late Interaction）

### 5.1 问题背景

Bi-Encoder 太快但太粗，Cross-Encoder 太准但太慢。有没有折中方案？

ColBERT 的回答是：**预计算文档的 token 级向量，查询时只做轻量级的 token 交互。**

### 5.2 原理

```
文档编码（离线预计算）:
  Doc: "家用充电桩安装费用"
    ↓
  每个 token 单独编码：
    "家" → [0.12, -0.05, ...]   (128维)
    "用" → [0.08, 0.33, ...]
    "充" → [0.35, 0.11, ...]
    "电" → [0.30, -0.02, ...]
    ...
  → 存储：Doc 的 token 向量矩阵 (doc_len × 128)

查询编码（在线）:
  Query: "充电桩费用"
    ↓
  每个 token 编码：
    "充" → [0.40, 0.15, ...]
    "电" → [0.38, 0.12, ...]
    "桩" → [0.45, 0.08, ...]
    "费" → [0.42, 0.20, ...]
    "用" → [0.10, 0.05, ...]

相似度计算（Late Interaction）:
  对 Query 的每个 token，在 Doc 的 token 中找最大相似度（MaxSim）:
    "充" → max(cos(充_q, 家_d), cos(充_q, 用_d), cos(充_q, 充_d), ...) = 0.95
    "电" → ... = 0.93
    "桩" → ... = 0.88
    "费" → ... = 0.91
    "用" → ... = 0.85
  
  最终得分 = sum(所有 MaxSim) = 0.95 + 0.93 + 0.88 + 0.91 + 0.85 = 4.52
```

### 5.3 为什么叫"晚期交互"？

| 阶段 | Bi-Encoder | ColBERT | Cross-Encoder |
|------|-----------|---------|---------------|
| 编码阶段 | Q/D 完全独立 | Q/D 独立，但保留 token 级向量 | Q/D 拼接后联合编码 |
| 交互阶段 | **无交互**，直接点积 | **Late**：token 级 MaxSim | **Early**：每层 attention 都交互 |
| 计算位置 | 最后一步 | 检索时 | 全程 |

> ColBERT 把交互推迟到了**检索阶段**，而不是编码阶段。这样既保留了预计算的能力，又在最后一步做了细粒度匹配。

### 5.4 性能对比

| 模型 | 精度 | 速度 | 存储 |
|------|------|------|------|
| Bi-Encoder | ⭐⭐ | ⭐⭐⭐⭐⭐ 极快 | 小（1个向量/文档） |
| ColBERT | ⭐⭐⭐⭐ 接近 CE | ⭐⭐⭐⭐ 快 10~100 倍 | 大（N个向量/文档，N=文档长度） |
| Cross-Encoder | ⭐⭐⭐⭐⭐ 最准 | ⭐⭐ 慢 | 无需预存 |

**代表模型**：`BAAI/bge-m3`（支持 ColBERT 模式）

### 5.5 一句话总结

> **ColBERT 是 Bi-Encoder 和 Cross-Encoder 的折中：编码阶段各自独立（可预计算），检索阶段做 token 级 Late Interaction（精度接近 CE）。代价是存储量大幅增加——每个文档存的是 token 向量矩阵，而非单个向量。**

## 6 补充：神经网络稀疏向量 SPLADE

SPLADE 是 BM25 和 Dense 的中间态，用神经网络生成稀疏向量。

```
输入文档："家用充电桩安装费用"
  ↓
过 BERT → 对每个词在词汇表上的 logits 做 expansion
  ↓
输出稀疏向量：
  充: 0.45, 电: 0.40, 桩: 0.50,
  设: 0.15, 备: 0.12,           ← 自动扩展出"设备"相关词
  费: 0.55, 用: 0.20, 安: 0.30, 装: 0.28
  ...

→ 非零维度仍然很少（几十个），但比传统 BM25 多
→ 能自动扩展同义词和相关词
→ 索引结构仍是倒排索引
```

| 特性 | BM25 | SPLADE | Dense |
|------|------|--------|-------|
| 向量维度 | 词汇表大小 | 词汇表大小 | 768/1024 |
| 非零元素 | 文档中出现过的词 | 几十个扩展词 | 全部非零 |
| 同义词扩展 | ❌ | ✅ | ✅ |
| 训练需求 | ❌ | ✅ | ✅ |
| 索引结构 | 倒排索引 | 倒排索引 | ANN 图/簇 |
| 精确匹配 | ✅ 极强 | ✅ 强 | ⚠️ 较弱 |

## 总结

| 技术 | 核心定位 | 速度 | 精度 | 适用 |
|------|---------|------|------|------|
| BM25 | 精确词项匹配 | 极快 | 低（语义弱） | 法规、代码、ID |
| Dense | 语义压缩匹配 | 快 | 中 | 口语化、跨语言 |
| Hybrid | 两者互补 | 快 | 高 | **生产环境默认** |
| SPLADE | 神经稀疏扩展 | 快 | 中高 | 想兼顾精确和泛化 |
| Bi-Encoder | 召回粗筛 | 极快 | 低 | Stage 1 |
| Cross-Encoder | 精排细筛 | 慢 | 极高 | Stage 2 |
| ColBERT | 晚期交互折中 | 快 | 高 | 存储可接受时优选 |
