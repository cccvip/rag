更熟悉G1,线上的业务场景以G1为主。
G1是物理上的分区,分为连续多个Region。
Region又细分为正常大小的Region以及humongous Region，用于存放大对象。
大对象是直接进入老年代。

内存结构
1 Region：物理上不再连续区分 Eden/Survivor/Old/Humongous，逻辑上仍存在代际
2 Humongous Region：专门存放大对象（> 0.5 个 Region），直接进老年代，可能成为内存碎片和 GC 瓶颈
3 Card Table + Remembered Set（RSet）：
RSet：每个 Region 维护一个，记录哪些其他 Region 的对象引用了本 Region 的对象
作用：Young GC 时不用扫描整个 Old 区，只需扫描 RSet，实现跨代引用的快速定位
Card Table：辅助 RSet，记录内存页（512B）的引用变化

Young GC的时候回收Eden区, 回收年轻代的Region
Mixed GC: 当老年代达到阈值的时候,并发标记回收（年轻代+部分老年代）
Full Gc: 单线程回收, 老年代资源不够存放。由于是单线程下STW耗时可能会很长

Humongous Region不仅仅是一个大对象,而且 分配策略、回收时机、内存布局"完全不同的一类 Region。

Spring 通过三级缓存解决单例 Bean 的 Setter/Field 注入循环依赖。当一个 Bean 完成实例化（构造函数执行完、对象已在堆上创建）但还未完成属性注入和初始化时，Spring 会通过第三级缓存 singletonFactories 中的 ObjectFactory 提前暴露一个引用（必要时生成 AOP 代理），放入二级缓存 earlySingletonObjects。这样当循环依赖的另一方需要注入它时，能拿到这个半成品引用，避免死循环。但如果是构造器注入，因为对象还未实例化，无法提前暴露，Spring 是解决不了的，需要改用 @Lazy 或重构代码。

18. MySQL的RR隔离级别下，幻读到底解决了没有？如果有一个场景：事务A先查询id>100的记录（返回10条），事务B插入了一条id=105的记录并提交，事务A再次查询id>100，结果是什么？InnoDB是怎么解决这个问题的？间隙锁在实际业务中带来过死锁问题吗？

> **参考意见**：

"InnoDB 的 REPEATABLE READ 没有完全解决幻读。它的 MVCC 机制可以保证纯快照读（普通 SELECT）不出现幻读，因为读取的是事务开始时的 ReadView 一致性快照。但一旦事务中出现了当前读（FOR UPDATE、UPDATE、DELETE），就会读取最新已提交数据，幻读就可能发生。

如果有一个场景：要求严格保证消息的顺序性，同时又要高吞吐，你会怎么设计？

1 性能的既要又要,但拆解问题可以分为两点
如果全局有序,并且还要高吞吐的时候,那么最终退化为单进程一个分区的List。
因此顺序性就必须是相对的,保证高吞吐,需要引入多分区的概念。

首先，生产者端用业务 Key（如订单号）做哈希路由，保证同一个 Key 的消息始终进入同一个 Partition。Partition 内部是追加写日志，天然有序。

其次，消费者端一个 Partition 只能被一个线程顺序处理，避免多线程乱序。如果消费逻辑有 IO 阻塞，我会在消费者内部用内存队列 + Key 哈希分发，做到线程间并行、线程内串行。

最后，为了提升吞吐，我会用批量发送减少网络 RTT，零拷贝降低 CPU 开销，消息压缩减少带宽。副本同步保证 ISR 内的顺序一致性。

这样设计的吞吐可以随分区数线性扩展，同时保证业务关心的 Key 级别有序。如果是金融撮合这种必须全局有序的场景，只能退化成单队列，靠硬件加速扛吞吐。"
