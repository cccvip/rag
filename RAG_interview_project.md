# 应急安全管理 RAG 项目 — 面试口述稿

> 基于应急安全领域（消防/逃生/巡检告警）的真实业务场景，总字数约 2600 字，口述语速 180 字/分钟，全程约 14~15 分钟。
> **加粗部分为必须记住的记忆锚点和关键数字。**

---

## 开场白（20 秒）

我介绍一下我之前做的 RAG 项目。这是一个**应急安全管理知识库系统**，服务对象是园区和楼宇的安全管理员、一线巡检人员。核心场景是紧急查询，比如"3 楼 A 区发生火灾，最近的逃生通道在哪"。这个场景下，**延迟不是性能问题，是安全问题**，答错了可能出人命。

---

## 一、项目背景与规模（1 分钟）

数据源有三类。

第一类是**静态文档**，安全操作手册、消防规范、逃生通道平面图、应急预案，大概 **5 万份**，PDF 和扫描件居多。

第二类是**实时上报数据**，IoT 传感器告警、人员签到记录、巡检上报的隐患描述，每天新增 **10 万条**。

第三类是**空间资产数据**，楼层信息、区域编号、逃生通道坐标、消防设施位置，存在 MySQL 里。

我在团队里负责整个 **RAG 检索链路和数据接入层**。团队 6 个人，我是技术负责人。核心目标三个：**端到端延迟 P99 小于 1.5 秒**；**精确信息准确率 100%**；**召回率大于 95%**。

---

## 二、技术架构（2.5 分钟）

我们的链路设计成了**"结构化查询 + RAG 文本检索"双轨并行**。

用户问"3 楼 A 区火灾逃生路线"，先进 Query Router 做意图识别和实体提取。提取出楼层等于 3、区域等于 A、事件等于火灾。然后分成两路走：

第一路是**结构化路径**，用 SQL 查空间资产库，拿到逃生通道坐标、消防设施位置。这一路负责**精确约束**，楼层号、区域号、设备编号绝对不能错。

第二路是**RAG 文本路径**，走 Hybrid Search，BM25 加 Dense，从安全手册里召回"火灾疏散""逃生通道"相关条款。这一路负责**语义理解**，比如用户说"着火了怎么办"这种口语化表达。

两路结果合并，做 Prompt 组装，然后 LLM 生成，最后做引用溯源和地图标注。

几个关键选型我想重点说。

**查询重写**：我们用了实体提取加规则层为主。应急查询里的"3 楼""A 区""灭火器"这些词必须精确匹配，HyDE 会引入不确定性。所以只在开放式问答，比如"火灾怎么处理"，才启用 HyDE。

**检索**：**必须用 Hybrid Search**。BM25 负责精确匹配楼层号、区域号、设备编号。Dense 负责理解口语化表达。融合用 RRF，k 取 60。

**分块**：安全手册按"章节加条款"分块，**chunk_size 256，overlap 20%**。逃生通道平面图先做 OCR 提取文字描述，再单独建块。

**索引**：Milvus，HNSW 索引。楼层、区域、坐标作为标量过滤条件。

**Re-rank**：**bge-reranker-base**，速度和精度的平衡。

**LLM**：GPT-4，**temperature 0**。应急场景下不允许创意发挥，必须严格基于资料。

---

## 三、核心数字（1.5 分钟）

系统上线跑了 3 个月，核心数字我列一下。

**QPS**：峰值 **180**，日常 60，演练的时候会冲到 180。

**端到端延迟 P99**：**1.2 秒**。拆解开：检索链路 200 毫秒，LLM 生成 900 毫秒。检索里 BM25 50 毫秒，HNSW 80 毫秒，Cross-Encoder 精排 70 毫秒。

**Recall@50**：**96%**。这是加了 Hybrid Search 和实体过滤之后的结果。

**Faithfulness**：**0.88**。

**精确信息准确率**：**99.5%**。这是人工抽检 200 条的结果，楼层、区域、设备号这些关键信息不能错。

**用户满意度**：点赞率 **84%**。

特别说一下延迟。**1.2 秒的 P99 在应急场景里算合格，但目标是压到 800 毫秒**。我们做了两件事：一是把逃生通道这类高频查询做了 **Redis 缓存**，命中率 55%。二是对 LLM 做了 **Prompt 缓存**，相同类型的应急 query 复用已生成的结构化模板。

---

## 四、最头疼的 Bad Case（4 分钟）

我挑两个最典型的，一个 Retrieval 问题，一个 Generation 问题。

### Bad Case 1：Retrieval 问题 —— 逃生通道图检索失败

**场景**：巡检员问"B2 层停车场逃生通道在哪？"系统回答了 B1 层的通道位置。

**排查**：先看检索回来的 Top 5，发现 5 个里有 3 个是 B1 层的。确认是 Retrieval 问题。

**根因有两个**。第一，**逃生通道平面图是扫描件**，OCR 提取后的文字是"B2 停车场逃生通道示意图，出口指向 B1 层大堂"。Dense Embedding 把"B2"和"B1"的向量学得很近，因为都是楼层词。BM25 也没精确过滤楼层。第二，query 里没有显式提到"停车场"，巡检员只说了"B2 层"，但系统默认 B2 是停车场，这个语义关联没建立。

**修复动作**：三步。第一步，**标量过滤强化**，把楼层号、区域号从文本中提取出来，作为 Milvus 的标量字段，检索时先做 floor 等于 2 的过滤，再走向量检索。第二步，**文档预处理升级**，逃生通道图不再只做 OCR，而是用多模态模型提取图中的楼层标注、箭头指向、文字说明，生成结构化文本描述后再入向量库。第三步，**同楼层文档 Boost**，Re-rank 阶段同楼层的文档分数乘以 1.2 倍。

**结果**：该 query 类型的准确率从 72% 提到 **98%**，类似"楼层混淆"的 case 减少了 **80%**。

### Bad Case 2：Generation 问题 —— 遗漏关键安全步骤

**场景**：用户问"化学品泄漏怎么处理？"系统回答了"疏散人员、封锁区域"，但**遗漏了"必须佩戴防毒面具"和"向上风向疏散"**这两个关键安全步骤。

**排查**：检查检索回来的 chunks，发现 Top 3 里明明有《化学品泄漏应急预案》的第 3.2 条"操作人员须佩戴防毒面具"。确认是 Generation 问题，上下文够但 LLM 没答全。

**根因**：第一，**上下文过长**，Top 5 chunks 塞了 3000 tokens，安全步骤散落在不同 chunk 里，LLM 漏掉了靠后的关键项。第二，**Prompt 约束不够**，没有要求 LLM"列出所有必须步骤，缺一不可"。

**修复动作**：三步。第一步，**重排序优化**，CE 阶段从 Top 5 改取 Top 3，减少噪音，同时把"化学品泄漏"相关的手册条款强制插入 Top 3。第二步，**Prompt 模板化**，应急场景的 Prompt 不再自由生成，而是要求按"发现 → 报告 → 处置逐项列出 → 疏散 → 善后"的固定结构输出。第三步，**后处理校验**，加了一层安全关键词校验——如果 query 涉及"化学品""火灾""触电"，但答案里没出现对应的关键防护词，比如防毒面具、灭火器、绝缘手套，**自动触发二次检索**，或者降级到"请查阅完整手册"。

**结果**：关键安全步骤遗漏率从 15% 降到 **3%**，Faithfulness 从 0.75 提到 **0.91**。

---

## 五、当前不足与后续规划（1.5 分钟）

现在系统还有三个明确的问题在排期。

**第一，实时数据融合不够深**。IoT 告警，比如"3 楼烟雾传感器触发"，和知识库检索目前是割裂的。用户得先看到告警，再去问系统。我们规划做**实时事件驱动的 RAG**，告警触发时自动预检索相关预案，用户一问直接返回，而不是等 query 来了再检索。

**第二，多模态理解弱**。逃生通道图、消防设施照片的理解还是靠 OCR。规划引入**视觉语言模型**直接理解图纸中的箭头、标识、距离。

**第三，离线场景兜底**。网络断了怎么办？规划把高频预案和逃生路线**预生成静态卡片**，同步到移动端本地，断网也能查。

另外我们也在做**模型蒸馏**，用 GPT-4 的输出训练本地 Qwen-14B，目标是 6 个月后切换到本地模型，降低延迟和成本。

---

## 六、收尾（20 秒）

总结下来，这个项目的核心挑战不是技术复杂度，而是**安全合规和延迟的双重约束**。我们的解法可以概括为：**SQL 做精确约束，RAG 做语义理解，Hybrid Search 做召回，结构化模板做生成兜底**。关键安全步骤遗漏率压到了 3%，精确信息准确率 99.5%，P99 延迟 1.2 秒。

## 附录：高频追问 & 标准应答

**Q：应急场景延迟要求这么高，为什么还用 GPT-4 不用本地模型？**
> 考虑过，但应急场景容错率极低，GPT-4 的指令遵循和事实准确性明显优于 7B、13B 本地模型。我们在做模型蒸馏，用 GPT-4 的输出训练本地 Qwen-14B，目标是 6 个月后切换。

**Q：楼层号、区域号这种结构化信息，为什么不直接走 SQL 查询？**
> 我们在走 SQL，但不是纯 SQL。楼层和区域是标量过滤条件，WHERE floor 等于 3 AND area 等于 A 区。但用户的 query 是"3 楼 A 区着火了怎么办"，"怎么办"这部分必须走 RAG。所以是 SQL 过滤加 RAG 检索的混合架构。

**Q：5 万份文档，向量库为什么选 Milvus 不是 pgvector？**
> pgvector 千万级以内够用，但我们有两类数据，5 万文档加每天 10 万条实时上报。Milvus 的分布式能力和标量过滤性能更好，而且原生支持 Hybrid Search，Sparse 加 Dense。我们实际是用 ES 做 BM25，Milvus 做 Dense，架构上更清晰。

**Q：应急场景的 Faithfulness 0.88 够吗？为什么不追求 0.95？**
> 0.88 是自动化指标，我们真正的红线是"关键安全步骤遗漏率小于 5%"，这个目前做到了 3%。Faithfulness 对"步骤是否完整"的敏感度不够高，所以我们在用结构化输出模板做补充校验，比单纯追 Faithfulness 更落地。

**Q：如果系统推荐了错误的逃生通道，怎么兜底？**
> 三层兜底。第一层，检索结果必须包含引用来源，用户可点击查看原始手册。第二层，后处理校验，如果检索结果里逃生通道的楼层和 query 楼层不一致，触发告警并返回"请核实楼层信息"。第三层，页面强制显示"本信息仅供参考，紧急情况下请遵循现场人员指挥"。

**Q：如果网络断了怎么办？**
> 这是我们正在做的离线兜底方案。高频预案和逃生路线会预生成静态卡片，同步到移动端本地，断网也能查。同时关键安全信息会在建筑现场的物理标识牌上同步更新，系统只是辅助手段。

**Q：演练时 QPS 冲到 180，怎么保证不崩？**
> 三个措施。一是 Redis 缓存，高频查询命中率 55%。二是 LLM Prompt 缓存，同类应急 query 复用模板。三是做了降级预案，极端情况下可以跳过 CE 精排，直接用 Bi-Encoder 的 Top 5 送给 LLM，精度掉 3% 但延迟降一半。
