# synchronized 锁升级与偏向锁撤销

## 一、锁升级的完整路径

```
无锁（New）          无竞争
    ↓ 第一个线程获取锁
偏向锁（Biased）      单线程反复进入
    ↓ 第二个线程尝试获取
轻量级锁（Lightweight） 低竞争、短持有
    ↓ 自旋多次失败 / 调用wait
重量级锁（Heavyweight） 激烈竞争、长持有
```

**核心原则**：锁只能**单向升级**（除了偏向锁可以撤销后重新偏向），不会降级。

---

## 二、各状态详解

### 1. 无锁（Unlocked）

- 对象刚创建时的初始状态
- Mark Word 中：偏向锁位 `0`，锁标志位 `01`
- 如果 JVM 开启了偏向锁延迟（默认延迟 4 秒），新创建的对象会先处于**匿名偏向**状态（偏向锁位 `1`，但线程 ID 为空）

### 2. 偏向锁（Biased Locking）

**设计目的**：消除无竞争情况下的 CAS 开销。

**获取过程**：
1. 线程 A 第一次进入 `synchronized` 块
2. CAS 将对象头 Mark Word 中的线程 ID 替换为自己的线程 ID
3. 后续线程 A 再次进入，只需检查 Mark Word 中的线程 ID 是否等于自己 —— **不需要 CAS**，开销接近无锁

> ⚠️ **JDK 15 默认禁用偏向锁，JDK 18 正式移除**。原因是现代应用多线程竞争普遍，偏向锁的维护成本（Safepoint 撤销）高于收益。面试时提到这一点是加分项。

### 3. 轻量级锁（Lightweight Locking）

**触发条件**：有第二个线程尝试获取该对象的偏向锁，或偏向锁已撤销。

**获取过程**：
1. 线程在当前栈帧中创建 **Lock Record**（锁记录）
2. 将对象头的 Mark Word **复制**到 Lock Record 的 `Displaced Mark Word` 中
3. CAS 将对象头的 Mark Word 替换为指向该 Lock Record 的指针
   - **成功**：获得轻量级锁，锁标志位变为 `00`
   - **失败**：说明存在竞争，进入自旋

**自旋优化**：
- 线程不会立即阻塞，而是**自旋**（忙等待）几次
- 自旋次数默认 10 次（可通过 `-XX:PreBlockSpin` 调整）
- JDK 6 后引入**自适应自旋**：JVM 根据历史成功率动态调整自旋次数

### 4. 重量级锁（Heavyweight Locking）

**触发条件**：轻量级锁自旋多次仍然失败。

**原理**：
- 对象头的 Mark Word 指向一个 **ObjectMonitor**（C++ 实现的监视器对象）
- 未获取到锁的线程进入 **EntryList**（阻塞队列），调用 `pthread_mutex_lock` 进入内核态挂起
- 涉及**用户态 ↔ 内核态**的上下文切换，开销最大（约 100 个时钟周期以上）

## 三、偏向锁的撤销（重点）

偏向锁不是"升级"，而是**撤销**（Revoke）。撤销后对象可能变为**无锁**或**轻量级锁**状态。

### 情况一：有其他线程尝试获取锁（最常见）

```
线程A（已偏向）          线程B（新来者）
    │                        │
    │    发现对象已偏向A      │
    │ ◄──────────────────────┤
    │                        │
    │    请求全局安全点        │
    │    撤销偏向锁            │
    │ ◄──────────────────────┤
```

**具体流程**：
1. 线程 B CAS 失败，触发偏向锁撤销
2. JVM 等待**全局安全点（Safepoint）**（STW，暂停所有线程）
3. 检查持有偏向锁的线程 A 是否还在同步代码块中：
   - **已退出**：直接撤销偏向锁，恢复为无锁状态，然后线程 B 竞争轻量级锁
   - **仍在执行**：暂停线程 A，将锁升级为轻量级锁，然后线程 B 竞争

> 这个 Safepoint 等待是偏向锁被移除的主要原因之一 —— 撤销成本不可控。

### 情况二：调用了对象的 `hashCode()`

**根本原因**：Mark Word 空间不够。

64 位 JVM 的 Mark Word 布局：

```
| 锁状态   | 存储内容                          | 偏向锁位 | 锁标志 |
|---------|----------------------------------|---------|--------|
| 无锁     | hashCode(31) + 分代年龄(4)        | 0       | 01     |
| 偏向锁   | 线程ID(54) + Epoch(2) + 年龄(4)   | 1       | 01     |
```

- 偏向锁状态下，Mark Word 被线程 ID 占满，**没有位置存 hashCode**
- 一旦调用 `Object.hashCode()` 或 `System.identityHashCode()`，**必须撤销偏向锁**，把 hashCode 写回 Mark Word

### 情况三：调用了 `wait()` / `notify()` / `notifyAll()`

- 偏向锁和轻量级锁都是**用户态**的锁机制，没有等待队列
- `wait/notify` 必须依赖 ObjectMonitor 的 WaitSet / EntryList
- 因此调用时会**直接升级为重量级锁**，跳过轻量级锁阶段

### 情况四：批量重偏向与批量撤销（JVM 优化）

JVM 不会为每一次撤销都付出 Safepoint 的高昂代价，而是有**批量优化机制**：

| 阈值 | 触发条件 | 行为 |
|------|---------|------|
| **20 次** | 同一类对象的偏向锁被不同线程撤销累计 20 次 | **批量重偏向**：该类所有对象的偏向锁重新偏向到新线程 |
| **40 次** | 同一类对象的偏向锁被撤销累计 40 次 | **批量撤销**：该类新创建的对象**禁用偏向锁**，直接进入轻量级锁 |

**Epoch 机制**：
- Mark Word 中有 2 位的 Epoch 字段
- 批量重偏向时，JVM 更新类元数据中的 Epoch 值
- 对象头的 Epoch 与类元数据不一致时，说明该对象的偏向锁已过期，可以直接 CAS 替换线程 ID，无需 Safepoint 撤销

## 五、一句话总结

> 锁升级的本质是 **"先用最轻量的方案，竞争加剧再逐步加重"**：偏向锁消除无竞争开销，轻量级锁避免内核切换，重量级锁保证激烈竞争下的正确性。偏向锁的撤销核心就两点：**别的线程来抢**、**需要空间存 hashCode**，而现代 JDK 已逐步淘汰偏向锁，正是因为它在多线程环境下撤销成本过高。
